API에 초점을 맞춘 프로젝트 형태로 Django 규칙을 원할 때 이 스택을 사용하세요.

```npm
npm create better-fullstack@latest my-django-api -- \
  --ecosystem python \
  --python-web-framework django \
  --python-orm sqlalchemy \
  --python-validation pydantic \
  --python-api django-rest-framework \
  --python-quality ruff
```

## 이것이 만드는 것

- Django을 사용하는 Python 프로젝트.
- Django REST Python API 옵션으로서의 프레임워크.
- SQLAlchemy 및 Pydantic 선택.
- 코드 품질을 위한 러프.

## 생성된 모양

이 스택은 Django 생태계를 원하지만 API 첫 번째 시작점을 원하는 개발자를 위한 것입니다. 생성된 프로젝트는 `--python-api django-rest-framework`을 기록하고, Django 및 DRF 종속성을 포함하고, 앱 진입점 내에 최소 Django 설정 모듈을 연결하고 상태 스타일 API 응답을 노출합니다.

```txt
my-django-api/
├── pyproject.toml
├── alembic.ini
├── migrations/
│   ├── env.py
│   └── versions/
├── src/app/
│   ├── main.py
│   ├── settings.py
│   ├── models.py
│   ├── schemas.py
│   ├── database.py
│   └── crud.py
└── tests/
    ├── test_main.py
    └── test_database.py
```

SQLAlchemy 및 Pydantic 선택은 Django HTTP 표면에 대한 명시적인 Python 데이터 모델 및 서비스 계층 유효성 검사를 원할 때 여전히 유용합니다. 팀에서 가장 관용적인 Django 경로를 원하는 경우 프로젝트가 성장함에 따라 지속성을 Django 모델 및 DRF 직렬 변환기로 마이그레이션할지 여부를 계획하세요.

## 그것을 선택하는 경우

최소 API 표면보다 규칙, 관리자 중심 구조 및 생태계 깊이가 더 중요한 경우 Django을 선택하세요.

Django 관리자, Django 인증, 기존 미들웨어 또는 여러 Django 앱을 추가하려는 경우 FastAPI 가이드 대신 이 옵션을 선택하세요. 서비스가 작고 경로 중심적이며 AI 또는 데이터 처리 코드 옆에 쉽게 포함되어야 하는 경우 FastAPI을 선택하세요.

| 결정 포인트 | Django REST API | FastAPI 서비스 |
| --- | --- | --- |
| 팀 기대 | Django 컨벤션 및 생태계 | 명시적 Python 모듈 및 경로 함수 |
| API 레이어 | Django REST 프레임워크 | FastAPI + 피단틱 |
| 백오피스/관리자 | 자연 확장 지점 | 별도의 도구 추가 |
| 미들웨어/인증 | Django-기본 경로 | 명시적으로 작성 |
| 시작 복잡성 | 더 많은 프레임워크 구조 | 더 작은 초기 표면 |

## 대표 조각

생성된 DRF 끝점은 의도적으로 작습니다. API이 커지면 보기 기능을 앱 모듈로 이동하여 이를 확장하세요.

```python
from rest_framework.decorators import api_view
from rest_framework.response import Response


@api_view(["GET"])
def api_status(request):
    return Response({"status": "ok", "framework": "django-rest-framework"})
```

실용적인 리소스 끝점의 경우 SQLAlchemy을 지속성 도구로 유지하더라도 직렬 변환기와 유사한 경계를 추가하세요.

```python
from pydantic import BaseModel, EmailStr
from rest_framework.decorators import api_view
from rest_framework.response import Response


class UserCreate(BaseModel):
    email: EmailStr
    name: str | None = None


@api_view(["POST"])
def create_user(request):
    payload = UserCreate.model_validate(request.data)
    user = user_service.create_user(email=payload.email, name=payload.name)
    return Response({"id": user.id, "email": user.email}, status=201)
```

나중에 관용적인 DRF 직렬 변환기로 전환하는 경우 동일한 경계를 유지하십시오. 즉, 요청 데이터를 검증하고, 애플리케이션 로직을 호출하고, 안정적인 응답 형태를 반환하십시오.

```python
from rest_framework import serializers


class UserSerializer(serializers.Serializer):
    id = serializers.IntegerField(read_only=True)
    email = serializers.EmailField()
    name = serializers.CharField(required=False, allow_blank=True)
```

## 마이그레이션 및 테스트

Django 모델을 채택하는 경우 Django 마이그레이션을 사용하세요. SQLAlchemy을 지속성 정보 소스로 유지하는 경우 Alembic을 사용하세요. 명확한 소유권 분할이 없는 한 동일한 테이블에 대해 두 마이그레이션 시스템을 모두 실행하지 마세요.

```bash
uv run ruff check .
uv run pytest
```

기존 Django 프로젝트 레이아웃에서 Django 모델 지속성의 경우:

```bash
uv run python manage.py makemigrations
uv run python manage.py migrate
```

SQLAlchemy 지속성의 경우:

```bash
uv run alembic revision --autogenerate -m "add users"
uv run alembic upgrade head
```

경로 테스트는 구현 세부 정보를 볼 뿐만 아니라 Django의 테스트 클라이언트 또는 DRF의 API 테스트 클라이언트를 사용하고 JSON 계약을 확인해야 합니다.

## 호환성 참고사항

- `--python-api django-rest-framework`에는 `--python-web-framework django`이 필요합니다.
- `--python-api django-rest-framework`을 FastAPI 스택에 추가하지 마세요. CLI 유효성 검사에서는 해당 조합을 거부합니다.
- `--python-orm sqlalchemy`은 Python 생태계에서 사용할 수 있지만 Django의 기본 제공 관리자 및 일반 보기는 Django 모델을 중심으로 설계되었습니다.
- `--python-validation pydantic`은 서비스 계층 페이로드 검증에 유용한 반면, DRF 직렬 변환기는 DRF 뷰 내의 일반적인 요청/응답 도구입니다.

## 배포 참고사항

Django 배포에는 일반적인 프로덕션 설정 단계(`SECRET_KEY`, `ALLOWED_HOSTS`, 신뢰할 수 있는 원본, 데이터베이스 자격 증명 및 정적 자산 처리 구성)가 필요합니다. 생성된 앱이 여전히 최소인 경우 이를 시작점으로 취급하고 프로덕션 전에 설정을 명시적인 환경 기반 모듈로 이동합니다.

```bash
HOST=0.0.0.0 PORT="${PORT:-8000}" uv run python -m app.main
```

전체 Django 프로젝트 레이아웃 대신 생성된 경량 진입점을 유지하는 경우 향후 Django 기여자가 기본 `settings.py` 및 `manage.py` 구조가 변경되지 않은 것으로 가정하지 않도록 설정이 로드되는 문서를 작성하세요. 프로덕션 전에 플랫폼에 필요한 WSGI/ASGI 서버 및 배포 설정을 추가하세요.

## 문제 해결

| 징후 | 확인하다 |
| --- | --- |
| DRF 가져오기 실패 | `--python-api django-rest-framework`을 사용하여 프로젝트가 생성되었는지 확인합니다. |
| Django 설정이 구성되지 않았습니다. | Django 모델 또는 뷰를 가져오기 전에 `DJANGO_SETTINGS_MODULE`이 설정되어 있는지 확인하세요. |
| 마이그레이션 혼란 | 각 테이블의 소유자로 Django 마이그레이션 또는 Alembic을 선택하세요. |
| FastAPI 예시는 적용되지 않습니다. | 이 스택은 FastAPI 데코레이터가 아닌 Django/DRF 라우팅을 사용합니다. |
