Verwenden Sie diesen Stapel, wenn Sie Django-Konventionen mit einer API-fokussierten Projektform wünschen.

```npm
npm create better-fullstack@latest my-django-api -- \
  --ecosystem python \
  --python-web-framework django \
  --python-orm sqlalchemy \
  --python-validation pydantic \
  --python-api django-rest-framework \
  --python-quality ruff
```

## Was dadurch entsteht

- Ein Python-Projekt mit Django.
- Django REST Framework als Python-Option API.
- SQLAlchemy und Pydantic-Auswahlen.
- Ruff für Codequalität.

## Erzeugte Form

Dieser Stack ist für Entwickler gedacht, die das Django-Ökosystem, aber einen API-ersten Ausgangspunkt wünschen. Das generierte Projekt zeichnet `--python-api django-rest-framework` auf, enthält Django- und DRF-Abhängigkeiten, verbindet ein minimales Django-Einstellungsmodul innerhalb des App-Einstiegspunkts und stellt eine API-Antwort im Integritätsstil bereit.

```txt
my-django-api/
├── pyproject.toml
├── alembic.ini
├── migrations/
│   ├── env.py
│   └── versions/
├── src/app/
│   ├── main.py
│   ├── settings.py
│   ├── models.py
│   ├── schemas.py
│   ├── database.py
│   └── crud.py
└── tests/
    ├── test_main.py
    └── test_database.py
```

Die Auswahlmöglichkeiten SQLAlchemy und Pydantic sind weiterhin nützlich, wenn Sie explizite Python-Datenmodelle und Service-Layer-Validierung rund um eine HTTP-Oberfläche Django wünschen. Wenn Ihr Team den idiomatischsten Django-Pfad wünscht, planen Sie, ob die Persistenz in Richtung Django-Modelle und DRF-Serialisierer migriert werden soll, wenn das Projekt wächst.

## Wann man es wählt

Wählen Sie Django, wenn Konventionen, admin-orientierte Struktur und Ökosystemtiefe wichtiger sind als eine minimale API-Oberfläche.

Wählen Sie diese Option anstelle der FastAPI-Anleitung, wenn Sie voraussichtlich Django Admin-, Django-Authentifizierungs-, herkömmliche Middleware- oder mehrere Django-Apps hinzufügen möchten. Wählen Sie FastAPI, wenn der Dienst klein, routenorientiert und einfach neben AI oder Datenverarbeitungscode einzubetten sein soll.

| Entscheidungspunkt | Django REST API | FastAPI-Dienst |
| --- | --- | --- |
| Teamerwartung | Django Konventionen und Ökosystem | Explizite Python-Module und Routenfunktionen |
| API Ebene | Django REST Framework | FastAPI + Pydantic |
| Backoffice/Administration | Natürlicher Erweiterungspunkt | Fügen Sie ein separates Tool hinzu |
| Middleware/Authentifizierung | Django-nativer Pfad | Verfassen Sie explizit |
| Komplexität des Startvorgangs | Mehr Rahmenstruktur | Kleinere Anfangsfläche |

## Repräsentative Auszüge

Der generierte DRF-Endpunkt ist absichtlich klein. Erweitern Sie dies, indem Sie Ansichtsfunktionen in App-Module verschieben, sobald API wächst.

```python
from rest_framework.decorators import api_view
from rest_framework.response import Response


@api_view(["GET"])
def api_status(request):
    return Response({"status": "ok", "framework": "django-rest-framework"})
```

Fügen Sie für einen praktischen Ressourcenendpunkt eine Serialisierungs-ähnliche Grenze hinzu, auch wenn Sie SQLAlchemy als Persistenztool beibehalten.

```python
from pydantic import BaseModel, EmailStr
from rest_framework.decorators import api_view
from rest_framework.response import Response


class UserCreate(BaseModel):
    email: EmailStr
    name: str | None = None


@api_view(["POST"])
def create_user(request):
    payload = UserCreate.model_validate(request.data)
    user = user_service.create_user(email=payload.email, name=payload.name)
    return Response({"id": user.id, "email": user.email}, status=201)
```

Wenn Sie später auf idiomatische DRF-Serialisierer umsteigen, behalten Sie die gleichen Grenzen bei: Validieren Sie Anforderungsdaten, rufen Sie die Anwendungslogik auf und geben Sie eine stabile Antwortform zurück.

```python
from rest_framework import serializers


class UserSerializer(serializers.Serializer):
    id = serializers.IntegerField(read_only=True)
    email = serializers.EmailField()
    name = serializers.CharField(required=False, allow_blank=True)
```

## Migrationen und Tests

Verwenden Sie Django-Migrationen, wenn Sie Django-Modelle übernehmen. Verwenden Sie Alembic, wenn Sie SQLAlchemy als Persistenzquelle der Wahrheit behalten. Vermeiden Sie es, beide Migrationssysteme für dieselben Tabellen auszuführen, es sei denn, Sie haben eine klare Eigentumsaufteilung.

```bash
uv run ruff check .
uv run pytest
```

Für die Django-Modellpersistenz in einem herkömmlichen Django-Projektlayout:

```bash
uv run python manage.py makemigrations
uv run python manage.py migrate
```

Für SQLAlchemy Persistenz:

```bash
uv run alembic revision --autogenerate -m "add users"
uv run alembic upgrade head
```

Routentests sollten den Testclient von Django oder den Testclient API von DRF verwenden und den JSON-Vertrag bestätigen, nicht nur Implementierungsdetails anzeigen.

## Kompatibilitätshinweise

- `--python-api django-rest-framework` erfordert `--python-web-framework django`.
- Fügen Sie `--python-api django-rest-framework` nicht zu einem FastAPI-Stack hinzu; Die CLI-Validierung lehnt diese Kombination ab.
- `--python-orm sqlalchemy` ist im Python-Ökosystem verfügbar, aber die integrierten Admin- und generischen Ansichten von Django basieren auf Django-Modellen.
- `--python-validation pydantic` ist nützlich für die Nutzlastvalidierung auf Service-Ebene, während DRF-Serialisierer das herkömmliche Anforderungs-/Antwort-Tool in DRF-Ansichten sind.

## Bereitstellungshinweise

Django-Bereitstellungen erfordern die üblichen Produktionseinstellungen: Konfigurieren Sie `SECRET_KEY`, `ALLOWED_HOSTS`, vertrauenswürdige Ursprünge, Datenbankanmeldeinformationen und die Handhabung statischer Assets. Wenn die generierte App noch minimal ist, betrachten Sie sie als Ausgangspunkt und verschieben Sie die Einstellungen vor der Produktion in explizite umgebungsgesteuerte Module.

```bash
HOST=0.0.0.0 PORT="${PORT:-8000}" uv run python -m app.main
```

Wenn Sie beim generierten einfachen Einstiegspunkt statt bei einem vollständigen Django-Projektlayout bleiben, dokumentieren Sie, wo die Einstellungen geladen werden, damit zukünftige Django-Mitwirkende nicht davon ausgehen, dass die Standardstrukturen `settings.py` und `manage.py` unverändert bestehen. Fügen Sie vor der Produktion den WSGI/ASGI-Server und die Bereitstellungseinstellungen hinzu, die Ihre Plattform erwartet.

## Fehlerbehebung

| Symptom | Überprüfen |
| --- | --- |
| DRF-Importe schlagen fehl | Bestätigen Sie, dass das Projekt mit `--python-api django-rest-framework` erstellt wurde. |
| Django-Einstellungen sind nicht konfiguriert | Stellen Sie sicher, dass `DJANGO_SETTINGS_MODULE` festgelegt ist, bevor Sie Django-Modelle oder -Ansichten importieren. |
| Migrationsverwirrung | Wählen Sie Django migrations oder Alembic als Besitzer für jede Tabelle. |
| FastAPI Beispiele gelten nicht | Dieser Stack verwendet Django/DRF-Routing, keine FastAPI-Dekoratoren. |
