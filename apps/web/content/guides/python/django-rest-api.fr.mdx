Utilisez cette pile lorsque vous souhaitez des conventions Django avec une forme de projet centrée sur API.

```npm
npm create better-fullstack@latest my-django-api -- \
  --ecosystem python \
  --python-web-framework django \
  --python-orm sqlalchemy \
  --python-validation pydantic \
  --python-api django-rest-framework \
  --python-quality ruff
```

## Ce que cela crée

- Un projet Python utilisant Django.
- Django REST Framework comme option Python API.
- Sélections SQLAlchemy et Pydantic.
- Ruff pour la qualité du code.

## Forme générée

Cette pile est destinée aux développeurs qui souhaitent l'écosystème Django mais un premier point de départ API. Le projet généré enregistre `--python-api django-rest-framework`, inclut les dépendances Django et DRF, connecte un module de paramètres Django minimal à l'intérieur du point d'entrée de l'application et expose une réponse API de style d'intégrité.

```txt
my-django-api/
├── pyproject.toml
├── alembic.ini
├── migrations/
│   ├── env.py
│   └── versions/
├── src/app/
│   ├── main.py
│   ├── settings.py
│   ├── models.py
│   ├── schemas.py
│   ├── database.py
│   └── crud.py
└── tests/
    ├── test_main.py
    └── test_database.py
```

Les sélections SQLAlchemy et Pydantic sont toujours utiles lorsque vous souhaitez des modèles de données Python explicites et une validation de couche de service autour d'une surface HTTP Django. Si votre équipe souhaite le chemin Django le plus idiomatique, prévoyez s'il faut migrer la persistance vers les modèles Django et les sérialiseurs DRF à mesure que le projet se développe.

## Quand le choisir

Choisissez Django lorsque les conventions, la structure orientée vers l'administration et la profondeur de l'écosystème comptent plus qu'une surface API minimale.

Choisissez ceci plutôt que le guide FastAPI lorsque vous prévoyez d'ajouter l'administrateur Django, l'authentification Django, un middleware conventionnel ou plusieurs applications Django. Choisissez FastAPI lorsque le service doit rester petit, axé sur les itinéraires et facile à intégrer à côté de AI ou du code de traitement des données.

| Point de décision | Django REST API | Service FastAPI |
| --- | --- | --- |
| Attente de l'équipe | Django conventions et écosystème | Modules Python explicites et fonctions de route |
| Couche API | Django REST Cadre | FastAPI + Pydantique |
| Back-office/administrateur | Point d'extension naturel | Ajouter un outil séparé |
| Middleware/authentification | Django-chemin natif | Composer explicitement |
| Complexité du démarrage | Plus de structure de cadre | Surface initiale plus petite |

## Extraits représentatifs

Le point de terminaison DRF généré est intentionnellement petit. Développez-le en déplaçant les fonctions d'affichage dans les modules d'application une fois que API grandit.

```python
from rest_framework.decorators import api_view
from rest_framework.response import Response


@api_view(["GET"])
def api_status(request):
    return Response({"status": "ok", "framework": "django-rest-framework"})
```

Pour un point de terminaison de ressource pratique, ajoutez une limite de type sérialiseur même si vous conservez SQLAlchemy comme outil de persistance.

```python
from pydantic import BaseModel, EmailStr
from rest_framework.decorators import api_view
from rest_framework.response import Response


class UserCreate(BaseModel):
    email: EmailStr
    name: str | None = None


@api_view(["POST"])
def create_user(request):
    payload = UserCreate.model_validate(request.data)
    user = user_service.create_user(email=payload.email, name=payload.name)
    return Response({"id": user.id, "email": user.email}, status=201)
```

Si vous optez ultérieurement pour des sérialiseurs DRF idiomatiques, conservez la même limite : validez les données de la requête, appelez la logique de l'application et renvoyez une forme de réponse stable.

```python
from rest_framework import serializers


class UserSerializer(serializers.Serializer):
    id = serializers.IntegerField(read_only=True)
    email = serializers.EmailField()
    name = serializers.CharField(required=False, allow_blank=True)
```

## Migrations et tests

Utilisez les migrations Django si vous adoptez les modèles Django. Utilisez Alembic si vous conservez SQLAlchemy comme source de vérité persistante. Évitez d'exécuter les deux systèmes de migration sur les mêmes tables, sauf si vous disposez d'une répartition claire de la propriété.

```bash
uv run ruff check .
uv run pytest
```

Pour la persistance du modèle Django dans une présentation de projet Django conventionnelle :

```bash
uv run python manage.py makemigrations
uv run python manage.py migrate
```

Pour la persistance SQLAlchemy :

```bash
uv run alembic revision --autogenerate -m "add users"
uv run alembic upgrade head
```

Les tests de route doivent utiliser le client de test Django ou le client de test API de DRF et affirmer le contrat JSON, et pas seulement afficher les détails d'implémentation.

## Remarques sur la compatibilité

- `--python-api django-rest-framework` nécessite `--python-web-framework django`.
- N'ajoutez pas `--python-api django-rest-framework` à une pile FastAPI ; la validation CLI rejette cette combinaison.
- `--python-orm sqlalchemy` est disponible dans l'écosystème Python, mais les vues d'administration et génériques intégrées de Django sont conçues autour des modèles Django.
- `--python-validation pydantic` est utile pour la validation de la charge utile de la couche de service, tandis que les sérialiseurs DRF sont l'outil de requête/réponse conventionnel dans les vues DRF.

## Notes de déploiement

Les déploiements Django nécessitent la transmission des paramètres de production habituels : configurez `SECRET_KEY`, `ALLOWED_HOSTS`, les origines approuvées, les informations d'identification de la base de données et la gestion des actifs statiques. Si l'application générée est encore minime, traitez-la comme un point de départ et déplacez les paramètres dans des modules explicites basés sur l'environnement avant la production.

```bash
HOST=0.0.0.0 PORT="${PORT:-8000}" uv run python -m app.main
```

Si vous restez avec le point d'entrée léger généré au lieu d'une présentation de projet Django complète, documentez où les paramètres sont chargés afin que les futurs contributeurs Django ne supposent pas que la structure par défaut `settings.py` et `manage.py` existe inchangée. Avant la production, ajoutez le serveur WSGI/ASGI et les paramètres de déploiement attendus par votre plateforme.

## Dépannage

| Symptôme | Vérifier |
| --- | --- |
| Les importations DRF échouent | Confirmez que le projet a été créé avec `--python-api django-rest-framework`. |
| Les paramètres Django ne sont pas configurés | Assurez-vous que `DJANGO_SETTINGS_MODULE` est défini avant d'importer des modèles ou des vues Django. |
| Confusion migratoire | Choisissez les migrations Django ou Alembic comme propriétaire pour chaque table. |
| FastAPI exemples ne s'appliquent pas | Cette pile utilise le routage Django/DRF, et non les décorateurs FastAPI. |
