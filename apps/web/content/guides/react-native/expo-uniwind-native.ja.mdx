モバイルファースト Expo アプリが必要で、同じスキャフォールド内に生成されたバックエンドが必要ない場合は、このスタックを使用します。

```npm
npm create better-fullstack@latest my-mobile-app -- \
  --ecosystem react-native \
  --frontend native-uniwind \
  --mobile-navigation expo-router \
  --mobile-ui uniwind \
  --mobile-storage mmkv \
  --mobile-testing maestro-react-native-testing-library \
  --mobile-push expo-notifications \
  --mobile-ota expo-updates \
  --mobile-deep-linking expo-linking \
  --ai-docs none \
  --no-install
```

## これが生み出すもの

- Uniwind を使用する Expo ネイティブ アプリ。
- Expo ファイルベースのネイティブ ルート用のルーター。
- MMKV ストレージ、React Native テスト ライブラリ、Maestro、プッシュ通知、OTA アップデート、ディープ リンク。
- デフォルトでは、Web フロントエンド、バックエンド、データベース、ORM、認証、または API レイヤーはありません。
- 後で API と組み合わせることができるモバイルファーストのプロジェクト形状。

## 生成された形状

このコマンドは、フルスタック Web アプリではなく、ネイティブ フロントエンド基盤を作成します。これにより、バックエンドの決定が最初のスキャフォールドから外されるため、モバイル アプリは後で既存の API または個別に生成されたサービスを参照できるようになります。

代表的な形状：

```txt
my-mobile-app/
  bts.jsonc
  app/
  components/
  package.json
  maestro/
```

正確な Expo パスはテンプレートのバージョンによって異なりますが、所有権は単純です。ネイティブの画面とコンポーネントはアプリ内に存在し、サーバーの問題は意図的に存在しません。

## 画面例

Uniwind を使用すると、React Native コンポーネントでユーティリティ スタイルのクラスを使用できます。

```tsx
import { Text, View } from "react-native";

export default function HomeScreen() {
  return (
    <View className="flex-1 items-center justify-center bg-white px-6">
      <Text className="text-2xl font-semibold text-slate-950">
        Mobile app
      </Text>
      <Text className="mt-2 text-center text-slate-600">
        Connect an API when the product contract is ready.
      </Text>
    </View>
  );
}
```

バックエンドを追加するときは、API クライアントを小さなモジュールの背後に保持してください。これにより、画面を変更せずに、ローカル、ステージング、本番の API オリジンを簡単に交換できるようになります。

## 互換性に関する注意事項

React Native は、Better Fullstack の独自のエコシステムです。 `--ecosystem react-native` を使用します。 TypeScript Web フロントエンド フラグとネイティブ フロントエンド フラグを組み合わせないでください。

このガイドでは、バックエンド、データベース、ORM、認証、API の生成を意図的にモバイル スキャフォールドから除外しています。モバイル認証とサーバー API には、製品要件が明確になった後で、プラットフォーム固有の決定を行う必要があります。

## いつ選択するか

API が個別に作成される可能性があるモバイル プロトタイプ、コンパニオン アプリ、およびネイティブ アプリ基盤の場合は、これを選択します。

## 後で API を追加する

一般的な次のステップは、軽量 TypeScript バックエンドの場合は Hono API、Web サーフェスも必要な場合は Next.js または TanStack Start Web アプリ、または Better Fullstack の外部にすでに存在する専用のバックエンドです。

API を追加するときは、ネイティブ アプリがどのように処理するかを決定します。

- 環境固有の API ベース URL。
- デバイス上の認証セッション ストレージ。
- リフレッシュトークンまたはセッションの更新。
- オフライン動作と楽観的なアップデート。
- 開発、プレビュー、実稼働用のネイティブ ビルド プロファイル。

## 導入メモ

Expo デプロイメントは Web デプロイメントとは異なります。ビルド時に使用されるネイティブ ビルド プロファイル、アプリ識別子、署名資格情報、および環境変数について計画します。サーバーの秘密をモバイル バンドルに含めないでください。ネイティブ アプリは、サーバー側のシークレットを保持するバックエンドを呼び出す必要があります。

## トラブルシューティング

- 生成された Web ファイルが存在しない場合、それは想定内です。このスタックはネイティブ フロントエンドを使用し、Web フロントエンドを使用しません。
- バックエンド ルートが存在しない場合、コマンドは明示的に `--backend none` を選択します。
- スタイルが適用されない場合は、生成されたアプリの Uniwind セットアップとクラス スキャン構成を確認してください。
- API 呼び出しがシミュレーターでは機能するが、デバイスでは機能しない場合は、API ベース URL がデバイスから到達可能なホストを指しているかどうかを確認してください。
