當你想先做手機端 Expo app，並且不需要在同一個 scaffold 裡產生 backend 時，使用這個 stack。

```npm
npm create better-fullstack@latest my-mobile-app -- \
  --ecosystem react-native \
  --frontend native-uniwind \
  --mobile-navigation expo-router \
  --mobile-ui uniwind \
  --mobile-storage mmkv \
  --mobile-testing maestro-react-native-testing-library \
  --mobile-push expo-notifications \
  --mobile-ota expo-updates \
  --mobile-deep-linking expo-linking \
  --ai-docs none \
  --no-install
```

## 創建內容

- 一個使用 Uniwind 的 Expo native app。
- 用 Expo Router 做基於檔案的 native routes。
- MMKV、React Native Testing Library、Maestro、push notifications、OTA updates 和 deep linking。
- 預設不產生 web frontend、backend、database、ORM、auth 或 API layer。
- 一個 mobile-first 專案基礎，之後可以再接 API。

## 生成結構

這個指令創建的是 native frontend 基礎，而不是 fullstack web app。它把 backend 決策留到初始 scaffold 之外，這樣移動 app 之後可以指向現有 API，或指向單獨產生的服務。

代表性結構：

```txt
my-mobile-app/
  bts.jsonc
  app/
  components/
  package.json
  maestro/
```

具體 Expo 路徑會隨 template 版本變化，但職責邊界很直接：native screens 和 components 在 app 裡，server concerns 刻意不生成。

## 範例 screen

Uniwind 讓你可以在 React Native components 中使用 utility-style classes：

```tsx
import { Text, View } from "react-native";

export default function HomeScreen() {
  return (
    <View className="flex-1 items-center justify-center bg-white px-6">
      <Text className="text-2xl font-semibold text-slate-950">
        Mobile app
      </Text>
      <Text className="mt-2 text-center text-slate-600">
        Connect an API when the product contract is ready.
      </Text>
    </View>
  );
}
```

當加入 backend 時，把 API clients 收在一個小模組後面。這樣切換 local、staging、production API origins 時，不需要改 screens。

## 相容性

React Native 在 Better Fullstack 中是獨立生態。使用 `--ecosystem react-native`；不要把 TypeScript web frontend flags 和 native frontend flags 混在一起。

這篇指南刻意不在 mobile scaffold 產生 backend、database、ORM、auth 和 API。行動端認證和 server APIs 涉及平台特定決策，最好等產品需求明確之後再做。

## 何時選擇

適合行動端原型、companion apps，以及 API 可能單獨建立的 native app foundations。

## 之後加入 API

常見下一步包括：用 Hono 做輕量級 TypeScript backend；如果還需要 web surface，則建立 Next.js 或 TanStack Start app；或接入 Better Fullstack 之外已經存在的專用 backend。

在加入 API 前，先決定 native app 如何處理：

- 不同環境的 API base URLs。
- 設備上的 auth session storage。
- Refresh tokens 或 session renewal。
- Offline behavior 和 optimistic updates。
- development、preview、production 的 native build profiles。

## 部署

Expo 部署不同於 web 部署。需要規劃 native build profiles、app identifiers、signing credentials，以及 build time 使用的 environment variables。不要把 server secrets 放進 mobile bundle；native apps 應該呼叫由伺服器儲存 secrets 的 backend。

## 故障排查

- 如果沒有產生 web files，這是預期行為：這個 stack 使用 native frontend，不產生 web frontend。
- 如果沒有 backend routes，指令明確選擇了 `--backend none`。
- 如果樣式沒有生效，檢查產生 app 中的 Uniwind setup 和 class scanning 設定。
- 如果 API calls 在 simulator 可用但真機不可用，檢查 API base URL 是否指向裝置可存取的 host。
