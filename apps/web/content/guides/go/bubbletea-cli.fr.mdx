Utilisez cette pile lorsque vous souhaitez une interface utilisateur de terminal Go au lieu d'un API Web.

```npm
npm create better-fullstack@latest my-go-cli -- \
  --ecosystem go \
  --go-web-framework none \
  --go-orm none \
  --go-api none \
  --go-cli bubbletea \
  --go-logging none \
  --go-auth none
```

## Ce que cela crée

- Un projet Go CLI.
- Bubble Tea comme option de l'interface utilisateur du terminal.
- Aucun framework Web, ORM, gRPC ou module complémentaire d'authentification.

## Forme générée

Cette pile crée une application de terminal interactif, pas un serveur. Bubble Tea organise le programme autour d'un modèle, de messages, de mises à jour et de vues.

La boucle principale est simple :

- Le modèle contient l’état actuel de l’application.
- Les messages décrivent les entrées, les ticks ou les résultats des commandes.
- `Update` modifie le modèle et renvoie les commandes de suivi.
- `View` restitue le modèle actuel sous forme de texte de terminal.

## Arborescence de fichiers représentative

```txt
my-go-cli/
  go.mod
  cmd/
    app/
      main.go
  internal/
    tui/
      model.go
      update.go
      view.go
```

Conservez les effets secondaires dans les commandes Bubble Tea lorsque cela est possible. Cela rend le modèle et la vue plus faciles à tester sans exécuter un véritable terminal.

## Exemples de modélisation et de mise à jour

Un modèle Bubble Tea peut démarrer avec juste assez d'état pour restituer et réagir aux entrées :

```go
package tui

type Model struct {
	Items    []string
	Selected int
	Quitting bool
}

func NewModel() Model {
	return Model{
		Items: []string{"Build", "Test", "Deploy"},
	}
}
```

La fonction update possède les transitions d'état :

```go
package tui

import tea "github.com/charmbracelet/bubbletea"

func (m Model) Update(msg tea.Msg) (tea.Model, tea.Cmd) {
	switch msg := msg.(type) {
	case tea.KeyMsg:
		switch msg.String() {
		case "q", "ctrl+c":
			m.Quitting = true
			return m, tea.Quit
		case "j", "down":
			if len(m.Items) > 0 {
				m.Selected = (m.Selected + 1) % len(m.Items)
			}
		case "k", "up":
			if len(m.Items) > 0 {
				m.Selected = (m.Selected - 1 + len(m.Items)) % len(m.Items)
			}
		}
	}

	return m, nil
}
```

La vue doit être rendue à partir de l'état sans effectuer de travail :

```go
package tui

import "strings"

func (m Model) View() string {
	if m.Quitting {
		return ""
	}

	var b strings.Builder
	b.WriteString("Tasks\n\n")

	for i, item := range m.Items {
		cursor := " "
		if i == m.Selected {
			cursor = ">"
		}
		b.WriteString(cursor + " " + item + "\n")
	}

	b.WriteString("\nq: quit  j/k: move\n")
	return b.String()
}
```

## Remarques sur la compatibilité

Les projets CLI doivent conserver les catégories réservées au serveur définies sur `none`, à moins qu'il ne s'agisse d'outils intentionnellement hybrides.

Conservez `--go-cli bubbletea` avec `--go-web-framework none`, `--go-orm none` et `--go-api none` pour une application orientée terminal. Si le CLI parle plus tard à un API, conservez ce client dans un package séparé et appelez-le via des commandes au lieu de depuis `View`.

## Quand le choisir

Choisissez Bubble Tea pour les tableaux de bord de terminal, les outils de développement et les interfaces de ligne de commande où l'interactivité est importante.

Il convient parfaitement aux outils qui nécessitent une sélection, une progression, des formulaires ou un statut en direct sans créer d'interface utilisateur de navigateur.

## Notes de tests et de déploiement

Testez directement les transitions du modèle :

```bash
go test ./...
```

Pour la distribution, créez un seul binaire et documentez les variables d'environnement ou les fichiers de configuration pris en charge :

```bash
go build ./cmd/app
```

Si le projet généré utilise un chemin de commande différent, suivez le chemin dans son répertoire `cmd/`.

## Dépannage

- Si les pressions sur une touche ne font rien, vérifiez que le terminal ciblé envoie la séquence de touches que Bubble Tea reçoit.
- Si l'écran laisse une sortie obsolète, vérifiez les options de démarrage et d'arrêt du programme pour connaître le comportement de l'écran alternatif.
- Si les tests sont difficiles à écrire, déplacez les effets secondaires dans les fonctions `tea.Cmd` et gardez `Update` déterministe.
- Si la sortie s'enroule mal, testez sur des largeurs étroites et conservez le formatage `View` réactif à l'espace disponible lorsque le modèle suit les dimensions.
