Web API の代わりに Go ターミナル UI が必要な場合は、このスタックを使用します。

```npm
npm create better-fullstack@latest my-go-cli -- \
  --ecosystem go \
  --go-web-framework none \
  --go-orm none \
  --go-api none \
  --go-cli bubbletea \
  --go-logging none \
  --go-auth none
```

## これが生み出すもの

- Go CLI プロジェクト。
- Bubble Tea をターミナル UI オプションとして使用します。
- Web フレームワーク、ORM、gRPC、または認証アドオンはありません。

## 生成された形状

このスタックは、サーバーではなく、対話型のターミナル アプリを作成します。 Bubble Tea は、モデル、メッセージ、更新、ビューを中心にプログラムを編成します。

メインループは単純です。

- モデルは現在のアプリケーションの状態を保持します。
- メッセージは、入力、ティック、またはコマンドの結果を説明します。
- `Update` はモデルを変更し、フォローアップ コマンドを返します。
- `View` は、現在のモデルを端末テキストとしてレンダリングします。

## 代表的なファイルツリー

```txt
my-go-cli/
  go.mod
  cmd/
    app/
      main.go
  internal/
    tui/
      model.go
      update.go
      view.go
```

可能な限り、副作用を Bubble Tea コマンドに含めてください。これにより、実際の端末を実行しなくても、モデルとビューをテストしやすくなります。

## モデルと更新の例

Bubble Tea モデルは、レンダリングして入力に反応するのに十分な状態から開始できます。

```go
package tui

type Model struct {
	Items    []string
	Selected int
	Quitting bool
}

func NewModel() Model {
	return Model{
		Items: []string{"Build", "Test", "Deploy"},
	}
}
```

更新関数は状態遷移を所有します。

```go
package tui

import tea "github.com/charmbracelet/bubbletea"

func (m Model) Update(msg tea.Msg) (tea.Model, tea.Cmd) {
	switch msg := msg.(type) {
	case tea.KeyMsg:
		switch msg.String() {
		case "q", "ctrl+c":
			m.Quitting = true
			return m, tea.Quit
		case "j", "down":
			if len(m.Items) > 0 {
				m.Selected = (m.Selected + 1) % len(m.Items)
			}
		case "k", "up":
			if len(m.Items) > 0 {
				m.Selected = (m.Selected - 1 + len(m.Items)) % len(m.Items)
			}
		}
	}

	return m, nil
}
```

ビューは作業を実行せずに状態からレンダリングする必要があります。

```go
package tui

import "strings"

func (m Model) View() string {
	if m.Quitting {
		return ""
	}

	var b strings.Builder
	b.WriteString("Tasks\n\n")

	for i, item := range m.Items {
		cursor := " "
		if i == m.Selected {
			cursor = ">"
		}
		b.WriteString(cursor + " " + item + "\n")
	}

	b.WriteString("\nq: quit  j/k: move\n")
	return b.String()
}
```

## 互換性に関する注意事項

CLI プロジェクトでは、意図的にハイブリッド ツールでない限り、サーバー専用カテゴリを `none` に設定しておく必要があります。

ターミナルファースト アプリの場合は、`--go-cli bubbletea` を `--go-web-framework none`、`--go-orm none`、および `--go-api none` とともに保持します。後で CLI が API と通信する場合は、そのクライアントを別のパッケージに保存し、`View` からではなくコマンドを通じて呼び出します。

## いつ選択するか

対話性が重要なターミナル ダッシュボード、開発者ツール、コマンド ライン インターフェイスには Bubble Tea を選択してください。

ブラウザー UI を構築せずに、選択、進行状況、フォーム、またはライブ ステータスを必要とするツールに適しています。

## テストと導入に関するメモ

モデル遷移を直接単体テストします。

```bash
go test ./...
```

配布するには、単一のバイナリをビルドし、サポートされている環境変数または構成ファイルを文書化します。

```bash
go build ./cmd/app
```

生成されたプロジェクトで別のコマンド パスが使用されている場合は、その `cmd/` ディレクトリ内のパスに従います。

## トラブルシューティング

- キーを押しても何も起こらない場合は、フォーカスされた端末が Bubble Tea が受信したキー シーケンスを送信していることを確認します。
- 画面に古い出力が残る場合は、代替画面の動作についてプログラムの起動およびシャットダウンのオプションを確認してください。
- テストを書くのが難しい場合は、副作用を `tea.Cmd` 関数に移動し、`Update` を決定性を維持します。
- 出力のラップが不十分な場合は、狭い幅でテストし、モデルが寸法を追跡するときに使用可能なスペースに応じて `View` フォーマットを維持します。
