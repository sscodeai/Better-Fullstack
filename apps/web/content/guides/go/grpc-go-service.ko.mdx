Go 서비스에 생성된 백엔드 프로젝트 형태와 함께 gRPC 계약이 필요한 경우 이 스택을 사용하세요.

```npm
npm create better-fullstack@latest my-go-grpc -- \
  --ecosystem go \
  --go-web-framework gin \
  --go-orm gorm \
  --go-api grpc-go \
  --go-cli none \
  --go-logging zap \
  --go-auth none
```

## 이것이 만드는 것

- Gin을 사용하는 Go 백엔드.
- gRPC-API 옵션으로 이동합니다.
- 데이터베이스 액세스용 GORM.
- 로깅을 위해 Zap.

## 생성된 모양

이 스택은 유형이 gRPC 경계인 Go 서비스를 생성합니다. Gin은 여전히 ​​상태 또는 운영 HTTP 엔드포인트를 제공할 수 있는 반면, gRPC-Go는 다른 백엔드 클라이언트가 사용하는 서비스 계약을 소유합니다.

다음과 같은 경우 서비스를 유지 관리하는 것이 가장 쉽습니다.

- Protobuf 파일은 요청 및 응답 계약을 정의합니다.
- gRPC 핸들러는 전송 입력을 서비스 호출로 변환합니다.
- GORM 모델은 지속성 계층에 유지됩니다.
- 진 경로는 gRPC 서비스 표면과 별도로 유지됩니다.

## 대표 파일 트리

```txt
my-go-grpc/
  go.mod
  proto/
    users/v1/users.proto
  cmd/
    server/
      main.go
  internal/
    grpc/
      users.go
    http/
      health.go
    models/
      user.go
    services/
      users.go
```

생성된 protobuf 출력을 스캐폴드가 예상하는 위치에 유지하고 생성된 파일을 직접 편집하지 마세요.

## 프로토 및 서비스 예시

protobuf 서비스는 클라이언트가 실제로 필요로 하는 계약을 모델링해야 합니다.

```proto
syntax = "proto3";

package users.v1;

option go_package = "my-go-grpc/internal/gen/users/v1;usersv1";

service UserService {
  rpc GetUser(GetUserRequest) returns (User);
}

message GetUserRequest {
  int64 id = 1;
}

message User {
  int64 id = 1;
  string email = 2;
  string name = 3;
}
```

Go 구현은 도메인 오류를 gRPC 상태 코드에 매핑해야 합니다.

```go
package grpc

import (
	"context"

	usersv1 "my-go-grpc/internal/gen/users/v1"

	"google.golang.org/grpc/codes"
	"google.golang.org/grpc/status"
)

type UserServer struct {
	usersv1.UnimplementedUserServiceServer
	Users *UserService
}

func (s *UserServer) GetUser(
	ctx context.Context,
	req *usersv1.GetUserRequest,
) (*usersv1.User, error) {
	user, err := s.Users.Get(ctx, req.GetId())
	if err != nil {
		return nil, status.Error(codes.NotFound, "user not found")
	}

	return &usersv1.User{
		Id:    user.ID,
		Email: user.Email,
		Name:  user.Name,
	}, nil
}
```

필요한 경우 운영 엔드포인트에 Gin을 사용하세요.

```go
func RegisterHTTP(router *gin.Engine) {
	router.GET("/health", func(c *gin.Context) {
		c.JSON(200, gin.H{"status": "ok"})
	})
}
```

## 호환성 참고사항

gRPC 배선을 생성하려면 `--go-api grpc-go`을 유지하세요. `--go-api none`을 변경하면 이는 일반 HTTP 백엔드가 되고 protobuf 흐름은 더 이상 스캐폴드의 일부가 아닙니다. `--go-cli none`는 프로젝트를 명령줄 도구가 아닌 서비스로 유지합니다.

## 그것을 선택하는 경우

브라우저에 표시되는 JSON 경로보다 강력한 형식의 서비스 계약이 더 중요한 백엔드 서비스의 경우 이 옵션을 선택하세요.

## 트레이드오프

gRPC은 계약 및 클라이언트 생성 워크플로를 추가합니다. 더 간단한 HTTP API의 경우 [PostgreSQL 및 GORM을 사용하는 Gin](/guides/go/gin-postgres-gorm/)으로 시작하세요.

GORM은 지속성을 접근 가능하게 유지하지만 protobuf 계약은 데이터베이스 모델을 맹목적으로 미러링해서는 안 됩니다. `.proto` 파일을 공용 서비스 계약으로 처리하고 GORM 모델을 내부 저장소 세부정보로 처리합니다.

## 테스트 및 배포 참고 사항

서비스 메서드를 직접 테스트한 다음 중요한 클라이언트 측 동작을 위해 테스트 서버와 gRPC 통합 테스트를 추가하세요.

```bash
go test ./...
```

배포 시 클라이언트가 예상하는 gRPC 포트를 노출하고 플랫폼이 프로브하는 포트에서 HTTP 상태 확인을 유지합니다. protobuf 생성이 빌드의 일부인 경우 필요한 생성기 도구가 CI에 설치되어 있는지 확인하세요.

## 문제 해결

- 클라이언트가 `unknown service`을 수신하면 gRPC 서버가 생성된 서비스 구현을 등록하는지 확인합니다.
- 생성된 protobuf 패키지가 제대로 가져오지 않는 경우 `go_package` 옵션과 스캐폴드의 출력 경로를 확인하세요.
- 상태 확인은 통과했지만 gRPC이 실패하는 경우 컨테이너 또는 플랫폼이 Gin 수신기뿐만 아니라 gRPC 수신기를 노출하는지 확인하세요.
- 데이터베이스 오류가 클라이언트로 유출되면 의도적으로 `codes.NotFound`, `codes.InvalidArgument` 또는 `codes.Internal`에 매핑하세요.
