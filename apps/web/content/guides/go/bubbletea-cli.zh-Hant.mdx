當你想要 Go 終端 UI，而不是 web API 時，就使用這個 stack。

```npm
npm create better-fullstack@latest my-go-cli -- \
  --ecosystem go \
  --go-web-framework none \
  --go-orm none \
  --go-api none \
  --go-cli bubbletea \
  --go-logging none \
  --go-auth none
```

## 創建內容

- Go CLI 項目。
- Bubble Tea 作為終端 UI 選項。
- 不包含 web framework、ORM、gRPC 或 auth add-on。

## 生成結構

這個 stack 創建的是互動式終端應用，不是 server。 Bubble Tea 會圍繞 model、message、update 和 view 組織程式。

主循環很簡單：

- model 儲存目前應用程式狀態。
- message 描述 input、tick 或 command result。
- `Update` 修改 model 並傳回後續 command。
- `View` 將目前 model 渲染為終端文字。

## 代表性文件樹

```txt
my-go-cli/
  go.mod
  cmd/
    app/
      main.go
  internal/
    tui/
      model.go
      update.go
      view.go
```

盡量把副作用放進 Bubble Tea command。這樣 model 和 view 更容易在不運行真實終端的情況下測試。

## Model 與 update 範例

Bubble Tea model 可以先只包含足夠渲染和回應輸入的狀態：

```go
package tui

type Model struct {
	Items    []string
	Selected int
	Quitting bool
}

func NewModel() Model {
	return Model{
		Items: []string{"Build", "Test", "Deploy"},
	}
}
```

update 函數負責狀態轉換：

```go
package tui

import tea "github.com/charmbracelet/bubbletea"

func (m Model) Update(msg tea.Msg) (tea.Model, tea.Cmd) {
	switch msg := msg.(type) {
	case tea.KeyMsg:
		switch msg.String() {
		case "q", "ctrl+c":
			m.Quitting = true
			return m, tea.Quit
		case "j", "down":
			if len(m.Items) > 0 {
				m.Selected = (m.Selected + 1) % len(m.Items)
			}
		case "k", "up":
			if len(m.Items) > 0 {
				m.Selected = (m.Selected - 1 + len(m.Items)) % len(m.Items)
			}
		}
	}

	return m, nil
}
```

view 應該只根據狀態渲染，不執行額外工作：

```go
package tui

import "strings"

func (m Model) View() string {
	if m.Quitting {
		return ""
	}

	var b strings.Builder
	b.WriteString("Tasks\n\n")

	for i, item := range m.Items {
		cursor := " "
		if i == m.Selected {
			cursor = ">"
		}
		b.WriteString(cursor + " " + item + "\n")
	}

	b.WriteString("\nq: quit  j/k: move\n")
	return b.String()
}
```

## 相容性說明

CLI 專案應保持 server-only category 為 `none`，除非你有意建構 hybrid tool。

為 terminal-first app 保持 `--go-cli bubbletea`、`--go-web-framework none`、`--go-orm none` 和 `--go-api none`。如果 CLI 之後要調用 API，請把 client 放在獨立 package 中，並透過 command 調用，不要從 `View` 呼叫。

## 何時選擇

當你要建立終端 dashboard、開發者工具，或需要互動體驗的命令列介面時，選擇 Bubble Tea。

它很適合需要選擇、進度、表單或即時狀態，但又不想建立瀏覽器 UI 的工具。

## 測試與分發

直接測試 model transition：

```bash
go test ./...
```

分發時建置單一 binary，並記錄支援的 environment variable 或 config file：

```bash
go build ./cmd/app
```

如果產生專案使用不同的 command 路徑，請以它的 `cmd/` 目錄為準。

## 故障排查

- 如果按鍵沒有反應，確認目前聚焦的終端發送的是 Bubble Tea 能收到的 key sequence。
- 如果退出後畫面殘留舊輸出，請檢查程式啟動和關閉時的 alternate-screen 選項。
- 如果測試難寫，把副作用移到 `tea.Cmd` 函數中，並保持 `Update` 可預測。
- 如果輸出換行很差，用窄寬度測試，並在 model 追蹤尺寸時讓 `View` 的格式適配可用空間。
