익숙한 라우터, 관계형 데이터베이스 액세스 및 실용적인 프로젝트 구조를 갖춘 Go HTTP API을 원할 때 이 스택을 사용하세요.

```npm
npm create better-fullstack@latest my-go-api -- \
  --ecosystem go \
  --go-web-framework gin \
  --database postgres \
  --go-orm gorm \
  --go-logging zap
```

## 이것이 만드는 것

- Gin을 사용하는 Go API 프로젝트.
- PostgreSQL을 데이터베이스 옵션으로 사용합니다.
- 데이터베이스 액세스용 GORM.
- 구조화된 로깅을 위한 Zap.
- 백엔드 개발을 위해 준비된 생성된 프로젝트 레이아웃입니다.

## 생성된 모양

이 스택은 일반적인 Go HTTP API를 생성합니다. Gin은 라우팅과 미들웨어를 처리하고, GORM은 모델 매핑과 지속성을 소유하고, PostgreSQL은 백업 저장소이고, Zap은 구조화된 로그를 제공합니다.

가장 깔끔하게 생성된 앱 경계는 다음과 같습니다.

- HTTP 처리기는 입력을 구문 분석하고 응답을 반환합니다.
- 서비스는 자신의 비즈니스 결정을 내립니다.
- GORM 모델은 지속된 레코드를 설명합니다.
- 데이터베이스 설정 및 로거 설정은 시작 중에 한 번 발생합니다.

## 대표 파일 트리

```txt
my-go-api/
  go.mod
  .env.example
  cmd/
    server/
      main.go
  internal/

      config.go
    database/
      postgres.go
    handlers/
      health.go
      users.go
    models/
      user.go
    services/
      users.go
```

생성된 Go 프로젝트는 옵션 세트에 따라 다르지만 이는 앱이 성장함에 따라 유지해야 하는 형태입니다. 즉, 핸들러 근처에 프레임워크 코드를 유지하고 서비스 또는 저장소 뒤에 지속성 코드를 유지합니다.

## 핸들러 및 모델 예시

Gin 핸들러는 요청 데이터를 바인딩하고, 애플리케이션 코드를 호출하고, 명확한 HTTP 상태를 반환해야 합니다.

```go
package handlers

import (
	"net/http"

	"github.com/gin-gonic/gin"
)

type CreateUserRequest struct {
	Email string `json:"email" binding:"required,email"`
	Name  string `json:"name" binding:"required"`
}

func (h *UserHandler) Create(c *gin.Context) {
	var input CreateUserRequest
	if err := c.ShouldBindJSON(&input); err != nil {
		c.JSON(http.StatusBadRequest, gin.H{"error": err.Error()})
		return
	}

	user, err := h.Users.Create(c.Request.Context(), input.Email, input.Name)
	if err != nil {
		c.JSON(http.StatusInternalServerError, gin.H{"error": "create user"})
		return
	}

	c.JSON(http.StatusCreated, user)
}
```

일치하는 GORM 모델은 작고 명시적으로 유지될 수 있습니다.

```go
package models

import "time"

type User struct {
	ID        uint      `json:"id" gorm:"primaryKey"`
	Email     string    `json:"email" gorm:"uniqueIndex;not null"`
	Name      string    `json:"name" gorm:"not null"`
	CreatedAt time.Time `json:"createdAt"`
	UpdatedAt time.Time `json:"updatedAt"`
}
```

경로 등록은 관련 엔드포인트를 함께 유지해야 합니다.

```go
func RegisterRoutes(router *gin.Engine, users *UserHandler) {
	router.GET("/health", Health)

	api := router.Group("/api")
	api.POST("/users", users.Create)
	api.GET("/users/:id", users.Get)
}
```

## 호환성 참고사항

<GuideCompatibilityNote kind="supported" title="Compatibility rule">
이 가이드의 흐름을 위해 `--go-web-framework gin`, `--database postgres` 및 `--go-orm gorm`을 함께 유지하세요. `--go-orm sqlc`을 변경하는 경우 GORM 모델 대신 SQL 파일 및 생성된 쿼리 유형을 예상하세요. `--go-web-framework none`를 설정하면 이는 비서버 프로젝트가 되며 Gin 핸들러 예제는 더 이상 적용되지 않습니다.
</GuideCompatibilityNote>

## 그것을 선택하는 경우

일반적인 미들웨어 패턴과 낮은 프레임워크 오버헤드를 갖춘 직접 Go API을 원할 경우 Gin을 선택하세요.

## 트레이드오프

Gin과 GORM은 많은 Go 팀에게 친숙한 선택입니다. 더 작은 라우터 표면이나 다른 API 스타일을 선호하는 경우 빌더에서 다른 Go 프레임워크 옵션을 비교하세요.

GORM은 CRUD가 많은 애플리케이션에 생산적이지만 SQL 세부 정보를 숨길 수 있습니다. 정확한 SQL이 계약의 일부인 쿼리 중심 서비스의 경우 Echo와 SQLC 가이드를 비교하세요.

## 테스트 및 배포 참고 사항

핸들러 테스트에는 `httptest`을 사용하고 GORM 동작에 대한 통합 테스트에는 실제 PostgreSQL 데이터베이스를 사용하세요.

```bash
go test ./...
```

배포하기 전에 생성된 앱이 환경에서 데이터베이스 URL을 읽고 마이그레이션 또는 자동 마이그레이션 동작이 팀의 릴리스 정책과 일치하는지 확인하세요. 요청 ID, 사용자 ID 및 종속성 오류에 대해 Zap 필드를 사용하면 문자열 구문 분석 없이 프로덕션 로그를 필터링할 수 있습니다.

## 문제 해결

- 요청이 400을 반환하는 경우 서비스 코드를 디버깅하기 전에 Gin 바인딩 태그를 검사하세요.
- 중복 키 오류로 인해 삽입이 실패하는 경우 GORM 인덱스와 PostgreSQL 제약 조건을 함께 확인하세요.
- 앱이 PostgreSQL에 연결할 수 없는 경우 호스트, 포트, SSL 모드 및 생성된 환경 변수 이름을 확인하세요.
- 로그가 구조화되지 않은 경우 `fmt.Println`을 사용하는 대신 Zap 로거가 처리기와 서비스에 전달되는지 확인하세요.
