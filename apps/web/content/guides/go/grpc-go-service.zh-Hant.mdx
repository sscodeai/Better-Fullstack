當你的 Go 服務需要 gRPC 契約，並且希望從一開始就獲得產生好的後端專案結構時，使用這個 stack。

```npm
npm create better-fullstack@latest my-go-grpc -- \
  --ecosystem go \
  --go-web-framework gin \
  --go-orm gorm \
  --go-api grpc-go \
  --go-cli none \
  --go-logging zap \
  --go-auth none
```

## 創建內容

- 使用 Gin 的 Go 後端。
- gRPC-Go 作為 API 選項。
- GORM 用於資料庫存取。
- Zap 用於日誌記錄。

## 生成結構

這個 stack 會建立一個帶有類型化 gRPC 邊界的 Go 服務。 Gin 仍然可以提供 health check 或其他維運 HTTP endpoint，而 gRPC-Go 負責其他後端用戶端所依賴的服務契約。

服務最容易維護的方式是：

- Protobuf 檔案定義 request 和 response 契約。
- gRPC handler 將傳輸層輸入轉換為服務呼叫。
- GORM model 留在持久化層。
- Gin route 與 gRPC 服務表面保持分離。

## 代表性文件樹

```txt
my-go-grpc/
  go.mod
  proto/
    users/v1/users.proto
  cmd/
    server/
      main.go
  internal/
    grpc/
      users.go
    http/
      health.go
    models/
      user.go
    services/
      users.go
```

把產生的 protobuf 輸出放在 scaffold 預期的位置，並避免直接編輯產生檔。

## Proto 與服務範例

protobuf service 應該描述客戶端真正需要的契約：

```proto
syntax = "proto3";

package users.v1;

option go_package = "my-go-grpc/internal/gen/users/v1;usersv1";

service UserService {
  rpc GetUser(GetUserRequest) returns (User);
}

message GetUserRequest {
  int64 id = 1;
}

message User {
  int64 id = 1;
  string email = 2;
  string name = 3;
}
```

Go 實作應把領域錯誤地對應成 gRPC 狀態碼：

```go
package grpc

import (
	"context"

	usersv1 "my-go-grpc/internal/gen/users/v1"

	"google.golang.org/grpc/codes"
	"google.golang.org/grpc/status"
)

type UserServer struct {
	usersv1.UnimplementedUserServiceServer
	Users *UserService
}

func (s *UserServer) GetUser(
	ctx context.Context,
	req *usersv1.GetUserRequest,
) (*usersv1.User, error) {
	user, err := s.Users.Get(ctx, req.GetId())
	if err != nil {
		return nil, status.Error(codes.NotFound, "user not found")
	}

	return &usersv1.User{
		Id:    user.ID,
		Email: user.Email,
		Name:  user.Name,
	}, nil
}
```

需要時可用 Gin 提供維運 endpoint：

```go
func RegisterHTTP(router *gin.Engine) {
	router.GET("/health", func(c *gin.Context) {
		c.JSON(200, gin.H{"status": "ok"})
	})
}
```

## 相容性說明

當你需要產生好的 gRPC wiring 時，保留 `--go-api grpc-go`。如果改成 `--go-api none`，項目會變成普通 HTTP 後端，protobuf flow 不再屬於 scaffold。 `--go-cli none` 讓項目保持為服務，而不是命令列工具。

## 何時選擇

當後端服務更重視強類型服務契約，而不是面向瀏覽器的 JSON route 時，選擇這個 stack。

## 取捨

gRPC 會增加契約維護和客戶端產生流程。如果只需要更簡單的 HTTP API，可以從 [Gin with PostgreSQL and GORM](/guides/go/gin-postgres-gorm/) 開始。

GORM 讓持久化層更容易上手，但 protobuf 契約不應該機械複製資料庫模型。把 `.proto` 檔案當作公開服務契約，把 GORM model 當作內部儲存細節。

## 測試與部署

先直接測試 service method，再用 test server 為重要的客戶端可見行為新增 gRPC 整合測試。

```bash
go test ./...
```

部署時暴露客戶端所需的 gRPC port，並將 HTTP health check 保持在平台探測的連接埠。如果 protobuf generation 是 build 的一部分，請確保 CI 安裝所需的 generator tool。

## 故障排查

- 如果用戶端收到 `unknown service`，確認 gRPC server 註冊了產生的服務實作。
- 如果產生的 protobuf package 無法正常 import，請檢查 `go_package` 選項和 scaffold 的輸出路徑。
- 如果 health check 正常但 gRPC 失敗，確認容器或平台暴露的是 gRPC listener，而不只是 Gin listener。
- 如果資料庫錯誤洩漏給客戶端，請明確對應為 `codes.NotFound`、`codes.InvalidArgument` 或 `codes.Internal`。
