使い慣れたルーター、リレーショナル データベース アクセス、および実用的なプロジェクト構造を備えた Go HTTP API が必要な場合は、このスタックを使用します。

```npm
npm create better-fullstack@latest my-go-api -- \
  --ecosystem go \
  --go-web-framework gin \
  --database postgres \
  --go-orm gorm \
  --go-logging zap
```

## これが生み出すもの

- Gin を使用した Go API プロジェクト。
- PostgreSQL をデータベース オプションとして使用します。
- GORM データベースアクセス用。
- 構造化されたログの Zap。
- バックエンド開発の準備が整った、生成されたプロジェクト レイアウト。

## 生成された形状

このスタックは、従来の Go HTTP API を作成します。 Gin はルーティングとミドルウェアを処理し、GORM はモデル マッピングと永続性を所有し、PostgreSQL はバッキング ストアで、Zap は構造化ログを提供します。

最もクリーンな生成アプリの境界は次のとおりです。

- HTTP ハンドラーは入力を解析し、応答を返します。
- サービスはビジネス上の決定を自ら決定します。
- GORM モデルは、永続化されたレコードを記述します。
- データベースのセットアップとロガーのセットアップは起動時に 1 回行われます。

生成される Go プロジェクトはオプション セットによって異なりますが、これはアプリの成長に合わせて保持する形状です。フレームワーク コードをハンドラーの近くに保持し、永続化コードをサービスまたはリポジトリの背後に保持します。

## ハンドラーとモデルの例

Gin ハンドラーは、リクエスト データをバインドし、アプリケーション コードを呼び出し、明確な HTTP ステータスを返す必要があります。

```go
package handlers

import (
	"net/http"

	"github.com/gin-gonic/gin"
)

type CreateUserRequest struct {
	Email string `json:"email" binding:"required,email"`
	Name  string `json:"name" binding:"required"`
}

func (h *UserHandler) Create(c *gin.Context) {
	var input CreateUserRequest
	if err := c.ShouldBindJSON(&input); err != nil {
		c.JSON(http.StatusBadRequest, gin.H{"error": err.Error()})
		return
	}

	user, err := h.Users.Create(c.Request.Context(), input.Email, input.Name)
	if err != nil {
		c.JSON(http.StatusInternalServerError, gin.H{"error": "create user"})
		return
	}

	c.JSON(http.StatusCreated, user)
}
```

一致する GORM モデルは、小さく明示的なままにすることができます。

```go
package models

import "time"

type User struct {
	ID        uint      `json:"id" gorm:"primaryKey"`
	Email     string    `json:"email" gorm:"uniqueIndex;not null"`
	Name      string    `json:"name" gorm:"not null"`
	CreatedAt time.Time `json:"createdAt"`
	UpdatedAt time.Time `json:"updatedAt"`
}
```

ルート登録では、関連するエンドポイントをまとめておく必要があります。

```go
func RegisterRoutes(router *gin.Engine, users *UserHandler) {
	router.GET("/health", Health)

	api := router.Group("/api")
	api.POST("/users", users.Create)
	api.GET("/users/:id", users.Get)
}
```

## 互換性に関する注意事項

<GuideCompatibilityNote kind="supported" title="Compatibility rule">
このガイドのフローでは、`--go-web-framework gin`、`--database postgres`、および `--go-orm gorm` をまとめておいてください。 `--go-orm sqlc` を変更する場合は、GORM モデルではなく、SQL ファイルと生成されたクエリ タイプが必要になります。 `--go-web-framework none` を設定すると、これは非サーバー プロジェクトになり、Gin ハンドラーのサンプルは適用されなくなります。
</GuideCompatibilityNote>

## いつ選択するか

一般的なミドルウェア パターンと低いフレームワーク オーバーヘッドを備えた直接 Go API が必要な場合は、Gin を選択してください。

## トレードオフ

Gin と GORM は、多くの Go チームにとって馴染みのある選択肢です。より小さいルーター表面または異なる API スタイルを好む場合は、ビルダーの他の Go フレームワーク オプションを比較してください。

GORM は CRUD を多用するアプリケーションに効果的ですが、SQL の詳細が隠蔽される可能性があります。正確な SQL が契約の一部であるクエリの多いサービスの場合は、Echo と SQLC ガイドを比較してください。

## テストと導入に関するメモ

ハンドラー テストには `httptest` を使用し、GORM 動作に関する統合テストには実際の PostgreSQL データベースを使用します。

```bash
go test ./...
```

デプロイ前に、生成されたアプリが環境からデータベース URL を読み取り、移行または自動移行の動作がチームのリリース ポリシーに一致していることを確認してください。リクエスト ID、ユーザー ID、依存関係のエラーに Zap フィールドを使用すると、文字列を解析せずに運用ログをフィルタリングできます。

## トラブルシューティング

- リクエストが 400 を返した場合は、サービス コードをデバッグする前に、Gin バインディング タグを検査してください。
- 重複キー エラーで挿入が失敗した場合は、GORM インデックスと PostgreSQL 制約を一緒に確認してください。
- アプリが PostgreSQL に接続できない場合は、ホスト、ポート、SSL モード、生成された環境変数名を確認してください。
- ログが構造化されていない場合は、`fmt.Println` を使用するのではなく、Zap ロガーがハンドラーとサービスに渡されていることを確認してください。
