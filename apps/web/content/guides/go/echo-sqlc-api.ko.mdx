Echo 및 SQL 우선 데이터베이스 액세스가 포함된 Go API을 원할 때 이 스택을 사용하세요.

```npm
npm create better-fullstack@latest my-echo-api -- \
  --ecosystem go \
  --go-web-framework echo \
  --go-orm sqlc \
  --go-api none \
  --go-cli none \
  --go-logging zap \
  --go-auth none
```

## 이것이 만드는 것

- Echo를 사용하는 Go API 프로젝트.
- Go ORM/데이터베이스 액세스 옵션인 SQLC.
- 로깅을 위해 Zap.
- 기본적으로 gRPC, CLI 또는 Go 인증 추가 기능은 없습니다.

## 생성된 모양

이 스택은 SQL 우선 Go API입니다. Echo는 HTTP 계층을 제공하고, SQLC는 SQL에서 유형화된 Go 메서드를 생성하며, Zap은 구조화된 로깅을 처리합니다.

주요 디자인 목표는 데이터베이스 동작을 가시적으로 유지하는 것입니다.

- SQL 파일은 쿼리를 정의합니다.
- SQLC는 Go 유형과 쿼리 메소드를 생성합니다.
- 처리기는 SQL을 포함하는 대신 서비스나 저장소를 호출합니다.
- Echo 미들웨어는 로깅, 복구, CORS와 같은 요청 문제를 처리합니다.

## 대표 파일 트리

```txt
my-echo-api/
  go.mod
  sqlc.yaml
  queries/
    users.sql
  migrations/
    001_create_users.sql
  cmd/
    server/
      main.go
  internal/
    db/
      generated/
    handlers/
      users.go
    services/
      users.go
```

Go 코드와 마찬가지로 SQL을 버전 제어 상태로 유지하고 쿼리 변경 사항을 검토하세요. SQLC는 Go 측을 더 안전하게 만들지만 스키마 디자인을 대체하지는 않습니다.

## 쿼리 및 핸들러 예시

SQLC 쿼리 파일은 생성된 메서드의 정보 소스입니다.

```sql
-- name: GetUser :one
select id, email, name, created_at
from users
where id = $1;

-- name: CreateUser :one
insert into users (email, name)
values ($1, $2)
returning id, email, name, created_at;
```

Echo 핸들러는 JSON을 바인딩하고 서비스를 통해 생성된 쿼리 레이어를 호출할 수 있습니다.

```go
package handlers

import (
	"net/http"
	"strconv"

	"github.com/labstack/echo/v4"
)

func (h *UserHandler) Get(c echo.Context) error {
	id, err := strconv.ParseInt(c.Param("id"), 10, 64)
	if err != nil {
		return echo.NewHTTPError(http.StatusBadRequest, "invalid user id")
	}

	user, err := h.Users.Get(c.Request().Context(), id)
	if err != nil {
		return echo.NewHTTPError(http.StatusNotFound, "user not found")
	}

	return c.JSON(http.StatusOK, user)
}
```

API 표면을 쉽게 스캔할 수 있도록 시작에 가까운 경로를 등록하세요.

```go
func RegisterRoutes(e *echo.Echo, users *UserHandler) {
	e.GET("/health", Health)

	api := e.Group("/api")
	api.GET("/users/:id", users.Get)
	api.POST("/users", users.Create)
}
```

## 호환성 참고사항

이 가이드에서는 의도적으로 `--go-api none`, `--go-cli none` 및 `--go-auth none`를 사용하므로 스캐폴드는 HTTP API에 계속 집중합니다. SQL에서 쿼리 코드를 생성하려면 `--go-orm sqlc`을 유지하세요. 서비스 경계가 JSON이 아닌 protobuf여야 하는 경우 gRPC 가이드를 사용하세요.

## 그것을 선택하는 경우

더 높은 수준의 ORM을 채택하지 않고 명시적인 SQL 및 생성된 Go 유형을 원할 경우 Echo with SQLC를 선택하세요.

## 트레이드오프

SQLC는 데이터베이스 동작을 가시적으로 유지하지만 더 많은 SQL을 소유하게 됩니다. 보다 일반적인 ORM 흐름을 원한다면 [Gin with PostgreSQL 및 GORM](/guides/go/gin-postgres-gorm/)을 사용하세요.

Echo는 가볍고 유연하므로 프로젝트 규칙이 중요합니다. 요청 검증, 서비스 호출 및 오류 매핑이 실행되는 위치에 대해 일관성을 유지하십시오.

## 테스트 및 배포 참고 사항

핸들러 및 서비스에 대해 Go 테스트를 실행하고 중요한 SQL 쿼리에 대한 데이터베이스 지원 통합 테스트를 포함합니다.

```bash
go test ./...
```

SQL이 변경되면 컴파일러 피드백을 신뢰하기 전에 생성된 프로젝트에 문서화된 명령을 사용하여 SQLC 출력을 다시 생성하십시오. 배포 시 API을 시작하기 전에 마이그레이션을 실행하여 생성된 쿼리 유형이 라이브 스키마와 일치하도록 하세요.

## 문제 해결

- 생성된 SQLC 코드가 오래된 경우 프로젝트의 SQLC 생성 명령을 다시 실행하고 스캐폴드가 생성된 파일을 추적하는 경우 SQL과 생성된 Go 변경 사항을 모두 커밋합니다.
- 쿼리가 컴파일되었지만 예상치 못한 데이터를 반환하는 경우 먼저 SQL을 확인하세요. SQLC는 제품 의미가 아닌 유형의 유효성을 검사합니다.
- Echo가 일반 텍스트 오류를 ​​반환하는 경우 응답이 일관성을 유지하도록 HTTP 오류 처리를 중앙 집중화하세요.
- `go test ./...`이 데이터베이스 테스트에서만 실패하는 경우 테스트 데이터베이스 URL과 마이그레이션 상태를 확인하세요.
