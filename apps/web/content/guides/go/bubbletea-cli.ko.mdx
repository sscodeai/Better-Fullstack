웹 API 대신 Go 터미널 UI를 원할 때 이 스택을 사용하세요.

```npm
npm create better-fullstack@latest my-go-cli -- \
  --ecosystem go \
  --go-web-framework none \
  --go-orm none \
  --go-api none \
  --go-cli bubbletea \
  --go-logging none \
  --go-auth none
```

## 이것이 만드는 것

- Go CLI 프로젝트.
- Bubble Tea을 터미널 UI 옵션으로 사용합니다.
- 웹 프레임워크, ORM, gRPC 또는 인증 추가 기능이 없습니다.

## 생성된 모양

이 스택은 서버가 아닌 대화형 터미널 앱을 생성합니다. Bubble Tea은 모델, 메시지, 업데이트 및 보기를 중심으로 프로그램을 구성합니다.

메인 루프는 간단합니다:

- 모델은 현재 애플리케이션 상태를 유지합니다.
- 메시지는 입력, 틱 또는 명령 결과를 설명합니다.
- `Update`은 모델을 변경하고 후속 명령을 반환합니다.
- `View`은 현재 모델을 터미널 텍스트로 렌더링합니다.

## 대표 파일 트리

```txt
my-go-cli/
  go.mod
  cmd/
    app/
      main.go
  internal/
    tui/
      model.go
      update.go
      view.go
```

가능한 경우 Bubble Tea 명령에 부작용을 유지하십시오. 이렇게 하면 실제 터미널을 실행하지 않고도 모델과 뷰를 더 쉽게 테스트할 수 있습니다.

## 모델 및 업데이트 예시

Bubble Tea 모델은 렌더링하고 입력에 반응하기에 충분한 상태로 시작할 수 있습니다.

```go
package tui

type Model struct {
	Items    []string
	Selected int
	Quitting bool
}

func NewModel() Model {
	return Model{
		Items: []string{"Build", "Test", "Deploy"},
	}
}
```

업데이트 함수는 상태 전환을 소유합니다.

```go
package tui

import tea "github.com/charmbracelet/bubbletea"

func (m Model) Update(msg tea.Msg) (tea.Model, tea.Cmd) {
	switch msg := msg.(type) {
	case tea.KeyMsg:
		switch msg.String() {
		case "q", "ctrl+c":
			m.Quitting = true
			return m, tea.Quit
		case "j", "down":
			if len(m.Items) > 0 {
				m.Selected = (m.Selected + 1) % len(m.Items)
			}
		case "k", "up":
			if len(m.Items) > 0 {
				m.Selected = (m.Selected - 1 + len(m.Items)) % len(m.Items)
			}
		}
	}

	return m, nil
}
```

뷰는 작업을 수행하지 않고 상태에서 렌더링되어야 합니다.

```go
package tui

import "strings"

func (m Model) View() string {
	if m.Quitting {
		return ""
	}

	var b strings.Builder
	b.WriteString("Tasks\n\n")

	for i, item := range m.Items {
		cursor := " "
		if i == m.Selected {
			cursor = ">"
		}
		b.WriteString(cursor + " " + item + "\n")
	}

	b.WriteString("\nq: quit  j/k: move\n")
	return b.String()
}
```

## 호환성 참고사항

CLI 프로젝트는 의도적으로 하이브리드 도구가 아닌 이상 서버 전용 범주를 `none`로 설정해야 합니다.

터미널 우선 앱의 경우 `--go-web-framework none`, `--go-orm none` 및 `--go-api none`과 함께 `--go-cli bubbletea`을 유지하세요. CLI가 나중에 API과 통신하는 경우 해당 클라이언트를 별도의 패키지에 보관하고 `View` 대신 명령을 통해 호출하세요.

## 그것을 선택하는 경우

대화형 기능이 중요한 터미널 대시보드, 개발자 도구 및 명령줄 인터페이스의 경우 Bubble Tea을 선택하세요.

브라우저 UI를 구축하지 않고도 선택, 진행, 양식 또는 실시간 상태가 필요한 도구에 적합합니다.

## 테스트 및 배포 참고 사항

모델 전환을 직접 단위 테스트합니다.

```bash
go test ./...
```

배포를 위해 단일 바이너리를 빌드하고 지원되는 환경 변수 또는 구성 파일을 문서화하세요.

```bash
go build ./cmd/app
```

생성된 프로젝트가 다른 명령 경로를 사용하는 경우 해당 `cmd/` 디렉터리의 경로를 따르세요.

## 문제 해결

- 키를 눌러도 아무 작업도 수행되지 않으면 초점을 맞춘 터미널이 Bubble Tea이 수신하는 키 시퀀스를 보내고 있는지 확인하세요.
- 화면에 오래된 출력이 남아 있으면 프로그램 시작 및 종료 옵션에서 대체 화면 동작을 확인하세요.
- 테스트 작성이 어렵다면 부작용을 `tea.Cmd` 함수로 옮기고 `Update` 결정성을 유지하세요.
- 출력이 제대로 래핑되지 않으면 좁은 너비에서 테스트하고 모델이 치수를 추적할 때 사용 가능한 공간에 반응하는 `View` 형식을 유지하세요.
