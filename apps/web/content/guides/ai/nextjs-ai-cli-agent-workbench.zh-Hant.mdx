當你希望 Next.js 專案一開始就支援 AI 輔助開發、本地 MCP 工具和自動產生的代理說明文件時，使用這個技術堆疊。

```npm
npm create better-fullstack@latest my-agent-workbench -- \
  --ecosystem typescript \
  --frontend next \
  --backend self \
  --database sqlite \
  --orm drizzle \
  --auth better-auth \
  --api trpc \
  --ai ai-cli \
  --addons mcp skills \
  --ai-docs claude-md agents-md cursorrules \
  --css-framework tailwind \
  --ui-library shadcn-ui \
```

## 創建內容

- 一個帶有本地全端配置的 Next.js 應用。
- SQLite 和 Drizzle。
- Better Auth。
- AI CLI 配置。
- MCP 和 Skills add-ons。
- 以代理為導向的說明文檔，例如 `CLAUDE.md`、`AGENTS.md` 或 `.cursorrules`，取決於所選 flags。

## 生成結構

這個預設既會產生應用程式本身，也會產生代理工作流程所需的配套文件。 Web 應用程式仍是標準的 Next.js 全端項目，附加項則為代碼代理程式和本地 AI 工具補充倉庫上下文。

代表性結構：

```txt
my-agent-workbench/
  bts.jsonc
  AGENTS.md
  CLAUDE.md
  .cursorrules
  app/
  components/
  lib/
    auth/
    db/
```

具體說明文件取決於 `--ai-docs claude-md agents-md cursorrules`。範例命令請求全部三種格式，這樣不同代理工具都能在自己偏好的格式中找到項目指導。

## AI 相關部分如何配合

`--ai ai-cli` 會啟用 AI CLI 支援。 `--addons mcp skills` 會加入圍繞 MCP 和 Skills 的本地代理能力。 `--ai-docs` 則控制會產生哪些倉庫層級說明檔。

這些部分各自承擔不同職責：

| 部分 | 作用 | 典型用途 |
| --- | --- | --- |
| AI CLI | 本機命令列 AI 流程 | 運行由代理輔助的專案任務 |
| MCP | 工具和上下文橋接 | 向代理暴露項目或服務能力 |
| Skills | 可復用的代理指令 | 標準化調試或功能開發等流程 |
| Agent docs | 倉庫級說明 | 告訴代理項目如何組織 |

## 範例代理指令

面向代理的說明文件應該具體。優先寫入可執行、可驗證的專案規則：

```md
## Verification

After editing authentication, run the smallest auth-related test or typecheck.
Do not run broad workspace commands unless the change crosses package boundaries.

## Project boundaries

Database access belongs in server-only modules.
Client components should call typed API helpers instead of importing Drizzle code.
```

當架構變更時，保持這些說明文件最新。過期的代理指令比沒有指令更糟，因為它們會自信地把後續工作引向錯誤檔案。

## 範例應用模型

應用側仍然是普通的 Better Fullstack TypeScript 技術堆疊。一個小型 SQLite 表可以這樣寫：

```ts
import { integer, sqliteTable, text } from "drizzle-orm/sqlite-core";

export const agentRun = sqliteTable("agent_run", {
  id: text("id").primaryKey(),
  goal: text("goal").notNull(),
  status: text("status").notNull(),
  createdAt: integer("created_at", { mode: "timestamp" }).notNull(),
});
```

應用表應該用於產品數據，不要用來存放 secrets。 MCP credentials、provider tokens 和 AI service keys 應放在環境變數或供應商專用的安全儲存中。

## 何時選擇

當倉庫本身需要容易被程式碼代理程式理解和修改時，選擇這個技術棧。它適合需要標準化代理流程、MCP 工具和生成式專案上下文的團隊。

## 相容性

- 這個預設使用 TypeScript Next.js 應用和 `--backend self`，因此 Next.js 負責服務端邊界。
- `--addons mcp skills` 是開發工作流程附加項目。它們不會取代應用的認證、資料庫或 API 決策。
- `--ai-docs claude-md agents-md cursorrules` 會有意為不同工具產生多種指令格式。
- SQLite 對本機 workbench 類別項目很方便。如果 workbench 變成具有共享狀態的多用戶生產產品，請使用 PostgreSQL。

## 部署

像普通全端應用程式一樣部署 Next.js app：設定 Better Auth secrets、app URL 和 database settings。把 AI provider keys、MCP credentials 和任何 tool tokens 都當作服務端 secrets。

代理說明文件通常應該進入版本控制。本機憑證、產生日誌和機器專屬 MCP 配置通常不應該進入。提交前檢查產生的 ignore 文件。

## 故障排查

- 如果 AI 工具忽略倉庫說明，確認它會讀取你用 `--ai-docs` 選擇產生的檔案名稱。
- 如果 MCP 工具不可用，請檢查本機 MCP 設定，以及使用它的代理是否指向該設定。
- 如果產生的 addon setup 在腳本化運行中彈出 prompts，像本指南一樣使用明確 flags，讓 defaults 可預測。
- 如果 app 本機可用但部署後 auth 失敗，請檢查任何 Next.js 技術堆疊中都要檢查的 Better Auth URL 和 cookie settings。

## 取捨

這個技術堆疊比最小 Next.js starter 更有主見。如果你只需要產品應用，不需要代理工具，使用 [Next.js with Drizzle and Better Auth](/guides/typescript/nextjs-drizzle-better-auth/)。

當人類和代理人會反覆在倉庫中協作時，額外文件很有價值。對於一次性原型，它們是額外負擔。如果團隊不會維護這些指令，先從更簡單的 Next.js guide 開始，之後再加入 AI docs。
