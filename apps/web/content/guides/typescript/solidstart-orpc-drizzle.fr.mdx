Utilisez cette pile lorsque vous souhaitez la réactivité fine de Solid avec un framework fullstack et des contrats tapés API.

```npm
npm create better-fullstack@latest my-solidstart-app -- \
  --ecosystem typescript \
  --frontend solid-start \
  --backend self \
  --database sqlite \
  --orm drizzle \
  --auth better-auth \
  --api orpc \
  --css-framework tailwind \
```

## Ce que cela crée

- Une application fullstack SolidStart.
- SQLite et Drizzle.
- Better Auth.
- oRPC pour les contrats dactylographiés.
- Tailwind CSS.

## Forme générée

Cette pile conserve SolidStart comme framework fullstack. L'application peut restituer une interface utilisateur solide, appeler des contrats de serveur tapés et conserver les données via Drizzle sans introduire de service backend distinct.

Forme représentative :

```txt
my-solidstart-app/
  bts.jsonc
  src/
    routes/
    components/
    lib/
      auth/
      db/
      rpc/
```

Utilisez cet arbre comme carte de propriété. Le code de l'interface utilisateur appartient aux composants et itinéraires Solid ; la base de données et le code d'authentification doivent rester réservés au serveur.

## Exemple de composant d'itinéraire

Les routes SolidStart peuvent rester concentrées sur le rendu tandis que le code du serveur gère l'accès aux données :

```tsx
export default function ProjectsPage() {
  return (
    <main class="mx-auto max-w-4xl p-6">
      <h1 class="text-2xl font-semibold">Projects</h1>
      <p class="text-gray-600">Create typed server calls for project data.</p>
    </main>
  );
}
```

Conservez les mutations du serveur derrière les fonctions validées afin que l'interface utilisateur ne devienne pas la seule source de vérité pour les règles de saisie.

## Exemple de modèle Drizzle

SQLite est une valeur par défaut pratique pour les petites applications SolidStart et le développement local :

```ts
import { integer, sqliteTable, text } from "drizzle-orm/sqlite-core";

export const note = sqliteTable("note", {
  id: text("id").primaryKey(),
  body: text("body").notNull(),
  ownerId: text("owner_id").notNull(),
  createdAt: integer("created_at", { mode: "timestamp" }).notNull(),
});
```

Passez à PostgreSQL lorsque plusieurs instances déployées doivent partager des données relationnelles durables.

## Remarques sur la compatibilité

SolidStart utilise oRPC dans Better Fullstack. tRPC est orienté React, donc cette pile conserve intentionnellement `--api orpc`.

`--backend self` signifie que SolidStart possède la couche serveur. N'ajoutez pas Hono sauf si vous souhaitez intentionnellement un déploiement API distinct.

## Quand le choisir

Choisissez SolidStart lorsque vous voulez le modèle d'exécution de Solid mais que vous avez toujours besoin d'un projet fullstack généré avec base de données, authentification et câblage tapé API.

## Notes de déploiement

Définissez les secrets Better Auth, l'URL de l'application et le chemin de base de données ou la chaîne de connexion pour l'environnement cible. Si votre plate-forme de déploiement ne fournit pas de disque persistant, ne comptez pas sur SQLite sauvegardé sur fichier pour les données de production.

Pour les petits outils internes, SQLite peut suffire. Pour les applications de production multi-utilisateurs de style SaaS, PostgreSQL est généralement plus facile à utiliser dans les déploiements.

## Dépannage

- Si les appels oRPC échouent après le déploiement, vérifiez le chemin de base de l'itinéraire et si l'adaptateur de déploiement gère les itinéraires du serveur comme prévu.
- Si les sessions d'authentification disparaissent, vérifiez les paramètres de sécurité des cookies et les valeurs de l'URL de l'application.
- Si les données SQLite sont réinitialisées, vérifiez si l'hôte fournit un stockage persistant.
- Si des composants Solid importent accidentellement des modules de serveur, déplacez l'appel derrière une fonction de serveur ou un contrat API.
