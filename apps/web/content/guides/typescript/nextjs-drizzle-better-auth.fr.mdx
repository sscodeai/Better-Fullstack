Utilisez cette pile lorsque vous souhaitez une application React fullstack grand public avec des données relationnelles, une authentification et une couche API typée.

```npm
npm create better-fullstack@latest my-next-app -- \
  --ecosystem typescript \
  --frontend next \
  --backend self \
  --database postgres \
  --orm drizzle \
  --auth better-auth \
  --api trpc \
  --css-framework tailwind \
  --ui-library shadcn-ui \
```

## Ce que cela crée

- Une application Next.js en mode fullstack.
- PostgreSQL comme base de données relationnelle.
- Drizzle comme ORM TypeScript.
- Better Auth pour l'authentification.
- tRPC pour les API typés de bout en bout.
- Tailwind CSS et shadcn/ui pour le style frontend.

## Forme générée

L'application générée utilise Next.js comme limite Web et serveur. PostgreSQL et Drizzle se trouvent derrière des modules serveur uniquement, tandis que tRPC fournit des appels typés pour les composants clients qui ont besoin de données d'application.

Forme représentative :

```txt
my-next-app/
  bts.jsonc
  app/
  components/
  lib/
    auth/
    db/
    trpc/
```

Les chemins exacts peuvent varier selon la version du générateur, mais la répartition prévue est cohérente : les composants de l'interface utilisateur restituent l'expérience du produit, les gestionnaires de routage et le code du serveur possèdent leurs propres limites de requête, et l'accès à la base de données reste en dehors des bundles du navigateur.

## Exemple de modèle Drizzle

Gardez les tables de produits explicites et faciles à migrer :

```ts
import { pgTable, text, timestamp } from "drizzle-orm/pg-core";

export const workspace = pgTable("workspace", {
  id: text("id").primaryKey(),
  name: text("name").notNull(),
  ownerId: text("owner_id").notNull(),
  createdAt: timestamp("created_at").defaultNow().notNull(),
});
```

Better Auth doit posséder les tables d'authentification et le comportement de session. Les tables de produits doivent référencer les ID utilisateur et éviter de copier les champs de session dans les modèles d'application.

## Exemple de procédure typée

Utilisez tRPC pour les opérations où le frontend bénéficie de types d'entrée et de sortie déduits :

```ts
import { z } from "zod";

export const createWorkspaceInput = z.object({
  name: z.string().min(2).max(80),
});

export async function createWorkspace(rawInput: unknown, userId: string) {
  const input = createWorkspaceInput.parse(rawInput);

  return {
    id: crypto.randomUUID(),
    name: input.name,
    ownerId: userId,
  };
}
```

Conservez le chèque d'autorisation à proximité de la mutation. Une opération de création d'espace de travail doit prouver qu'il existe un utilisateur connecté avant d'insérer quoi que ce soit.

## Quand le choisir

Choisissez cette option pour les applications SaaS, les produits basés sur un compte, les tableaux de bord, les portails d'administration et les projets pour lesquels la familiarité avec l'hébergement Next.js est importante.

## Pourquoi Drizzle et Better Auth

Drizzle maintient l'accès à la base de données proche de TypeScript. Better Auth donne à l'application une couche d'authentification moderne sans forcer un fournisseur d'identité hébergé par défaut.

## Remarques sur la compatibilité

- `--backend self` signifie que Next.js est propriétaire de la limite du serveur ; n'ajoutez pas de backend séparé, sauf si vous souhaitez une répartition explicite du service.
- `--database postgres` se marie bien avec l'hébergement de production, les connexions groupées et les fournisseurs de bases de données gérées.
- `--api trpc` convient parfaitement à cette pile basée sur React. Les interfaces non-React TypeScript dans Better Fullstack utilisent généralement oRPC à la place.
- shadcn/ui attend Tailwind et React. Gardez ces choix alignés si vous régénérez ou ajoutez des composants ultérieurement.

## Notes de déploiement

Définissez la chaîne de connexion PostgreSQL, les secrets Better Auth et l'URL de base de l'application dans l'environnement de déploiement. Exécutez des migrations avant d'envoyer le trafic de production vers un nouveau code qui attend de nouvelles colonnes ou tables.

Sur l'hébergement sans serveur, vérifiez les recommandations de mise en pool du fournisseur de base de données. Les connexions directes PostgreSQL peuvent rapidement dépasser les limites lorsque plusieurs instances sans serveur démarrent simultanément.

## Dépannage

- Si l'authentification fonctionne localement mais pas en production, vérifiez les URL de rappel, les origines fiables, les cookies et l'URL de l'application de production.
- Si les migrations s'exécutent mais que l'application voit toujours d'anciennes erreurs de schéma, confirmez l'application et le point de commande de migration dans la même base de données.
- Si les composants clients importent des modules de base de données serveur uniquement, déplacez la requête derrière un gestionnaire de route, une action serveur ou une procédure tapée API.
- Si les types tRPC ne sont pas mis à jour, redémarrez le serveur TypeScript et réexécutez la vérification du type de l'application ciblée.
