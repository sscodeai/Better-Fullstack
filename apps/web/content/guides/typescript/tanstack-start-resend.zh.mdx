当你的 TanStack Start app 需要 transactional email，用于 sign-in flows、product notifications、invites 或 onboarding messages 时，使用这个 stack。

```npm
npm create better-fullstack@latest my-email-app -- \
  --ecosystem typescript \
  --frontend tanstack-start \
  --backend self \
  --database sqlite \
  --orm drizzle \
  --auth better-auth \
  --email resend \
  --api trpc \
  --css-framework tailwind \
  --ui-library shadcn-ui \
```

## 创建内容

- 带自身 server route layer 的 TanStack Start app。
- Resend 作为 email provider option。
- Better Auth，带 relational database support。
- Drizzle 和 SQLite 用于 local development。
- Tailwind CSS 和 shadcn/ui 用于 interface。

## 生成结构

这个 stack 从 TanStack Start fullstack shape 开始，并添加 email provider configuration。email provider 通过 `--email resend` 选择；应用其余部分仍然使用 `--backend self`，所以不需要为了发送邮件而单独创建 API server。

代表性结构：

```txt
my-email-app/
  bts.jsonc
  src/
    routes/
    lib/
      auth/
      db/
      email/
```

Email delivery code 应放在小型 server-only helper 后面，确保 React components 永远不会导入 provider secrets。

## 使用场景

- Magic links 和 verification emails。
- Team invites。
- Trial 或 onboarding emails。
- 来自 server actions 或 API routes 的 product notifications。

## 兼容性

- `--email resend` 是 additive，不会替换 `--auth better-auth`、`--api trpc` 或 `--backend self`。
- Email delivery 必须在 server side 运行。不要通过 public environment variables 或 browser bundles 暴露 Resend API key。
- SQLite 适合 local development。如果 email workflows 依赖 durable audit trails、retries 或 team-scale records，PostgreSQL 可能更适合生产环境。
- Better Auth email flows 和 product notification flows 可以共享 provider configuration，但 function level 应保持分离。

## 部署

在部署环境中配置 Resend API key、sender domain、Better Auth secrets 和 app URL。依赖生产邮件前，请先完成 domain verification。

对于 preview deployments，如果 app 可能从不可信 preview traffic 触发邮件，请使用 restricted sender 或 test recipient allowlist。
