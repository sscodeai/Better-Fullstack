初日から別のバックエンドやデータベースを追加せずに Qwik アプリが必要な場合は、このスタックを使用します。

```npm
npm create better-fullstack@latest my-qwik-app -- \
  --ecosystem typescript \
  --frontend qwik \
  --backend none \
  --runtime none \
  --database none \
  --orm none \
  --auth none \
  --api none \
  --css-framework tailwind \
```

## これが生み出すもの

- Qwik アプリ。
- Tailwind CSS。
- デフォルトでは、データベース、ORM、認証、または API レイヤーはありません。
- 生成されたプロジェクト構成は後で拡張できます。

## 生成された形状

このガイドは意図的にフロントエンドのみの Qwik City アプリから始めます。これは、バックエンドまたは永続化レイヤーを選択する前に、アプリケーション シェル、ルーティング、およびスタイルを適切に設定する必要がある場合に便利です。

代表的な形状：

```txt
my-qwik-app/
  bts.jsonc
  src/
    routes/
    components/
```

生成された `bts.jsonc` は、バックエンド、ランタイム、データベース、ORM、認証、および API がすべて `none` であることを記録します。これにより、スタックの最小の性質が暗黙的ではなく明示的になります。

## ルート例

Qwik City ルートは小規模から始めることができます。

```tsx
import { component$ } from "@builder.io/qwik";

export default component$(() => {
  return (
    <main class="mx-auto max-w-4xl p-6">
      <h1 class="text-3xl font-semibold">Qwik app</h1>
      <p class="text-slate-600">Add server features when the product needs them.</p>
    </main>
  );
});
```

初期のルートでは UI とデータの要件に重点を置いてください。バックエンドは、認証、永続性、Webhook、プライベート API 呼び出しなど、実際のサーバー側の責任がある場合にのみ追加してください。

## 互換性に関する注意事項

Qwik の Better Fullstack におけるバックエンド/API サポートは、React、Nuxt、Svelte、または Solid オプションよりも狭いです。このガイドでは、スタックを意図的に最小限に抑え、互換性を保ちます。

このコマンドは `--backend none`、`--database none`、`--orm none`、`--auth none`、および `--api none` を使用するため、サーバー コントラクトまたはデータベース モデルを生成すべきではありません。初日からそれらが必要な場合は、より完全な Better Fullstack サポートを備えたフレームワーク/バックエンドの組み合わせを選択してください。

## いつ選択するか

初日に生成されたバックエンドよりも、再開可能性と最小限の初期ペイロードが重要であるフロントエンド ファーストのアプリには Qwik を選択してください。

## 後でバックエンドを追加する

アプリがデータを必要とする場合、別のサービスが API を所有している間、Qwik をフロントエンドとして維持するか、または別の Better Fullstack プリセットの方がフルスタック要件に適合するかを決定します。多くのチームにとって、Hono API を別のサービスとして追加することは、バックエンドの問題をフロントエンド優先のスキャフォールドに強制するよりもクリーンです。

将来的に役立つ追加機能は次のとおりです。

- プライベートサーバーロジック用の Hono API。
- PostgreSQL と、リレーショナル データの場合は Drizzle または Prisma。
- Better Auth アカウント管理が製品の一部になる場合。
- フロントエンドと API が共有型を必要とする場合は、oRPC または別の型付きコントラクト層。

## 導入メモ

最小限の Qwik アプリは、生成されたアダプターとホスティング ターゲットに応じて、Web アプリまたは静的対応フロントエンドとして簡単にデプロイできます。このコマンドはデータベースや認証を追加しないため、運用セットアップでは主にビルド​​出力、ルート、およびパブリック環境変数に重点が置かれます。

## トラブルシューティング

- データベース ファイルを期待している場合は、コマンドを確認してください。このガイドでは、意図的に `--database none` と `--orm none` を渡しています。
- 認証ルートが欠落している場合、スタックは `--auth none` を使用します。
- Tailwind クラスが適用されない場合は、生成された Tailwind コンテンツ パスとフレームワーク統合を検査します。
- 後で API を追加する場合は、パブリック クライアント変数をプライベート サーバー シークレットから分離してください。

## 比較メモ
