Utilisez cette pile lorsque vous souhaitez une application fullstack Vue/Nuxt avec des données relationnelles, une authentification et des contrats API tapés.

```npm
npm create better-fullstack@latest my-nuxt-app -- \
  --ecosystem typescript \
  --frontend nuxt \
  --backend self \
  --database postgres \
  --orm prisma \
  --auth better-auth \
  --api orpc \
  --css-framework tailwind \
```

## Ce que cela crée

- Une application Nuxt en mode fullstack.
- PostgreSQL et Prisma.
- Better Auth.
- oRPC pour les contrats tapés API.
- Tailwind CSS.
- Une configuration de pile `bts.jsonc` reproductible.

## Forme générée

Cette pile conserve l'application dans le modèle fullstack de Nuxt. Prisma possède le schéma de base de données et les migrations, Better Auth possède le comportement d'authentification et oRPC donne des contrats tapés sans assumer une interface React.

Forme représentative :

```txt
my-nuxt-app/
  bts.jsonc
  app/
  server/
  prisma/
    schema.prisma
```

Traitez cela comme une mise en page représentative. La limite importante est que la base de données serveur uniquement et le code d'authentification restent dans les modules serveur Nuxt ou les gestionnaires API, tandis que les composants Vue appellent des contrats de serveur typés.

## Exemple de modèle Prisma

Un petit modèle de produit peut commencer comme ceci :

```prisma
model Workspace {
  id        String   @id
  name      String
  ownerId   String
  createdAt DateTime @default(now())
}
```

Conservez les modèles requis par Better Auth alignés sur la documentation de l'adaptateur et séparés des tableaux spécifiques à l'application. Les tables de produits doivent référencer les utilisateurs autorisés par identifiant plutôt que de dupliquer les données de connexion.

## Exemple de forme de contrat oRPC

Utilisez oRPC pour décrire les opérations de serveur validées pour l'application Nuxt :

```ts
import { z } from "zod";

export const createWorkspaceInput = z.object({
  name: z.string().min(2),
});

export async function createWorkspace(rawInput: unknown, userId: string) {
  const input = createWorkspaceInput.parse(rawInput);

  return {
    id: crypto.randomUUID(),
    name: input.name,
    ownerId: userId,
  };
}
```

Placez les contrôles d'autorisation dans l'implémentation du serveur, pas seulement dans les composants Vue. Les protections côté client améliorent l'UX mais ne protègent pas les données.

## Remarques sur la compatibilité

Nuxt utilise oRPC dans Better Fullstack pour les contrats tapés. tRPC est réservé aux interfaces TypeScript basées sur React, donc ce guide utilise intentionnellement `--api orpc`.

`--backend self` est le bon choix fullstack Nuxt. Si vous souhaitez un serveur API déployé séparément, utilisez une option backend telle que Hono dans une pile prenant en charge ce couplage.

## Quand le choisir

Choisissez Nuxt lorsque votre équipe préfère Vue, souhaite un routage compatible avec le serveur et a besoin d'une forme d'application fullstack plutôt qu'une interface distincte et un serveur API.

## Notes de déploiement

Définissez la chaîne de connexion PostgreSQL, les secrets Better Auth et l'URL de l'application publique dans l'environnement de déploiement. Exécutez les migrations Prisma avant de déployer du code qui dépend du nouveau schéma.

Pour les plates-formes sans serveur, vérifiez comment votre client Prisma est généré et comment les connexions à la base de données sont regroupées. Les fournisseurs PostgreSQL gérés documentent souvent la chaîne de connexion la plus sûre pour les environnements d'exécution sans serveur.

## Dépannage

- Si les types de clients Prisma sont obsolètes, régénérez le client après avoir modifié `schema.prisma`.
- Si les appels oRPC échouent uniquement en production, vérifiez l’URL de base de l’itinéraire du serveur et le comportement de l’adaptateur de déploiement.
- Si l'authentification est redirigée en boucle, vérifiez les origines fiables, les cookies et l'URL de l'application utilisée par Better Auth.
- Si les migrations pointent vers la mauvaise base de données, inspectez la variable d'environnement utilisée à la fois par les commandes locales et par le runtime déployé.
