Usa este stack cuando quieras Astro para paginas con mucho contenido e islas, junto con un servidor API TypeScript.

```npm
npm create better-fullstack@latest my-astro-app -- \
  --ecosystem typescript \
  --frontend astro \
  --astro-integration react \
  --backend hono \
  --runtime bun \
  --database sqlite \
  --orm drizzle \
  --auth better-auth \
  --api trpc \
  --css-framework tailwind \
  --ui-library daisyui \
```

## Que crea

- Un frontend Astro con integracion React.
- Un backend Hono que corre en Bun.
- SQLite y Drizzle.
- Better Auth.
- tRPC, porque esta configuracion de Astro usa React.
- Tailwind CSS y DaisyUI.

## Forma generada

Este stack combina el frontend centrado en contenido de Astro con un servidor API Hono separado. Las islas React quedan habilitadas por `--astro-integration react`, por eso tRPC esta disponible en esta guia.

Forma representativa:

```txt
my-astro-app/
  bts.jsonc
  apps/
    web/
    server/
  packages/
    shared/
```

Las paginas Astro deberian centrarse en rutas, contenido y composicion de islas. Hono deberia hacerse cargo de rutas API, comportamiento de servidor con auth y acceso a la base de datos.

## Patron de uso

Usa Astro para el contorno de la pagina y React solo donde haga falta interactividad. Eso mantiene baratas las paginas estaticas o casi estaticas, y aun permite que los componentes interactivos llamen a la API tipada.

Los endpoints sencillos y los health checks pertenecen en Hono. Para endpoints con base de datos, deja las consultas Drizzle detras de helpers solo de servidor y exponlas mediante procedimientos tRPC o rutas Hono.

## Compatibilidad

La compatibilidad de API en Astro depende de la integracion de UI elegida. Esta pagina usa `--astro-integration react`, lo que mantiene disponible tRPC. Para integraciones Astro que no sean React, prefiere oRPC.

DaisyUI se selecciona como libreria UI en este comando. Si luego cambias a shadcn/ui, revisa que la integracion Astro React y Tailwind sigan alineadas con la libreria de componentes.

## Cuando elegirlo

Elige Astro cuando contenido, docs, paginas de marketing o hidratacion parcial sean centrales para la app, pero aun necesites un backend generado y una base de datos.

## Despliegue

Planifica origenes claros para el frontend Astro y la API Hono. El frontend necesita la URL publica de la API. El servicio Hono necesita secretos de base de datos y auth. SQLite puede servir para desarrollo local, pero produccion necesita almacenamiento persistente si decides conservar SQLite.

Si frontend y API se despliegan por separado, configura CORS y cookies de auth para los dominios finales, no para el comportamiento de localhost.

## Solucion de problemas

- Si las islas React no hidratan, revisa la directiva cliente de Astro y que el componente importe codigo de navegador de forma segura.
- Si tRPC deja de estar disponible tras cambiar integraciones, confirma que la integracion Astro siga siendo React.
- Si las llamadas API fallan desde paginas desplegadas, verifica el origen publico de la API y la configuracion CORS.
- Si los datos SQLite desaparecen tras desplegar, confirma que tu hosting ofrece almacenamiento persistente.
