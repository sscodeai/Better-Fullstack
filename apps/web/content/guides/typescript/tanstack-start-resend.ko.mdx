TanStack Start 앱에 로그인 흐름, 제품 알림, 초대 또는 온보딩 메시지를 위한 거래 이메일이 필요한 경우 이 스택을 사용하세요.

```npm
npm create better-fullstack@latest my-email-app -- \
  --ecosystem typescript \
  --frontend tanstack-start \
  --backend self \
  --database sqlite \
  --orm drizzle \
  --auth better-auth \
  --email resend \
  --api trpc \
  --css-framework tailwind \
  --ui-library shadcn-ui \
```

## 이것이 만드는 것

- 자체 서버 경로 계층이 있는 TanStack Start 앱입니다.
- 이메일 제공업체 옵션으로 다시 보냅니다.
- Better Auth 관계형 데이터베이스를 지원합니다.
- 로컬 개발을 위한 Drizzle 및 SQLite
- 인터페이스의 경우 Tailwind CSS 및 shadcn/ui입니다.

## 생성된 모양

이 스택은 TanStack Start 풀 스택 형태에서 시작하고 이메일 공급자 구성을 추가합니다. 이메일 제공자는 `--email resend`로 선택됩니다. 앱의 나머지 부분은 여전히 ​​`--backend self`을 사용하므로 단지 메일을 보내기 위해 별도의 API 서버가 필요하지 않습니다.

대표적인 형태:

```txt
my-email-app/
  bts.jsonc
  src/
    routes/
    lib/
      auth/
      db/
      email/
```

위의 트리를 정확한 생성 경로에 대한 약속이 아니라 소유권에 대한 가이드로 취급하십시오. 이메일 전달 코드는 작은 서버 전용 도우미 뒤에 있어야 하므로 React 구성 요소는 공급자 비밀을 가져오지 않습니다.

## 이메일 도우미 예시

좋은 첫 번째 추상화는 각 제품 이메일에 대한 좁은 기능입니다. 이는 인증 콜백, 온보딩 흐름 및 경로 핸들러에서 재전송 관련 세부 정보를 유지합니다.

```ts
type SendInviteEmailInput = {
  to: string;
  inviterName: string;
  inviteUrl: string;
};

export async function sendInviteEmail(input: SendInviteEmailInput) {
  const subject = `${input.inviterName} invited you`;

  return sendTransactionalEmail({
    to: input.to,
    subject,
    html: `
      <p>${input.inviterName} invited you to join their workspace.</p>
      <p><a href="${input.inviteUrl}">Accept the invite</a></p>
    `,
  });
}
```

구체적인 `sendTransactionalEmail` 구현은 서버 환경 변수에서 API 재전송 키를 읽어야 하며 클라이언트 구성 요소에서 가져오면 안 됩니다.

## 경로 사용 예시

내구성 있는 제품 상태를 생성하는 데이터베이스 트랜잭션 후에 메일을 보냅니다. 초대의 경우 먼저 초대 레코드를 생성한 후 이메일을 보냅니다.

```ts
import { z } from "zod";

const inviteInput = z.object({
  email: z.string().email(),
  workspaceId: z.string().min(1),
});

export async function inviteMember(rawInput: unknown) {
  const input = inviteInput.parse(rawInput);

  const invite = {
    id: crypto.randomUUID(),
    email: input.email,
    workspaceId: input.workspaceId,
  };

  await sendInviteEmail({
    to: invite.email,
    inviterName: "A teammate",
    inviteUrl: `https://example.com/invites/${invite.id}`,
  });

  return invite;
}
```

프로덕션 시스템의 경우 공급자에게 일시적인 오류가 발생하더라도 전달되어야 하는 이메일에 대한 작업 대기열 또는 재시도 테이블을 고려하세요.

## 그것을 선택하는 경우

이메일이 나중에 고려되는 것이 아니라 제품 워크플로의 일부인 경우 이 옵션을 선택하십시오. 재전송은 개발자 중심의 트랜잭션 이메일에 매우 적합하며 생성된 스택을 간단하게 유지합니다.

## 일반적인 사용 사례

- 매직 링크 및 확인 이메일.
- 팀을 초대합니다.
- 평가판 또는 온보딩 이메일.
- 서버 작업 또는 API 경로의 제품 알림입니다.

## 호환성 참고사항

- `--email resend`은 추가됩니다. `--auth better-auth`, `--api trpc` 또는 `--backend self`을 대체하지 않습니다.
- 이메일 전달은 서버 측에서 실행되어야 합니다. 공개 환경 변수나 브라우저 번들을 통해 Resend API 키를 노출하지 마세요.
- SQLite은 로컬 개발에 편리합니다. 이메일 워크플로가 내구성 있는 감사 추적, 재시도 또는 팀 규모 기록에 의존하는 경우 PostgreSQL이 더 나은 프로덕션 데이터베이스일 수 있습니다.
- Better Auth 이메일 흐름과 제품 알림 흐름은 공급자 구성을 공유해야 하지만 기능 수준에서는 별도로 유지되어야 합니다.

## 배포 참고사항

배포 환경에서 재전송 API 키, 보낸 사람 도메인, Better Auth 비밀 및 앱 URL을 구성합니다. 프로덕션 메일 전달에 의존하기 전에 도메인 확인을 완료해야 합니다.

미리 보기 배포의 경우 앱이 신뢰할 수 없는 미리 보기 트래픽에서 이메일을 트리거할 수 있는 경우 제한된 발신자 또는 테스트 수신자 허용 목록을 사용하세요.

## 문제 해결

- 이메일이 도착하지 않으면 애플리케이션 코드를 변경하기 전에 공급자 로그를 확인하세요.
- 링크가 localhost를 가리키는 경우 콜백 또는 초대 링크를 구축할 때 사용되는 프로덕션 앱 URL을 업데이트하세요.
- 인증 이메일은 작동하지만 제품 이메일은 실패하는 경우 두 경로 모두 동일한 서버 전용 공급자 구성을 사용하는지 확인하세요.
- 로컬 테스트에서 실수로 실제 이메일을 보낸 경우 개발 전송을 추가하거나 비프로덕션 환경을 보호하세요.

## 비교 참고 사항

아직 이메일이 필요하지 않은 경우 더 광범위한 [TanStack Start 가이드](/guides/typescript/create-tanstack-start-project/)를 사용하세요. 거래 이메일이 초기 제품 계약의 일부인 경우 이 페이지의 명령을 선택하세요. 첫 날부터 해당 결정이 `bts.jsonc`에 기록되기 때문입니다.
