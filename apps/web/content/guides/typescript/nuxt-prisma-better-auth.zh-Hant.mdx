當你想要帶有 relational data、authentication 和 typed API contracts 的 Vue/Nuxt fullstack app 時，使用這個 stack。

```npm
npm create better-fullstack@latest my-nuxt-app -- \
  --ecosystem typescript \
  --frontend nuxt \
  --backend self \
  --database postgres \
  --orm prisma \
  --auth better-auth \
  --api orpc \
  --css-framework tailwind \
```

## 創建內容

- 一個 fullstack mode 的 Nuxt app。
- PostgreSQL 和 Prisma。
- Better Auth。
- 用於 typed API contracts 的 oRPC。
- Tailwind CSS。
- 可複現的 `bts.jsonc` stack configuration。

## 生成結構

這個 stack 把應用程式保持在 Nuxt fullstack model 中。 Prisma 负责 database schema 和 migrations，Better Auth 负责 authentication behavior，oRPC 提供 typed contracts，并且不假设你使用 React frontend。

代表性結構：

```txt
my-nuxt-app/
  bts.jsonc
  app/
  server/
  prisma/
    schema.prisma
```

把它看作代表性 layout。关键边界是：server-only database 和 auth code 留在 Nuxt server modules 或 API handlers 中，而 Vue components 调用 typed server contracts。

## 範例 Prisma model

一個小的 product model 可以這樣開始：

```prisma
model Workspace {
  id        String   @id
  name      String
  ownerId   String
  createdAt DateTime @default(now())
}
```

Better Auth 所需的 models 應與 adapter 文件保持一致，並與產品自己的表分開。 Product tables 應透過 identifier 引用 auth users，不要複製 login data。

## 範例 oRPC contract shape

使用 oRPC 描述 Nuxt app 的 validated server operations：

```ts
import { z } from "zod";

export const createWorkspaceInput = z.object({
  name: z.string().min(2),
});

export async function createWorkspace(rawInput: unknown, userId: string) {
  const input = createWorkspaceInput.parse(rawInput);

  return {
    id: crypto.randomUUID(),
    name: input.name,
    ownerId: userId,
  };
}
```

Authorization checks 要放在 server implementation 中，而不是只是在 Vue components 裡。 Client-side guards 能改善 UX，但無法保護資料。

## 相容性

Nuxt 在 Better Fullstack 中使用 oRPC 做 typed contracts。 tRPC 保留給 React-based TypeScript frontends，因此本指南有意使用 `--api orpc`。

`--backend self` 是 Nuxt fullstack 的正確選擇。如果想要單獨部署的 API server，請在支援該組合的 stack 中使用 Hono 等 backend option。

## 何時選擇

當團隊偏好 Vue、需要 server-capable routing，並且想要 fullstack app shape 而不是獨立 frontend 與 API server 時，選擇 Nuxt。

## 部署

在部署環境中設定 PostgreSQL connection string、Better Auth secrets 和 public app URL。在部署依賴新 schema 的程式碼之前，先執行 Prisma migrations。

對於 serverless platforms，檢查 Prisma client 的產生方式和 database connections 的 pooling 方式。 Managed PostgreSQL providers 通常會說明 serverless runtimes 最安全的 connection string。

## 故障排查

- 如果 Prisma Client types 過期，修改 `schema.prisma` 後重新產生 client。
- 如果 oRPC calls 只在 production 失敗，請檢查 server route base URL 和 deployment adapter behavior。
- 如果 auth redirects 循環，檢查 trusted origins、cookies，以及 Better Auth 使用的 app URL。
- 如果 migrations 指向錯誤 database，檢查 local commands 和 deployed runtime 使用的 environment variable。
