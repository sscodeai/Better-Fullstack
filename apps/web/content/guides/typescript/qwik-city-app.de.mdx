Verwenden Sie diesen Stack, wenn Sie eine Qwik-App benötigen, ohne am ersten Tag ein separates Backend oder eine Datenbank hinzuzufügen.

```npm
npm create better-fullstack@latest my-qwik-app -- \
  --ecosystem typescript \
  --frontend qwik \
  --backend none \
  --runtime none \
  --database none \
  --orm none \
  --auth none \
  --api none \
  --css-framework tailwind \
```

## Was dadurch entsteht

- Eine Qwik-App.
- Tailwind CSS.
- Standardmäßig keine Datenbank-, ORM-, Authentifizierungs- oder API-Ebene.
- Eine generierte Projektkonfiguration können Sie später erweitern.

## Erzeugte Form

Dieser Leitfaden beginnt absichtlich mit einer reinen Frontend-App Qwik City. Dies ist nützlich, wenn Sie die Anwendungs-Shell, das Routing und das Styling festlegen möchten, bevor Sie ein Backend oder eine Persistenzschicht auswählen.

Repräsentative Form:

```txt
my-qwik-app/
  bts.jsonc
  src/
    routes/
    components/
```

Die generierten `bts.jsonc`-Datensätze für Backend, Laufzeit, Datenbank, ORM, Authentifizierung und API sind alle `none`. Dadurch wird die minimale Natur des Stapels explizit statt implizit.

## Beispielroute

Qwik City Routen können klein anfangen:

```tsx
import { component$ } from "@builder.io/qwik";

export default component$(() => {
  return (
    <main class="mx-auto max-w-4xl p-6">
      <h1 class="text-3xl font-semibold">Qwik app</h1>
      <p class="text-slate-600">Add server features when the product needs them.</p>
    </main>
  );
});
```

Konzentrieren Sie sich bei frühen Routen auf die Anforderungen der Benutzeroberfläche und der Daten. Fügen Sie ein Backend nur hinzu, wenn eine echte serverseitige Verantwortung besteht, z. B. Authentifizierung, Persistenz, Webhooks oder private API-Aufrufe.

## Kompatibilitätshinweise

Qwik bietet in Better Fullstack eine engere Backend-/API-Unterstützung als die Optionen React, Nuxt, Svelte oder Solid. Dieser Leitfaden hält den Stapel absichtlich minimal und kompatibel.

Da der Befehl `--backend none`, `--database none`, `--orm none`, `--auth none` und `--api none` verwendet, sollte er keine Serververträge oder Datenbankmodelle generieren. Wenn Sie diese vom ersten Tag an benötigen, wählen Sie eine Framework/Backend-Paarung mit umfassenderer Better Fullstack-Unterstützung.

## Wann man es wählt

Wählen Sie Qwik für Frontend-First-Apps, bei denen Wiederverwendbarkeit und eine minimale anfängliche Nutzlast wichtiger sind als ein generiertes Backend am ersten Tag.

## Später ein Backend hinzufügen

Wenn die App Daten benötigt, entscheiden Sie, ob Qwik als Frontend bleiben soll, während ein anderer Dienst das API besitzt, oder ob eine andere Better Fullstack-Voreinstellung den Fullstack-Anforderungen besser entsprechen würde. Für viele Teams ist das Hinzufügen eines Hono API als separaten Dienst sauberer, als Backend-Anliegen in ein Frontend-First-Gerüst zu zwingen.

Zu den nützlichen zukünftigen Ergänzungen gehören:

- Ein Hono API für private Serverlogik.
- PostgreSQL plus Drizzle oder Prisma für relationale Daten.
- Better Auth, wenn die Kontoverwaltung Teil des Produkts wird.
- oRPC oder eine andere typisierte Vertragsschicht, wenn das Frontend und API gemeinsame Typen benötigen.

## Bereitstellungshinweise

Eine minimale Qwik-App lässt sich je nach generiertem Adapter und Hosting-Ziel problemlos als Web-App oder statisch fähiges Frontend bereitstellen. Da dieser Befehl keine Datenbank oder Authentifizierung hinzufügt, konzentriert sich die Produktionseinrichtung hauptsächlich auf Build-Ausgabe, Routen und öffentliche Umgebungsvariablen.

## Fehlerbehebung

- Wenn Sie Datenbankdateien erwartet haben, überprüfen Sie den Befehl: Diese Anleitung übergibt absichtlich `--database none` und `--orm none`.
- Wenn Authentifizierungsrouten fehlen, verwendet der Stack `--auth none`.
- Wenn Tailwind-Klassen nicht zutreffen, überprüfen Sie die generierten Tailwind-Inhaltspfade und die Framework-Integration.
- Wenn Sie später einen API hinzufügen, halten Sie öffentliche Clientvariablen von privaten Servergeheimnissen getrennt.
