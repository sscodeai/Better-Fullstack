當你想要主流 React fullstack app，並包含 relational data、authentication 和 typed API layer 時，使用這個 stack。

```npm
npm create better-fullstack@latest my-next-app -- \
  --ecosystem typescript \
  --frontend next \
  --backend self \
  --database postgres \
  --orm drizzle \
  --auth better-auth \
  --api trpc \
  --css-framework tailwind \
  --ui-library shadcn-ui \
```

## 創建內容

- 一個 fullstack mode 的 Next.js app。
- PostgreSQL 作為 relational database。
- Drizzle 作為 TypeScript ORM。
- Better Auth 用於 authentication。
- tRPC 用於 end-to-end typed APIs。
- Tailwind CSS 和 shadcn/ui 用於 frontend styling。

## 生成結構

產生的 app 使用 Next.js 作為 web 和 server boundary。 PostgreSQL 與 Drizzle 位於 server-only modules 後面，而 tRPC 為需要 application data 的 client components 提供 typed calls。

代表性結構：

```txt
my-next-app/
  bts.jsonc
  app/
  components/
  lib/
    auth/
    db/
    trpc/
```

確切路徑會隨 generator 版本變化，但意圖一致：UI components 渲染產品體驗，route handlers 和 server code 負責 request boundaries，database access 不進入 browser bundles。

## 範例 Drizzle model

保持 product tables 明確且易於 migrate：

```ts
import { pgTable, text, timestamp } from "drizzle-orm/pg-core";

export const workspace = pgTable("workspace", {
  id: text("id").primaryKey(),
  name: text("name").notNull(),
  ownerId: text("owner_id").notNull(),
  createdAt: timestamp("created_at").defaultNow().notNull(),
});
```

Better Auth 應負責 authentication tables 和 session behavior。 Product tables 應引用 user IDs，避免把 session fields 複製進 application models。

## 範例 typed procedure

對於 frontend 能從 inferred input/output types 受益的操作，使用 tRPC：

```ts
import { z } from "zod";

export const createWorkspaceInput = z.object({
  name: z.string().min(2).max(80),
});

export async function createWorkspace(rawInput: unknown, userId: string) {
  const input = createWorkspaceInput.parse(rawInput);

  return {
    id: crypto.randomUUID(),
    name: input.name,
    ownerId: userId,
  };
}
```

Authorization check 應靠近 mutation。在建立 workspace 前，應證明存在 signed-in user。

## 何時選擇

適用於 SaaS apps、account-based products、dashboards、admin portals，以及重視 Next.js hosting familiarity 的項目。

## 為什麼選擇 Drizzle 和 Better Auth

Drizzle 讓 database access 貼近 TypeScript。 Better Auth 提供現代 auth layer，同時預設不強迫使用 hosted identity provider。

## 相容性

- `--backend self` 表示 Next.js 負責 server boundary；除非你明確想拆分服務，否則不要加入獨立 backend。
- `--database postgres` 適合 production hosting、pooled connections 和 managed database providers。
- `--api trpc` 很適合這個 React-based stack。 Better Fullstack 中非 React TypeScript frontends 通常使用 oRPC。
- shadcn/ui 期待 Tailwind 和 React。如果之後重新產生或新增元件，保持這些選擇一致。

## 部署

在部署環境中設定 PostgreSQL connection string、Better Auth secrets 和 app base URL。在把依賴新 columns 或 tables 的程式碼導入 production traffic 前，先執行 migrations。

在 serverless hosting 上，檢查 database provider 的 pooling recommendation。許多 serverless instances 同時啟動時，直接 PostgreSQL connections 很快會耗盡 limits。

## 故障排查

- 如果 auth 本地可用但 production 不可用，請檢查 callback URLs、trusted origins、cookies 和 production app URL。
- 如果 migrations 運作成功但 app 仍看到舊 schema errors，確認 app 和 migration command 指向同一個 database。
- 如果 client components 導入 server-only database modules，把 query 移到 route handler、server action 或 typed API procedure 後面。
- 如果 tRPC types 沒有更新，重新啟動 TypeScript server，並重新執行目標 app 的 typecheck。

## 對比說明

當你想要最廣泛的 Next.js deployment examples 和 ecosystem familiarity 時，選擇這個 stack 而不是 TanStack Start。當 TanStack Router 和更小的 React fullstack framework 比 Next.js conventions 更重要時，選擇 [TanStack Start](/guides/typescript/create-tanstack-start-project/)。
