當你的 TanStack Start app 需要 transactional email，用於 sign-in flows、product notifications、invites 或 onboarding messages 時，使用這個 stack。

```npm
npm create better-fullstack@latest my-email-app -- \
  --ecosystem typescript \
  --frontend tanstack-start \
  --backend self \
  --database sqlite \
  --orm drizzle \
  --auth better-auth \
  --email resend \
  --api trpc \
  --css-framework tailwind \
  --ui-library shadcn-ui \
```

## 創建內容

- 一個帶有自身 server route layer 的 TanStack Start app。
- Resend 作為 email provider option。
- 帶有 relational database support 的 Better Auth。
- 用於 local development 的 Drizzle 和 SQLite。
- 用於介面的 Tailwind CSS 和 shadcn/ui。

## 生成結構

這個 stack 從 TanStack Start fullstack shape 開始，並且加入 email provider configuration。 Email provider 透過 `--email resend` 選擇；應用其餘部分仍然使用 `--backend self`，因此不需要為了發送郵件再加一個獨立 API server。

代表性結構：

```txt
my-email-app/
  bts.jsonc
  src/
    routes/
    lib/
      auth/
      db/
      email/
```

把上面的 tree 看作 ownership 指南，而不是確切路徑承諾。 Email delivery code 應放在小型 server-only helper 後面，避免 React components 導入 provider secrets。

## 範例 email helper

好的第一層抽象化是為每種 product email 建立一個窄函數。這樣 Resend-specific details 不會散落到 auth callbacks、onboarding flows 和 route handlers 中：

```ts
type SendInviteEmailInput = {
  to: string;
  inviterName: string;
  inviteUrl: string;
};

export async function sendInviteEmail(input: SendInviteEmailInput) {
  const subject = `${input.inviterName} invited you`;

  return sendTransactionalEmail({
    to: input.to,
    subject,
    html: `
      <p>${input.inviterName} invited you to join their workspace.</p>
      <p><a href="${input.inviteUrl}">Accept the invite</a></p>
    `,
  });
}
```

特定的 `sendTransactionalEmail` implementation 應從 server environment variables 讀取 Resend API key，並且不應被 client components 導入。

## 範例 route usage

在建立 durable product state 的 database transaction 之後再發送郵件。對於 invites，先建立 invite record，再發送 email：

```ts
import { z } from "zod";

const inviteInput = z.object({
  email: z.string().email(),
  workspaceId: z.string().min(1),
});

export async function inviteMember(rawInput: unknown) {
  const input = inviteInput.parse(rawInput);

  const invite = {
    id: crypto.randomUUID(),
    email: input.email,
    workspaceId: input.workspaceId,
  };

  await sendInviteEmail({
    to: invite.email,
    inviterName: "A teammate",
    inviteUrl: `https://example.com/invites/${invite.id}`,
  });

  return invite;
}
```

生產系統中，如果郵件必須在 provider 暫時失敗後仍能送達，請考慮使用 job queue 或 retry table。

## 何時選擇

當 email 是產品 workflow 的一部分，而不是事後補充時，選擇這個 stack。 Resend 很適合 developer-focused transactional email，並能讓產生的 stack 保持直接。

## 常見使用場景

- Magic links 和 verification emails。
- Team invites。
- Trial 或 onboarding emails。
- 來自 server actions 或 API routes 的 product notifications。

## 相容性

- `--email resend` 是 additive。它不會取代 `--auth better-auth`、`--api trpc` 或 `--backend self`。
- Email delivery 必須在 server side 運作。不要透過 public environment variables 或 browser bundles 暴露 Resend API key。
- SQLite 適合 local development。如果 email workflows 依賴 durable audit trails、retries 或 team-scale records，PostgreSQL 可能是更好的生產資料庫。
- Better Auth email flows 和 product notification flows 應共用 provider configuration，但在函數層級保持分離。

## 部署

在部署環境中設定 Resend API key、sender domain、Better Auth secrets 和 app URL。在依賴 production mail delivery 之前，應完成 domain verification。

對於 preview deployments，如果 app 可能從不可信 preview traffic 觸發郵件，使用受限 sender 或 test recipient allowlist。

## 故障排查

- 如果 emails 沒有到達，先檢查 provider logs，再改 application code。
- 如果 links 指向 localhost，更新用於建置 callback 或 invite links 的 production app URL。
- 如果 auth emails 可用但 product emails 失敗，確認兩條路徑使用同一個 server-only provider configuration。
- 如果 local tests 意外發送真實郵件，請添加 development transport 或 non-production environment guard。

## 對比說明

暫時不需要 email 時，使用更廣泛的 [TanStack Start guide](/guides/typescript/create-tanstack-start-project/)。當 transactional email 是初始 product contract 的一部分時，請選擇本頁指令，因為它會從第一天就在 `bts.jsonc` 中記錄這個決策。
