Utilisez cette pile lorsque vous souhaitez une application Qwik sans ajouter un backend ou une base de données distincte dès le premier jour.

```npm
npm create better-fullstack@latest my-qwik-app -- \
  --ecosystem typescript \
  --frontend qwik \
  --backend none \
  --runtime none \
  --database none \
  --orm none \
  --auth none \
  --api none \
  --css-framework tailwind \
```

## Ce que cela crée

- Une application Qwik.
- Tailwind CSS.
- Aucune couche de base de données, ORM, d'authentification ou API par défaut.
- Une configuration de projet générée que vous pouvez étendre ultérieurement.

## Forme générée

Ce guide commence intentionnellement par une application Qwik City frontend uniquement. Ceci est utile lorsque vous souhaitez que le shell, le routage et le style de l'application soient en place avant de choisir une couche backend ou de persistance.

Forme représentative :

```txt
my-qwik-app/
  bts.jsonc
  src/
    routes/
    components/
```

Les enregistrements `bts.jsonc` générés pour le backend, l'exécution, la base de données, l'ORM, l'authentification et API sont tous `none`. Cela rend la nature minimale de la pile explicite plutôt qu'implicite.

## Exemple d'itinéraire

Les itinéraires Qwik City peuvent commencer petit :

```tsx
import { component$ } from "@builder.io/qwik";

export default component$(() => {
  return (
    <main class="mx-auto max-w-4xl p-6">
      <h1 class="text-3xl font-semibold">Qwik app</h1>
      <p class="text-slate-600">Add server features when the product needs them.</p>
    </main>
  );
});
```

Gardez les premiers itinéraires axés sur les exigences en matière d’interface utilisateur et de données. Ajoutez un backend uniquement lorsqu'il existe une véritable responsabilité côté serveur telle que l'authentification, la persistance, les webhooks ou les appels privés API.

## Remarques sur la compatibilité

Qwik a une prise en charge backend/API plus étroite dans Better Fullstack que les options React, Nuxt, Svelte ou Solid. Ce guide maintient la pile intentionnellement minimale et compatible.

Étant donné que la commande utilise `--backend none`, `--database none`, `--orm none`, `--auth none` et `--api none`, elle ne doit pas générer de contrats de serveur ou de modèles de base de données. Si vous en avez besoin dès le premier jour, choisissez un couple framework/backend avec un support Better Fullstack plus complet.

## Quand le choisir

Choisissez Qwik pour les applications frontend où la possibilité de reprise et une charge utile initiale minimale comptent plus qu'un backend généré dès le premier jour.

## Ajouter un backend plus tard

Lorsque l'application a besoin de données, décidez si Qwik doit rester en tant qu'interface pendant qu'un autre service possède le API, ou si un autre préréglage Better Fullstack correspondrait mieux aux exigences de la pile complète. Pour de nombreuses équipes, ajouter un Hono API en tant que service distinct est plus propre que de forcer les problèmes de back-end dans un échafaudage front-end d'abord.

Les futurs ajouts utiles incluent :

- Un Hono API pour la logique du serveur privé.
- PostgreSQL plus Drizzle ou Prisma pour les données relationnelles.
- Better Auth lorsque la gestion des comptes devient partie intégrante du produit.
- oRPC ou une autre couche de contrat typée lorsque le frontend et API ont besoin de types partagés.

## Notes de déploiement

Une application Qwik minimale est simple à déployer en tant qu'application Web ou interface statique, en fonction de l'adaptateur généré et de la cible d'hébergement. Étant donné que cette commande n'ajoute pas de base de données ni d'authentification, la configuration de la production se concentre principalement sur la sortie de build, les routes et les variables d'environnement publiques.

## Dépannage

- Si vous attendiez des fichiers de base de données, vérifiez la commande : ce guide transmet intentionnellement `--database none` et `--orm none`.
- Si les routes d'authentification sont manquantes, la pile utilise `--auth none`.
- Si les classes Tailwind ne s'appliquent pas, inspectez les chemins de contenu Tailwind générés et l'intégration du framework.
- Si vous ajoutez ultérieurement un API, séparez les variables client publiques des secrets du serveur privé.

## Notes de comparaison

Choisissez cette option plutôt que Astro lorsque la possibilité de reprise et le modèle de composant de Qwik sont les principales raisons du projet. Choisissez [Astro avec React et Hono](/guides/typescript/astro-react-hono/) lorsque les pages de contenu nécessitent un backend et une base de données Hono générés dès le début.
