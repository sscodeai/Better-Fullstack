Utilisez cette pile lorsque vous souhaitez un serveur TypeScript API rapide et une interface React distincte avec des appels client-serveur typés.

```npm
npm create better-fullstack@latest my-hono-app -- \
  --ecosystem typescript \
  --frontend tanstack-router \
  --backend hono \
  --runtime bun \
  --database postgres \
  --orm drizzle \
  --auth better-auth \
  --api trpc \
  --css-framework tailwind \
  --ui-library shadcn-ui \
```

## Ce que cela crée

- Une interface TanStack Router.
- Un backend Hono exécuté sur Bun.
- PostgreSQL et Drizzle.
- Better Auth.
- tRPC entre l'application Web et API.
- Une structure monorepo pour les packages frontend et serveur.

## Forme générée

Cette commande crée une forme d'application divisée : le frontend et le backend sont des packages distincts, mais ils sont générés ensemble et partagent la même configuration de pile. Cela rend la limite API explicite sans abandonner l'inférence TypeScript.

Forme représentative :

```txt
my-hono-app/
  bts.jsonc
  apps/
    web/
    server/
  packages/
    shared/
```

Les noms exacts des espaces de travail peuvent différer, mais le point architectural est le même : TanStack Router possède l'application de navigateur, Hono possède HTTP, Drizzle possède l'accès à la base de données et tRPC est le contrat tapé entre eux.

## Exemple d'itinéraire Hono

Hono est utile lorsque la gestion des requêtes doit rester explicite et légère :

```ts
import { Hono } from "hono";

const app = new Hono();

app.get("/health", (c) => {
  return c.json({ ok: true });
});

export default app;
```

Conservez le middleware au niveau du framework, le contexte d’authentification et la journalisation des demandes dans le package du serveur. Évitez de divulguer des objets de requête Hono dans une logique métier partagée.

## Exemple de forme de procédure tRPC

Les procédures tRPC doivent être de petits points d'entrée validés autour des opérations du serveur :

```ts
import { z } from "zod";

const createTaskInput = z.object({
  title: z.string().min(1),
});

export const taskProcedures = {
  create: async (rawInput: unknown, userId: string) => {
    const input = createTaskInput.parse(rawInput);

    return {
      id: crypto.randomUUID(),
      title: input.title,
      ownerId: userId,
    };
  },
};
```

Pour les applications plus volumineuses, placez les schémas de validation dans un module partagé uniquement lorsque le client et le serveur en ont besoin. Les écritures dans la base de données doivent rester uniquement sur le serveur.

## Quand le choisir

Choisissez Hono lorsque le backend doit rester petit, rapide et explicite. Cette configuration fonctionne bien pour les produits, tableaux de bord, outils internes et services API-first qui peuvent ultérieurement diviser le déploiement front-end et back-end.

## Remarques sur la compatibilité

- `--frontend tanstack-router` plus `--backend hono` donnent une application Web React distincte et un serveur API.
- `--runtime bun` sélectionne Bun pour le service Hono dans ce guide.
- `--api trpc` est approprié ici car le frontend est basé sur React. Pour les guides SvelteKit, Nuxt ou SolidStart, Better Fullstack utilise oRPC à la place.
- PostgreSQL et Drizzle forment une paire orientée production pour cette forme de service divisé.

## Notes de déploiement

Prévoyez deux cibles déployables si votre hébergeur traite le frontend et le serveur Hono séparément. Configurez délibérément CORS, les domaines de cookies d'authentification et les URL publiques API ; les services divisés rendent ces détails visibles.

Le serveur API a besoin de la chaîne de connexion PostgreSQL et des secrets d'authentification. L'application Web a besoin de l'origine API et de toutes les valeurs d'URL d'application publique. Gardez les secrets hors des variables publiques au moment de la construction du frontend.

## Dépannage

- Si les requêtes du navigateur échouent avant d’atteindre Hono, inspectez CORS et l’URL de base API configurée.
- Si les cookies d'authentification sont manquants dans tous les domaines, vérifiez les paramètres du même site, sécurisez les cookies et si les domaines web/API correspondent à votre déploiement prévu.
- Si tRPC appelle la compilation mais échoue au moment de l'exécution, vérifiez que l'adaptateur Hono tRPC est monté sur le même chemin utilisé par le client.
- Si les applications générées ne peuvent pas se connecter à PostgreSQL localement, confirmez que la base de données est en cours d'exécution et que la chaîne de connexion correspond à la base de données sélectionnée.

## Compromis

Si vous souhaitez qu'un seul framework possède l'intégralité de l'application, utilisez [TanStack Start](/guides/typescript/create-tanstack-start-project/) ou [Next.js](/guides/typescript/nextjs-drizzle-better-auth/). Si vous souhaitez une limite API distincte, Hono est la valeur par défaut la plus propre.

Par rapport à un framework fullstack auto-backend, cette pile comporte plus de pièces mobiles. Le résultat est une limite de service plus propre, des tests plus faciles axés sur API et un backend qui peut ensuite servir plusieurs clients.
