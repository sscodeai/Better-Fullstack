当你希望用 Astro 承载内容密集页面和 islands，同时保留一个 TypeScript API server 时，使用这个 stack。

```npm
npm create better-fullstack@latest my-astro-app -- \
  --ecosystem typescript \
  --frontend astro \
  --astro-integration react \
  --backend hono \
  --runtime bun \
  --database sqlite \
  --orm drizzle \
  --auth better-auth \
  --api trpc \
  --css-framework tailwind \
  --ui-library daisyui \
```

## 创建内容

- 带 React integration 的 Astro frontend。
- 运行在 Bun 上的 Hono backend。
- SQLite 和 Drizzle。
- Better Auth。
- tRPC，因为这个 Astro 配置使用 React。
- Tailwind CSS 和 DaisyUI。

## 生成结构

这个 stack 将 Astro 的内容优先 frontend 与独立的 Hono API server 组合起来。`--astro-integration react` 会启用 React islands，因此本指南可以使用 tRPC。

代表性结构：

```txt
my-astro-app/
  bts.jsonc
  apps/
    web/
    server/
  packages/
    shared/
```

Astro pages 应该专注于 routing、content 和 island composition。Hono 应该负责 API routes、需要 auth 的 server behavior，以及 database access。

## 使用模式

用 Astro 承载页面外壳，只在需要交互的位置使用 React。这样静态或近似静态页面仍然很轻，同时交互式组件也能调用 typed API。

简单 endpoint 和 health check 放在 Hono 中。涉及数据库的 endpoint 应把 Drizzle queries 放在 server-only helpers 后面，再通过 tRPC procedures 或 Hono routes 暴露。

## 兼容性

Astro 的 API 兼容性取决于所选 UI integration。本页使用 `--astro-integration react`，所以 tRPC 可用。对于非 React 的 Astro integration，优先使用 oRPC。

这个命令选择 DaisyUI 作为 UI library。如果之后切换到 shadcn/ui，请同时检查 Astro React integration 和 Tailwind 设置是否仍然匹配所选组件库。

## 何时选择

当内容、docs、marketing pages 或 partial hydration 是应用核心，同时你仍然需要生成好的 backend 和 database setup 时，选择 Astro。

## 部署

为 Astro frontend 和 Hono API 明确配置 origins。frontend 需要公开的 API URL；Hono service 需要 database 和 auth secrets。SQLite 适合本地开发，但如果生产环境继续使用 SQLite，hosting 必须提供持久化存储。

如果 frontend 和 API 分开部署，请按最终域名配置 CORS 和 auth cookies，而不是依赖 localhost 行为。

## 故障排查

- 如果 React islands 没有 hydrate，检查 Astro client directive，以及组件是否安全地导入了 browser-only code。
- 如果切换 integration 后 tRPC 不可用，确认 Astro integration 仍然是 React。
- 如果部署后 API calls 失败，验证公开 API origin 和 CORS 设置。
- 如果部署后 SQLite 数据消失，确认 hosting platform 支持持久化可写存储。
