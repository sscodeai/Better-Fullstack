Utilisez cette pile lorsque vous voulez SvelteKit sur le frontend et un petit serveur TypeScript API à côté.

```npm
npm create better-fullstack@latest my-sveltekit-app -- \
  --ecosystem typescript \
  --frontend svelte \
  --backend hono \
  --runtime bun \
  --database postgres \
  --orm drizzle \
  --auth better-auth \
  --api orpc \
  --css-framework tailwind \
```

## Ce que cela crée

- Une interface SvelteKit.
- Un backend Hono sur Bun.
- oRPC pour les contrats API tapés sur les choix d'interface non-React.
- PostgreSQL et Drizzle.
- Better Auth où la base de données sélectionnée et l'ORM le prennent en charge.

## Forme générée

Il s'agit d'une pile TypeScript divisée : SvelteKit gère l'expérience frontale, Hono gère le serveur API et oRPC est la limite typée entre eux.

Forme représentative :

```txt
my-sveltekit-app/
  bts.jsonc
  apps/
    web/
    server/
  packages/
    shared/
```

Les noms spécifiques des espaces de travail peuvent évoluer, mais gardez la propriété claire. Les composants Svelte ne doivent pas importer directement les modules de base de données ; ils doivent appeler des opérations de serveur typées.

## Exemple de modèle d'appel SvelteKit

Une fonction de page ou de chargement doit communiquer avec le client API généré au lieu de coder en dur les charges utiles des requêtes dans l'ensemble de l'application :

```ts
export async function load() {
  return {
    title: "Projects",
  };
}
```

Pour les mutations, conservez la validation et l'autorisation du côté Hono même si l'interface utilisateur SvelteKit valide déjà les entrées du formulaire.

## Exemple de modèle Drizzle

PostgreSQL plus Drizzle vous donne des définitions de table explicites :

```ts
import { pgTable, text, timestamp } from "drizzle-orm/pg-core";

export const project = pgTable("project", {
  id: text("id").primaryKey(),
  name: text("name").notNull(),
  ownerId: text("owner_id").notNull(),
  createdAt: timestamp("created_at").defaultNow().notNull(),
});
```

Utilisez de petites fonctions de serveur autour des requêtes afin que les gestionnaires Hono et les procédures oRPC ne deviennent pas un dépotoir pour les règles métier.

## Quand le choisir

Choisissez cette option lorsque votre équipe préfère SvelteKit mais souhaite toujours un serveur API explicite et des contrats tapés. Il est utile pour les applications qui peuvent se développer en déploiements front-end et back-end distincts.

## Remarques sur la compatibilité

- Ce guide utilise `--api orpc` car SvelteKit n'est pas une interface React.
- `--backend hono` et `--runtime bun` créent un service API distinct au lieu de s'appuyer uniquement sur les routes du serveur SvelteKit.
- Better Auth nécessite que la base de données sélectionnée et le chemin ORM restent compatibles. Synchronisez les modifications du schéma d'authentification et les migrations Drizzle.
- Si vous supprimez ultérieurement le backend séparé, revisitez ensemble la configuration du client API, CORS et la gestion des cookies d'authentification.

## Notes de déploiement

Attendez-vous à des variables d'environnement Web et API distinctes. L'application SvelteKit a besoin de l'origine publique API. Le serveur Hono a besoin de PostgreSQL et de secrets d'authentification. S'ils se déploient sur des domaines différents, configurez intentionnellement CORS et les cookies.

Exécutez les migrations de bases de données avant de déployer le code API qui attend de nouvelles tables ou colonnes.

## Dépannage

- Si les requêtes du navigateur échouent mais fonctionnent avec curl, vérifiez les informations d'identification CORS et des cookies.
- Si SvelteKit s'affiche sans données, vérifiez l'origine API utilisée par le code côté serveur et côté navigateur.
- Si l'authentification est incohérente entre les domaines, vérifiez le domaine des cookies, le même site et les paramètres sécurisés.
- Si les modifications du schéma Drizzle n’apparaissent pas, confirmez que les migrations ont été générées et appliquées à la base de données cible PostgreSQL.

## Notes de comparaison

Par rapport à une pile auto-backend Nuxt ou SolidStart, ce guide vous donne une limite API plus explicite. Comparé à [Hono avec tRPC et Drizzle](/guides/typescript/hono-trpc-drizzle/), il utilise oRPC car l'interface est SvelteKit plutôt que React.
