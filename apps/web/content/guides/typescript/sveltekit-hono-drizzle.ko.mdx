프런트엔드에 SvelteKit을, 그 옆에 작은 TypeScript API 서버를 원할 때 이 스택을 사용하세요.

```npm
npm create better-fullstack@latest my-sveltekit-app -- \
  --ecosystem typescript \
  --frontend svelte \
  --backend hono \
  --runtime bun \
  --database postgres \
  --orm drizzle \
  --auth better-auth \
  --api orpc \
  --css-framework tailwind \
```

## 이것이 만드는 것

- SvelteKit 프런트엔드.
- Bun의 Hono 백엔드.
- React가 아닌 프런트엔드 선택 사항에 걸쳐 입력된 API 계약의 경우 oRPC.
- PostgreSQL 및 Drizzle.
- Better Auth 선택한 데이터베이스와 ORM이 이를 지원합니다.

## 생성된 모양

이는 분할된 TypeScript 스택입니다. SvelteKit은 프런트엔드 경험을 처리하고, Hono는 API 서버를 처리하며, oRPC은 둘 사이의 유형화된 경계입니다.

대표적인 형태:

```txt
my-sveltekit-app/
  bts.jsonc
  apps/
    web/
    server/
  packages/
    shared/
```

특정 작업공간 이름은 바뀔 수 있지만 소유권을 명확하게 유지하세요. Svelte 구성 요소는 데이터베이스 모듈을 직접 가져와서는 안 됩니다. 형식화된 서버 작업을 호출해야 합니다.

## 예 SvelteKit 호출 패턴

페이지 또는 로드 함수는 앱 전체에서 요청 페이로드를 하드코딩하는 대신 생성된 API 클라이언트와 통신해야 합니다.

```ts
export async function load() {
  return {
    title: "Projects",
  };
}
```

변형의 경우 SvelteKit UI가 이미 양식 입력의 유효성을 검사하더라도 Hono 측에서 유효성 검사 및 승인을 유지하세요.

## 예 Drizzle 모델

PostgreSQL과 Drizzle은 명시적인 테이블 정의를 제공합니다.

```ts
import { pgTable, text, timestamp } from "drizzle-orm/pg-core";

export const project = pgTable("project", {
  id: text("id").primaryKey(),
  name: text("name").notNull(),
  ownerId: text("owner_id").notNull(),
  createdAt: timestamp("created_at").defaultNow().notNull(),
});
```

Hono 핸들러와 oRPC 프로시저가 비즈니스 규칙의 쓰레기장이 되지 않도록 쿼리 주변에 작은 서버 기능을 사용하십시오.

## 그것을 선택하는 경우

팀에서 SvelteKit을 선호하지만 여전히 명시적인 API 서버와 입력된 계약을 원할 경우 이 옵션을 선택하세요. 별도의 프런트엔드 및 백엔드 배포로 성장할 수 있는 앱에 유용합니다.

## 호환성 참고사항

- SvelteKit은 React 프런트엔드가 아니기 때문에 이 가이드에서는 `--api orpc`을 사용합니다.
- `--backend hono` 및 `--runtime bun`은 SvelteKit 서버 경로에만 의존하는 대신 별도의 API 서비스를 만듭니다.
- Better Auth의 호환성을 유지하려면 선택한 데이터베이스와 ORM 경로가 필요합니다. 인증 스키마 변경 사항과 Drizzle 마이그레이션을 동기화 상태로 유지하세요.
- 나중에 별도의 백엔드를 제거하는 경우 API 클라이언트 설정, CORS 및 인증 쿠키 처리를 함께 다시 방문하세요.

## 배포 참고사항

별도의 웹 및 API 환경 변수가 필요합니다. SvelteKit 앱에는 공개 API 원본이 필요합니다. Hono 서버에는 PostgreSQL 및 인증 비밀이 필요합니다. 다른 도메인에 배포하는 경우 CORS와 쿠키를 의도적으로 구성하세요.

새 테이블이나 열이 필요한 API 코드를 배포하기 전에 데이터베이스 마이그레이션을 실행하세요.

## 문제 해결

- 브라우저에서는 요청이 실패했지만 컬에서는 작동하는 경우 CORS 및 쿠키 자격 증명을 확인하세요.
- SvelteKit이 데이터 없이 렌더링되는 경우 서버 측 및 브라우저 측 코드에서 사용되는 API 원본을 확인하세요.
- 도메인 전체에서 인증이 일관되지 않는 경우 쿠키 도메인, 동일 사이트 및 보안 설정을 검토하세요.
- Drizzle 스키마 변경 사항이 표시되지 않으면 마이그레이션이 생성되어 대상 PostgreSQL 데이터베이스에 적용되었는지 확인하세요.
