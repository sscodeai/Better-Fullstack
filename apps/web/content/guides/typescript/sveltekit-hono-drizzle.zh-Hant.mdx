當你想要在 frontend 使用 SvelteKit，並在旁邊放一個小型 TypeScript API server 時，使用這個 stack。

```npm
npm create better-fullstack@latest my-sveltekit-app -- \
  --ecosystem typescript \
  --frontend svelte \
  --backend hono \
  --runtime bun \
  --database postgres \
  --orm drizzle \
  --auth better-auth \
  --api orpc \
  --css-framework tailwind \
```

## 創建內容

- 一 SvelteKit frontend。
- Bun 上的 Hono backend。
- oRPC，用於非 React frontend choices 之間的 typed API contracts。
- PostgreSQL 和 Drizzle。
- 在所選 database 和 ORM 支援時啟用 Better Auth。

## 生成結構

這是一個 split TypeScript stack：SvelteKit 處理 frontend experience，Hono 處理 API server，oRPC 是它們之間的 typed boundary。

代表性結構：

```txt
my-sveltekit-app/
  bts.jsonc
  apps/
    web/
    server/
  packages/
    shared/
```

具體 workspace 名稱可能演進，但職責要清楚。 Svelte components 不應直接匯入 database modules；它們應呼叫 typed server operations。

## 範例 SvelteKit call pattern

頁面或 load function 應該與產生的 API client 通信，而不是在整個 app 中硬編碼 request payloads：

```ts
export async function load() {
  return {
    title: "Projects",
  };
}
```

對於 mutations，即使 SvelteKit UI 已經驗證 form inputs，也要把 validation 和 authorization 保持在 Hono 側。

## 範例 Drizzle model

PostgreSQL 加 Drizzle 給你明確的 table definitions：

```ts
import { pgTable, text, timestamp } from "drizzle-orm/pg-core";

export const project = pgTable("project", {
  id: text("id").primaryKey(),
  name: text("name").notNull(),
  ownerId: text("owner_id").notNull(),
  createdAt: timestamp("created_at").defaultNow().notNull(),
});
```

圍繞 queries 使用小型 server functions，避免 Hono handlers 和 oRPC procedures 變成 business rules 的堆疊處。

## 何時選擇

當團隊偏好 SvelteKit，但仍想要明確 API server 和 typed contracts 時，選擇這個 stack。它適合未來可能拆成獨立 frontend/backend deployments 的 apps。

## 相容性

- 本指南使用 `--api orpc`，因為 SvelteKit 不是 React frontend。
- `--backend hono` 和 `--runtime bun` 會建立獨立 API service，而不是只依賴 SvelteKit server routes。
- Better Auth 要求所選 database 和 ORM 路徑保持相容。保持 auth schema changes 和 Drizzle migrations 同步。
- 如果之後移除獨立 backend，則需要一起重新檢查 API client setup、CORS 和 auth cookie handling。

## 部署

預期 web 和 API 會有不同 environment variables。 SvelteKit app 需要 public API origin;Hono server 需要 PostgreSQL 和 auth secrets。如果它們部署在不同 domains，請有意配置 CORS 和 cookies。

在部署依賴新 tables 或 columns 的 API code 前，先執行 database migrations。

## 故障排查

- 如果 requests 從 browser 失敗但 curl 可用，檢查 CORS 和 cookie credentials。
- 如果 SvelteKit render 沒有數據，確認 server-side 和 browser-side code 使用的 API origin。
- 如果 auth 在跨網域時不一致，請檢查 cookie domain、same-site 和 secure settings。
- 如果 Drizzle schema changes 沒出現，確認 migrations 已產生並套用到目標 PostgreSQL database。

## 對比說明

相較於 Nuxt 或 SolidStart self-backend stack，本指南提供更明確的 API boundary。相較於 [Hono with tRPC and Drizzle](/guides/typescript/hono-trpc-drizzle/)，它使用 oRPC，因為 frontend 是 SvelteKit 而不是 React。
