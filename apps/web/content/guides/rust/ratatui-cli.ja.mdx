Web サービスではなくターミナル UI を構築する場合は、このスタックを使用します。

```npm
npm create better-fullstack@latest my-rust-cli -- \
  --ecosystem rust \
  --rust-web-framework none \
  --rust-frontend none \
  --rust-orm none \
  --rust-api none \
  --rust-cli ratatui \
  --rust-logging none
```

## これが生み出すもの

- Rust CLI プロジェクト。
- Ratatui をターミナル UI オプションとして使用します。
- Web フレームワーク、フロントエンド、ORM、または API レイヤーはありません。

## 生成された形状

このスタックは、サーバーではなくターミナル アプリケーションを作成します。 Ratatui はターミナルへのウィジェットのレンダリングを処理しますが、アプリ コードは状態、イベント、コマンドの動作を所有します。

有用なメンタル モデルはループです。

- キーボードまたは端末のイベントを読み取ります。
- アプリケーションの状態を更新します。
- 現在の状態を Ratatui ウィジェットにレンダリングします。
- ユーザーが終了を要求した場合は、正常に終了します。

## 代表的なファイルツリー

```txt
my-rust-cli/
  Cargo.toml
  src/
    main.rs
    app.rs
    event.rs
    ui.rs
```

`app.rs` は状態遷移に重点を置き、`event.rs` は入力処理に重点を置き、`ui.rs` は描画に重点を置きます。この分割により、端末の動作のテストが容易になります。

## CLI と UI の例

アプリの状態タイプが小さいと、UI レンダリングが決定的に保たれます。

```rust
#[derive(Debug, Default)]
pub struct App {
    pub should_quit: bool,
    pub selected_index: usize,
    pub items: Vec<String>,
}

impl App {
    pub fn quit(&mut self) {
        self.should_quit = true;
    }

    pub fn next(&mut self) {
        if !self.items.is_empty() {
            self.selected_index = (self.selected_index + 1) % self.items.len();
        }
    }
}
```

描画関数は、アプリケーションの動作を変更することなく、状態を受け取り、それをレンダリングする必要があります。

```rust
use ratatui::{
    layout::{Constraint, Direction, Layout},
    widgets::{Block, Borders, List, ListItem, Paragraph},
    Frame,
};

use crate::app::App;

pub fn draw(frame: &mut Frame, app: &App) {
    let chunks = Layout::default()
        .direction(Direction::Vertical)
        .constraints([Constraint::Min(1), Constraint::Length(1)])
        .split(frame.area());

    let items = app.items.iter().map(|item| ListItem::new(item.as_str()));
    let list = List::new(items).block(Block::default().title("Tasks").borders(Borders::ALL));
    let help = Paragraph::new("q: quit  j/k: move");

    frame.render_widget(list, chunks[0]);
    frame.render_widget(help, chunks[1]);
}
```

イベント処理は小規模かつ明示的に行うことができます。

```rust
use crossterm::event::{KeyCode, KeyEvent};

use crate::app::App;

pub fn handle_key(app: &mut App, key: KeyEvent) {
    match key.code {
        KeyCode::Char('q') => app.quit(),
        KeyCode::Char('j') | KeyCode::Down => app.next(),
        _ => {}
    }
}
```

## 互換性に関する注意事項

CLI プロジェクトでは、明示的に必要でない限り、無関係な Web フレームワークとデータベースの選択を避ける必要があります。このガイドでは、これらのフラグを `none` に設定したままにします。

ターミナルファーストプロジェクトの場合は、`--rust-cli ratatui` と web/API/database フラグを `none` に設定したままにします。後でバックエンドまたはデータベースを追加する場合は、それをハイブリッド アプリとして扱い、レンダリング コードにネットワーク呼び出しを混在させるのではなく、明確なモジュール境界を導入します。

## いつ選択するか

ダッシュボード、開発者ツール、インフラストラクチャ ツール、ターミナルファーストのワークフローには Ratatui を選択してください。

これは、ユーザーが一日のほとんどをシェルで過ごし、繰り返しの操作に集中したインターフェースが必要な場合に特に便利です。

## テストと導入に関するメモ

ほとんどの Ratatui ロジックは、端末層の下でテストするのが最も簡単です。単体テストの状態遷移とコマンドの直接解析:

```bash
cargo test
```

リリース ビルドの場合は、Cargo の通常の最適化されたビルドを使用します。

```bash
cargo build --release
```

バイナリを配布するときは、カラー サポート、キーボード ショートカット、構成ファイルの場所、ツールが読み取る環境変数など、端末の期待事項を文書化します。

## トラブルシューティング

- クラッシュ後に端末が通常とは異なるモードのままの場合は、アプリが raw モードに復元し、シャットダウン中に代替画面が表示されることを確認してください。
- レンダリングがちらつく場合は、イベント ループが入力やティックなしで積極的に再描画されていないことを確認してください。
- テキストが予期せずクリップされる場合は、狭い端末幅でテストし、ハードコーディングされた座標の代わりに Ratatui レイアウト制約を使用します。
- キーボード ショートカットがリモート シェルで機能しない場合は、アプリのロジックを変更する前に、ターミナル エミュレーターの動作を比較してください。
