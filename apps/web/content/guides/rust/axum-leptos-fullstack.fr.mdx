Utilisez cette pile lorsque vous souhaitez que Rust soit à la fois côté serveur et côté frontend de l'application générée.

```npm
npm create better-fullstack@latest my-rust-fullstack -- \
  --ecosystem rust \
  --rust-web-framework axum \
  --rust-frontend leptos \
  --rust-orm sea-orm \
  --rust-api none \
  --rust-cli none \
  --rust-logging tracing \
  --rust-error-handling anyhow-thiserror
```

## Ce que cela crée

- Un serveur Rust basé sur Axum.
- Leptos comme option d'interface de Rust.
- SeaORM pour l'accès à la base de données.
- Paramètres par défaut de traçage et de gestion structurée des erreurs.

## Forme générée

Cette pile est une première application fullstack Rust. Axum possède le runtime du serveur, Leptos possède la surface du frontend et SeaORM donne au backend une couche de persistance typée.

L'application générée est utile lorsque vous souhaitez qu'un espace de travail Rust contienne :

- Démarrage du serveur, routage et état partagé.
- Leptos vues et code d'interface utilisateur interactif.
- Entités de base de données et accès aux données côté serveur.
- Journalisation et gestion des erreurs configurées depuis le début.

## Arborescence de fichiers représentative

Attendez-vous à une présentation qui sépare les problèmes de serveur des problèmes d'interface utilisateur tout en les conservant dans un seul projet :

```txt
my-rust-fullstack/
  Cargo.toml
  .env.example
  src/
    main.rs
    app.rs
    error.rs
    routes/
      mod.rs
      api.rs
    entities/
      mod.rs
      user.rs
    pages/
      mod.rs
      home.rs
      users.rs
    components/
      mod.rs
```

Utilisez l'arborescence générée comme guide de délimitation : les composants Leptos doivent rester concentrés sur le rendu et l'interaction, tandis que le travail de base de données doit rester côté serveur.

## Exemples de serveur et d'interface utilisateur

Une route Axum peut exposer les points de terminaison JSON pour les actions côté serveur ou les clients d'intégration :

```rust
use axum::{extract::State, Json};
use serde::Serialize;

use crate::{error::AppError, AppState};

#[derive(Debug, Serialize)]
pub struct HealthResponse {
    pub status: &'static str,
}

pub async fn health(State(_state): State<AppState>) -> Result<Json<HealthResponse>, AppError> {
    Ok(Json(HealthResponse { status: "ok" }))
}
```

Une page Leptos doit appeler des fonctions de serveur ou des assistants API plutôt que d'ouvrir des connexions à la base de données à partir du code de l'interface utilisateur :

```rust
use leptos::prelude::*;

#[component]
pub fn HomePage() -> impl IntoView {
    view! {
        <main>
            <h1>"Dashboard"</h1>
            <p>"Rust server, Rust UI, and one generated project shape."</p>
        </main>
    }
}
```

Pour l'interface utilisateur basée sur une base de données, conservez l'opération de la base de données derrière les limites du serveur :

```rust
use sea_orm::{DatabaseConnection, EntityTrait};

use crate::entities::user;

pub async fn list_users(db: &DatabaseConnection) -> Result<Vec<user::Model>, sea_orm::DbErr> {
    user::Entity::find().all(db).await
}
```

## Remarques sur la compatibilité

Ce guide utilise des indicateurs spécifiques à Rust au lieu des indicateurs frontend/backend TypeScript. Conservez les choix d’interface Rust sous `--rust-frontend`.

Les indicateurs `--rust-api none` et `--rust-cli none` maintiennent cet échafaudage concentré sur l'application Web. Ajoutez une option gRPC ou CLI uniquement lorsque le projet généré dessert intentionnellement plusieurs interfaces. Si vous souhaitez un backend uniquement API, utilisez `--rust-frontend none` et suivez plutôt le guide Axum API.

## Quand le choisir

Choisissez cette option lorsque le projet bénéficie d'une architecture Rust-first au-delà des limites de l'application et que votre équipe est à l'aise avec le modèle de compilation de Rust.

Il convient parfaitement aux outils internes, aux surfaces d'administration et aux domaines de produits où le partage des types Rust et la validation sur le serveur et l'interface utilisateur sont plus précieux que l'adoption d'un écosystème frontal JavaScript.

## Compromis

Fullstack Rust réduit le changement de langue, mais le flux de travail frontend est différent de React, Vue ou Svelte. Prévoyez les temps de compilation de Rust, les modèles de routage spécifiques à Leptos et une équipe qui est à l'aise pour déboguer le comportement du navigateur à partir du code d'interface utilisateur généré par Rust.

## Notes de tests et de déploiement

Utilisez les tests unitaires Rust pour les services et la logique du modèle, puis ajoutez des tests de route ou au niveau du navigateur autour des flux les plus importants.

```bash
cargo test
```

Pour le déploiement, vérifiez les deux moitiés de l'application : le serveur doit recevoir ses variables d'environnement et les ressources frontales doivent être créées à l'aide des scripts de projet générés. Conservez les migrations de bases de données dans le chemin de déploiement si l'application stocke les données utilisateur.

## Dépannage

- Si l'interface utilisateur est créée mais que les données du serveur sont manquantes, vérifiez que la route du serveur ou la fonction du serveur est enregistrée dans le routeur Axum.
- Si les types de bases de données dérivent, régénérez ou mettez à jour les entités SeaORM via le flux de travail des documents d'échafaudage.
- Si une route de navigateur fonctionne localement mais pas après le déploiement, vérifiez ensemble le routage du serveur généré et la configuration des actifs statiques.
- Si les journaux sont trop silencieux, définissez `RUST_LOG=info` avant de démarrer le serveur.
