최신 HTTP 프레임워크, 관계형 지속성, 로깅 및 명시적 오류 처리 기능을 갖춘 Rust 백엔드를 원할 때 이 스택을 사용하세요.

```npm
npm create better-fullstack@latest my-rust-api -- \
  --ecosystem rust \
  --rust-web-framework axum \
  --database postgres \
  --rust-orm sea-orm \
  --rust-logging tracing \
  --rust-error-handling anyhow-thiserror
```

## 이것이 만드는 것

- Axum을 사용하는 Rust API 프로젝트.
- PostgreSQL을 데이터베이스 옵션으로 사용합니다.
- 데이터베이스 액세스용 SeaORM.
- 구조화된 로깅을 추적합니다.
- 어쨌든 Thiserror 스타일 오류 처리.

## 생성된 모양

이 스택은 전체 스택 앱이 아닌 백엔드 서비스를 생성하기 위한 것입니다. 생성된 프로젝트는 Axum에 형식화된 HTTP 경계, SeaORM이 소유한 데이터베이스 계층, `tracing`을 통한 요청 범위 로깅을 제공합니다.

중요한 구현 경계는 다음과 같습니다.

- 경로는 얇아지고 HTTP 입력을 애플리케이션 호출로 변환합니다.
- SeaORM 모델과 엔터티는 데이터베이스 매핑을 소유합니다.
- 오류는 명시적으로 모델링된 다음 에지에서 HTTP 응답으로 변환됩니다.
- 구성은 시작 시 한 번 읽혀지므로 데이터베이스 풀, 라우터 및 로깅을 예측 가능하게 연결할 수 있습니다.

## 대표 파일 트리

정확한 파일 이름은 생성기가 발전함에 따라 변경될 수 있지만 이는 PostgreSQL 및 SeaORM이 있는 Axum API에서 기대할 수 있는 모양입니다.

```txt
my-rust-api/
  Cargo.toml
  .env.example
  src/
    main.rs
    config.rs
    error.rs
    routes/
      mod.rs
      health.rs
      users.rs
    entities/
      mod.rs
      prelude.rs
      user.rs
    services/
      mod.rs
      users.rs
```

HTTP 메커니즘에는 `routes/`을, 애플리케이션 동작에는 `services/`을, SeaORM의 생성 또는 유지 관리되는 데이터베이스 모델 유형에는 `entities/`를 사용하세요.

## 경로 및 모델 예시

Axum 핸들러는 일반적으로 추출된 상태 및 입력된 요청 데이터를 받아들인 다음 공유 오류 유형이 응답으로 변환할 수 있는 결과를 반환해야 합니다.

```rust
use axum::{extract::State, Json};
use serde::{Deserialize, Serialize};

use crate::{error::AppError, services::users::UserService, AppState};

#[derive(Debug, Deserialize)]
pub struct CreateUserRequest {
    pub email: String,
    pub name: String,
}

#[derive(Debug, Serialize)]
pub struct UserResponse {
    pub id: i32,
    pub email: String,
    pub name: String,
}

pub async fn create_user(
    State(state): State<AppState>,
    Json(input): Json<CreateUserRequest>,
) -> Result<Json<UserResponse>, AppError> {
    let user = UserService::new(state.db)
        .create(input.email, input.name)
        .await?;

    Ok(Json(UserResponse {
        id: user.id,
        email: user.email,
        name: user.name,
    }))
}
```

SeaORM은 활성 모델을 통해 삽입을 명시적으로 유지합니다.

```rust
use sea_orm::{ActiveModelTrait, ActiveValue::Set, DatabaseConnection};

use crate::entities::user;

pub async fn create_user(
    db: &DatabaseConnection,
    email: String,
    name: String,
) -> Result<user::Model, sea_orm::DbErr> {
    user::ActiveModel {
        email: Set(email),
        name: Set(name),
        ..Default::default()
    }
    .insert(db)
    .await
}
```

## 호환성 참고사항

<GuideCompatibilityNote kind="supported" title="Compatibility rule">
이 가이드의 모양을 원할 때 `--rust-web-framework axum`, `--database postgres` 및 `--rust-orm sea-orm`을 함께 유지하세요. ORM을 `sqlx`으로 전환하면 직접 작성한 SQL이 늘어나고 엔터티 스타일 모델 유형이 줄어듭니다. `--rust-web-framework none`을 설정하면 이 가이드의 경로 예시가 더 이상 적용되지 않습니다.
</GuideCompatibilityNote>

## 그것을 선택하는 경우

성능, 정확성 및 명시적인 서버 코드가 중요한 경우 Axum을 선택하세요. 이는 APIs, 백엔드 서비스 및 Rust의 유형 시스템이 제품 전략의 일부인 시스템에 적합합니다.

## 트레이드오프

Rust는 일반적으로 TypeScript, Python 또는 Go보다 더 적극적인 유형 모델링을 요구합니다. 그 대가로 강력한 컴파일 타임 검사와 예측 가능한 런타임 동작을 얻을 수 있습니다.

SeaORM은 팀에서 생성된 엔터티 유형과 구성 가능한 쿼리 빌더를 원할 때 생산적입니다. 직접 작성한 SQL이 디자인의 일부인 쿼리 중심 서비스의 경우 SQLx보다 무겁게 느껴질 수 있습니다.

## 테스트 및 배포 참고 사항

핸들러와 서비스를 별도로 테스트하는 것부터 시작하세요. 핸들러 테스트는 테스트 상태로 Axum 라우터를 구축할 수 있는 반면 서비스 테스트는 일회용 PostgreSQL 데이터베이스에 대해 실행할 수 있습니다.

```bash
cargo test
```

배포하기 전에 런타임 환경이 PostgreSQL 연결 문자열과 프로젝트에서 사용하는 마이그레이션 단계를 제공하는지 확인하세요. 프로덕션의 경우 `RUST_LOG`을 명시적으로 유지하여 너무 시끄럽지 않고 추적 출력이 유용하도록 합니다.

```bash
RUST_LOG=info cargo run
```

## 문제 해결

- 앱이 시작되지만 데이터베이스 호출이 실패하는 경우 생성된 `.env.example` 키가 셸 또는 배포 플랫폼에 설정된 변수와 일치하는지 확인하세요.
- SeaORM 유형이 데이터베이스와 일치하지 않으면 생성된 코드를 수동으로 편집하는 대신 프로젝트의 엔터티 또는 마이그레이션 워크플로를 다시 실행하세요.
- 오류가 일반 500 응답으로 반환되면 공유 오류 유형과 해당 `IntoResponse` 구현을 확인하세요.
- 로그에 요청 컨텍스트가 누락된 경우 라우터가 트래픽 제공을 시작하기 전에 추적 구독자가 초기화되었는지 확인하세요.
