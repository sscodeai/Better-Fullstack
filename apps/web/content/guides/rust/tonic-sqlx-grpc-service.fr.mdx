Utilisez cette pile lorsque vous avez besoin d'un backend Rust conçu pour les API de service à service et les contrats gRPC tapés.

```npm
npm create better-fullstack@latest my-rust-grpc -- \
  --ecosystem rust \
  --rust-web-framework axum \
  --rust-frontend none \
  --rust-orm sqlx \
  --rust-api tonic \
  --rust-cli none \
  --rust-logging tracing \
  --rust-error-handling anyhow-thiserror
```

## Ce que cela crée

- Un service Rust utilisant Axum.
- Tonic pour gRPC.
- SQLx pour l'accès à la base de données.
- Traçage et gestion structurée des erreurs.

## Forme générée

Cette pile est conçue pour les appels de service à service typés. Tonic possède le contrat gRPC et l'implémentation du serveur, SQLx possède l'accès à la base de données et Axum peut toujours fournir des routes HTTP opérationnelles telles que des contrôles d'état lorsque le projet généré les inclut.

Les limites principales sont :

- `proto/` définit le contrat de service.
- Les types Rust gRPC générés restent proches de l’implémentation du service Tonic.
- Les requêtes SQLx restent dans le référentiel ou les modules de service plutôt que dans les gestionnaires de transport.
- Les routes Axum, si elles sont utilisées, sont des points de terminaison opérationnels plutôt que le produit principal API.

```txt
my-rust-grpc/

```

Le fichier `build.rs` est généralement l'endroit où la génération de protobuf est câblée. Gardez le code généré hors de la logique métier éditée manuellement.

## Exemples de proto et de services

Un petit contrat protobuf donne aux clients et aux serveurs la même limite typée :

```proto
syntax = "proto3";

package users.v1;

service UserService {
  rpc GetUser(GetUserRequest) returns (User);
}

message GetUserRequest {
  int64 id = 1;
}

message User {
  int64 id = 1;
  string email = 2;
  string name = 3;
}
```

L'implémentation Tonic doit traduire les erreurs de transport en statuts gRPC et conserver l'accès à la base de données dans un module séparé :

```rust
use tonic::{Request, Response, Status};

use crate::grpc::users_v1::{
    user_service_server::UserService, GetUserRequest, User,
};
use crate::repositories::users::UserRepository;

pub struct UsersGrpc {
    repo: UserRepository,
}

#[tonic::async_trait]
impl UserService for UsersGrpc {
    async fn get_user(
        &self,
        request: Request<GetUserRequest>,
    ) -> Result<Response<User>, Status> {
        let id = request.into_inner().id;
        let user = self
            .repo
            .find_by_id(id)
            .await
            .map_err(|error| Status::internal(error.to_string()))?
            .ok_or_else(|| Status::not_found("user not found"))?;

        Ok(Response::new(User {
            id: user.id,
            email: user.email,
            name: user.name,
        }))
    }
}
```

SQLx maintient la forme de la requête explicite :

```rust
use sqlx::PgPool;

pub struct UserRow {
    pub id: i64,
    pub email: String,
    pub name: String,
}

pub async fn find_user(pool: &PgPool, id: i64) -> Result<Option<UserRow>, sqlx::Error> {
    sqlx::query_as!(
        UserRow,
        "select id, email, name from users where id = $1",
        id
    )
    .fetch_optional(pool)
    .await
}
```

## Remarques sur la compatibilité

Conservez `--rust-api tonic` lorsque vous souhaitez générer un câblage de service gRPC. Si vous modifiez `--rust-api none`, les exemples protobuf et Tonic de ce guide deviennent un ajout manuel. SQLx fonctionne bien pour les contrats de service car la sortie de la requête peut être mappée directement dans les types de réponse protobuf.

## Quand le choisir

Choisissez cette option pour les services internes, les microservices et les systèmes backend où les contrats gRPC conviennent mieux que les routes JSON.

## Compromis

gRPC est excellent pour les limites de service fortement typées, mais il ajoute des attentes en matière de protocole et d'outils. Pour les API HTTP destinés au navigateur, commencez par [Axum avec PostgreSQL et SeaORM](/guides/rust/axum-postgres-seaorm/).

SQLx maintient le comportement de la base de données transparent, mais vous devez gérer délibérément les fichiers SQL, les migrations et la compatibilité des requêtes.

## Notes de tests et de déploiement

Testez le comportement au niveau de Protobuf avec les clients générés ou les types de requêtes Tonic, puis testez les fonctions du référentiel par rapport à une véritable base de données PostgreSQL lorsque l'exactitude des requêtes est importante.

```bash
cargo test
```

Dans CI, assurez-vous que les outils protobuf sont disponibles si le projet génère des types Rust pendant la construction. Pour le déploiement, exposez le port gRPC attendu par les clients et conservez toute route d'intégrité HTTP séparée de l'adresse du service gRPC si votre plate-forme nécessite des sondes distinctes.

## Dépannage

- Si les types protobuf sont manquants, vérifiez `build.rs` et confirmez que le chemin `.proto` est inclus.
- Si les vérifications SQLx au moment de la compilation échouent, vérifiez l'URL de la base de données ou les métadonnées de la requête hors ligne attendues par le projet généré.
- Si les clients reçoivent `UNIMPLEMENTED`, confirmez que le serveur de service généré est ajouté au générateur de serveur Tonic.
- Si les vérifications de l'état réussissent mais que les appels gRPC échouent, vérifiez que le port gRPC est celui exposé par votre gestionnaire de processus ou votre conteneur.
