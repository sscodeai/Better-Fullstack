當你需要帶有 modern HTTP framework、relational persistence、logging 和 explicit error handling 的 Rust backend 時，使用這個 stack。

```npm
npm create better-fullstack@latest my-rust-api -- \
  --ecosystem rust \
  --rust-web-framework axum \
  --database postgres \
  --rust-orm sea-orm \
  --rust-logging tracing \
  --rust-error-handling anyhow-thiserror
```

## 創建內容

- 使用 Axum 的 Rust API project。
- PostgreSQL 作為 database option。
- SeaORM 用於 database access。
- tracing 用於 structured logging。
- Anyhow 和 Thiserror 風格的 error handling。

## 生成結構

這個 stack 產生 backend service，而不是 fullstack app。生成項目提供 Axum 中的 typed HTTP boundary、SeaORM 擁有的 database layer，以及透過 `tracing` 實現的 request-scoped logging。

重要邊界是：

- Routes 保持輕薄，將 HTTP input 轉換為 application calls。
- SeaORM models 和 entities 擁有 database mapping。
- Errors 被明確建模，並在 edge 轉換為 HTTP responses。
- Configuration 在 startup 時讀取一次，方便可預測地連接 database pools、routers 和 logging。

## 代表性文件樹

準確檔案名稱可能隨 generator 演進而變化，但這是 Axum API 搭配 PostgreSQL 和 SeaORM 時應預期的結構：

```txt
my-rust-api/
  Cargo.toml
  .env.example
  src/
    main.rs
    config.rs
    error.rs
    routes/
      mod.rs
      health.rs
      users.rs
    entities/
      mod.rs
      prelude.rs
      user.rs
    services/
      mod.rs
      users.rs
```

用 `routes/` 放 HTTP mechanics，`services/` 放 application behavior，`entities/` 放 SeaORM generated 或 maintained database model types。

## Route 和 model 範例

Axum handler 通常應接受提取的 state 和 typed request data，然後回傳一個 result，讓共用 error type 可以把它轉換成 response：

```rust
use axum::{extract::State, Json};
use serde::{Deserialize, Serialize};

use crate::{error::AppError, services::users::UserService, AppState};

#[derive(Debug, Deserialize)]
pub struct CreateUserRequest {
    pub email: String,
    pub name: String,
}

#[derive(Debug, Serialize)]
pub struct UserResponse {
    pub id: i32,
    pub email: String,
    pub name: String,
}

pub async fn create_user(
    State(state): State<AppState>,
    Json(input): Json<CreateUserRequest>,
) -> Result<Json<UserResponse>, AppError> {
    let user = UserService::new(state.db)
        .create(input.email, input.name)
        .await?;

    Ok(Json(UserResponse {
        id: user.id,
        email: user.email,
        name: user.name,
    }))
}
```

SeaORM 透過 active models 讓 inserts 保持明確：

```rust
use sea_orm::{ActiveModelTrait, ActiveValue::Set, DatabaseConnection};

use crate::entities::user;

pub async fn create_user(
    db: &DatabaseConnection,
    email: String,
    name: String,
) -> Result<user::Model, sea_orm::DbErr> {
    user::ActiveModel {
        email: Set(email),
        name: Set(name),
        ..Default::default()
    }
    .insert(db)
    .await
}
```

## 相容性

<GuideCompatibilityNote kind="supported" title="兼容性规则">
當你需要本指南的 shape 時，請保持 `--rust-web-framework axum`、`--database postgres` 和 `--rust-orm sea-orm` 組合。如果 ORM 切換到 `sqlx`，會有更多 handwritten SQL 和更少 entity-style model types。如果設定 `--rust-web-framework none`，本指南的 route examples 不再適用。
</GuideCompatibilityNote>

## 何時選擇

當效能、正確性和明確 server code 很重要時，選擇 Axum。它適合 APIs、backend services，以及 Rust type system 是產品策略一部分的系統。

## Tradeoffs

相較於 TypeScript、Python 或 Go，Rust 通常需要更多 upfront type modeling。作為回報，你會得到強編譯期檢查和可預測的 runtime behavior。

當團隊想要 generated entity types 和 composable query builders 時，SeaORM 很有效率。對於大量查詢、且 handwritten SQL 是設計一部分的服務，它可能會比 SQLx 顯得更重。

## 測試與部署

先分別測試 handlers 和 services。 Handler tests 可以用 test state 建構 Axum router；service tests 可以針對 disposable PostgreSQL database 運作。

```bash
cargo test
```

部署前請確保 runtime environment 提供 PostgreSQL connection string，以及專案使用的 migration step。生產環境中保持 `RUST_LOG` 明確設置，這樣 tracing output 有用但不會過於嘈雜：

```bash
RUST_LOG=info cargo run
```

## 故障排查

- 如果 app 能啟動但 database calls 失敗，請確認產生的 `.env.example` keys 與 shell 或 deployment platform 中設定的變數一致。
- 如果 SeaORM types 與 database 不匹配，請重新執行專案的 entity 或 migration workflow，而不是手動編輯 generated code。
- 如果 errors 回傳為 generic 500 responses，請檢查 shared error type 及其 `IntoResponse` implementation。
- 如果 logs 缺少 request context，請確認 tracing subscriber 在 router 開始服務流量前完成初始化。
