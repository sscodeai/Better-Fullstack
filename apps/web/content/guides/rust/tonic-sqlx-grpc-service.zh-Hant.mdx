當你需要面向 service-to-service API 和 typed gRPC contract 的 Rust 後端時，使用這個 stack。

```npm
npm create better-fullstack@latest my-rust-grpc -- \
  --ecosystem rust \
  --rust-web-framework axum \
  --rust-frontend none \
  --rust-orm sqlx \
  --rust-api tonic \
  --rust-cli none \
  --rust-logging tracing \
  --rust-error-handling anyhow-thiserror
```

## 創建內容

- 使用 Axum 的 Rust 服務。
- Tonic 用於 gRPC。
- SQLx 用於資料庫存取。
- tracing 和結構化錯誤處理。

## 生成結構

這個 stack 面向類型化的 service-to-service 呼叫。 Tonic 負責 gRPC 契約和 server implementation，SQLx 負責資料庫存取；當產生項目包含 Axum 時，它仍然可以提供 health check 等運維 HTTP route。

核心邊界是：

- `proto/` 定義服務契約。
- 產生的 Rust gRPC type 靠近 Tonic service implementation。
- SQLx query 放在 repository 或 service module 中，而不是 transport handler 裡。
- Axum route 如果存在，應作為運維 endpoint，而不是主要產品 API。

```txt
my-rust-grpc/

```

`build.rs` 通常是連接 protobuf generation 的地方。把生成程式碼和手寫業務邏輯分開。

## Proto 與服務範例

一個小的 protobuf 契約讓客戶端和服務端擁有同一條 typed boundary：

```proto
syntax = "proto3";

package users.v1;

service UserService {
  rpc GetUser(GetUserRequest) returns (User);
}

message GetUserRequest {
  int64 id = 1;
}

message User {
  int64 id = 1;
  string email = 2;
  string name = 3;
}
```

Tonic implementation 應該把 transport error 轉換為 gRPC status，並且把資料庫存取留在單獨模組中：

```rust
use tonic::{Request, Response, Status};

use crate::grpc::users_v1::{
    user_service_server::UserService, GetUserRequest, User,
};
use crate::repositories::users::UserRepository;

pub struct UsersGrpc {
    repo: UserRepository,
}

#[tonic::async_trait]
impl UserService for UsersGrpc {
    async fn get_user(
        &self,
        request: Request<GetUserRequest>,
    ) -> Result<Response<User>, Status> {
        let id = request.into_inner().id;
        let user = self
            .repo
            .find_by_id(id)
            .await
            .map_err(|error| Status::internal(error.to_string()))?
            .ok_or_else(|| Status::not_found("user not found"))?;

        Ok(Response::new(User {
            id: user.id,
            email: user.email,
            name: user.name,
        }))
    }
}
```

SQLx 讓 query shape 保持明確：

```rust
use sqlx::PgPool;

pub struct UserRow {
    pub id: i64,
    pub email: String,
    pub name: String,
}

pub async fn find_user(pool: &PgPool, id: i64) -> Result<Option<UserRow>, sqlx::Error> {
    sqlx::query_as!(
        UserRow,
        "select id, email, name from users where id = $1",
        id
    )
    .fetch_optional(pool)
    .await
}
```

## 相容性說明

當你需要產生好的 gRPC service wiring 時，保留 `--rust-api tonic`。如果改成 `--rust-api none`，本指南裡的 protobuf 和 Tonic 範例就變成手動新增項目。 SQLx 很適合服務契約，因為 query output 可以直接對應到 protobuf response type。

## 何時選擇

當內部服務、微服務或後端系統更適合 gRPC contract，而不是 JSON route 時，選擇這個 stack。

## 取捨

gRPC 非常適合強型別服務邊界，但會帶來協定和 tooling 預期。對於 browser-facing HTTP API，請從 [Axum with PostgreSQL and SeaORM](/guides/rust/axum-postgres-seaorm/) 開始。

SQLx 讓資料庫行為保持透明，但你需要有意管理 SQL file、migration 和 query compatibility。

## 測試與部署

用 generated client 或 Tonic request type 測試 protobuf-level behavior；當 query correctness 很重要時，用真實 PostgreSQL database 測試 repository function。

```bash
cargo test
```

在 CI 中，如果專案 build 時產生 Rust type，請確保 protobuf tooling 可用。部署時暴露客戶端期望的 gRPC port；如果平台需要獨立 probe，請將 HTTP health route 與 gRPC service address 分開。

## 故障排查

- 如果 protobuf type 缺失，檢查 `build.rs` 並確認 `.proto` path 已包含。
- 如果 SQLx compile-time check 失敗，驗證產生專案期望的 database URL 或 offline query metadata。
- 如果用戶端收到 `UNIMPLEMENTED`，確認 generated service server 已新增至 Tonic server builder。
- 如果 health check 通過但 gRPC call 失敗，檢查 process manager 或 container 暴露的是否為 gRPC port。
