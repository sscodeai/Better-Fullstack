サービス間 API および型付き gRPC コントラクト用に形成された Rust バックエンドが必要な場合は、このスタックを使用します。

```npm
npm create better-fullstack@latest my-rust-grpc -- \
  --ecosystem rust \
  --rust-web-framework axum \
  --rust-frontend none \
  --rust-orm sqlx \
  --rust-api tonic \
  --rust-cli none \
  --rust-logging tracing \
  --rust-error-handling anyhow-thiserror
```

## これが生み出すもの

- Axum を使用する Rust サービス。
- gRPC の場合は Tonic。
- SQLx データベースアクセス用。
- トレースと構造化されたエラー処理。

## 生成された形状

このスタックは、型指定されたサービス間呼び出し用に形成されています。 Tonic は gRPC コントラクトとサーバー実装を所有し、SQLx はデータベース アクセスを所有し、生成されたプロジェクトにヘルス チェックなどの運用可能な HTTP ルートが含まれている場合、Axum は引き続き提供できます。

核となる境界は次のとおりです。

- `proto/` はサービス コントラクトを定義します。
- 生成された Rust gRPC タイプは、Tonic サービス実装に近いままになります。
- SQLx クエリは、トランスポート ハンドラー内ではなく、リポジトリまたはサービス モジュールに残ります。
- Axum ルートを使用する場合、メイン製品 API ではなく、運用エンドポイントになります。

通常、`build.rs` ファイルは protobuf 生成が接続される場所です。生成されたコードを手動で編集したビジネス ロジックから遠ざけます。

## プロトとサービスの例

小さな protobuf コントラクトにより、クライアントとサーバーに同じ型付き境界が与えられます。

```proto
syntax = "proto3";

package users.v1;

service UserService {
  rpc GetUser(GetUserRequest) returns (User);
}

message GetUserRequest {
  int64 id = 1;
}

message User {
  int64 id = 1;
  string email = 2;
  string name = 3;
}
```

Tonic 実装では、トランスポート エラーを gRPC ステータスに変換し、データベース アクセスを別のモジュールに保持する必要があります。

```rust
use tonic::{Request, Response, Status};

use crate::grpc::users_v1::{
    user_service_server::UserService, GetUserRequest, User,
};
use crate::repositories::users::UserRepository;

pub struct UsersGrpc {
    repo: UserRepository,
}

#[tonic::async_trait]
impl UserService for UsersGrpc {
    async fn get_user(
        &self,
        request: Request<GetUserRequest>,
    ) -> Result<Response<User>, Status> {
        let id = request.into_inner().id;
        let user = self
            .repo
            .find_by_id(id)
            .await
            .map_err(|error| Status::internal(error.to_string()))?
            .ok_or_else(|| Status::not_found("user not found"))?;

        Ok(Response::new(User {
            id: user.id,
            email: user.email,
            name: user.name,
        }))
    }
}
```

SQLx はクエリの形状を明示的に保ちます。

```rust
use sqlx::PgPool;

pub struct UserRow {
    pub id: i64,
    pub email: String,
    pub name: String,
}

pub async fn find_user(pool: &PgPool, id: i64) -> Result<Option<UserRow>, sqlx::Error> {
    sqlx::query_as!(
        UserRow,
        "select id, email, name from users where id = $1",
        id
    )
    .fetch_optional(pool)
    .await
}
```

## 互換性に関する注意事項

生成された gRPC サービス配線が必要な場合は、`--rust-api tonic` を保持します。 `--rust-api none` を変更すると、このガイドの protobuf および Tonic の例は手動で追加されたものになります。 SQLx は、クエリ出力を protobuf 応答タイプに直接マッピングできるため、サービス コントラクトに適しています。

## いつ選択するか

JSON ルートよりも gRPC コントラクトが適している内部サービス、マイクロサービス、バックエンド システムにはこれを選択してください。

## トレードオフ

gRPC は、厳密に型指定されたサービス境界に優れていますが、プロトコルとツールの期待が追加されます。ブラウザー向けの HTTP API の場合は、[PostgreSQL および SeaORM を含む Axum](/guides/rust/axum-postgres-seaorm/) から始めます。

SQLx はデータベースの動作を透過的に保ちますが、SQL ファイル、移行、およびクエリの互換性を意図的に管理する必要があります。

## テストと導入に関するメモ

生成されたクライアントまたは Tonic リクエスト タイプを使用して protobuf レベルの動作をテストし、クエリの正確さが重要な場合は実際の PostgreSQL データベースに対してリポジトリ関数をテストします。

```bash
cargo test
```

CI では、プロジェクトがビルド中に Rust タイプを生成する場合、protobuf ツールが利用可能であることを確認してください。デプロイメントでは、クライアントが予期する gRPC ポートを公開し、プラットフォームで個別のプローブが必要な場合は、HTTP 正常性ルートを gRPC サービス アドレスから分離しておきます。

## トラブルシューティング

- protobuf タイプが欠落している場合は、`build.rs` をチェックし、`.proto` パスが含まれていることを確認してください。
- SQLx コンパイル時チェックが失敗した場合は、生成されたプロジェクトで予期されるデータベース URL またはオフライン クエリ メタデータを確認してください。
- クライアントが `UNIMPLEMENTED` を受信した場合は、生成されたサービス サーバーが Tonic サーバー ビルダーに追加されていることを確認します。
- ヘルスチェックには合格したが、gRPC 呼び出しが失敗した場合は、gRPC ポートがプロセス マネージャーまたはコンテナーによって公開されているポートであることを確認してください。
