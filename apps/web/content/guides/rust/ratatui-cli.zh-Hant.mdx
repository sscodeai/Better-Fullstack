當你想要建立終端 UI，而不是 web service 時，使用這個 stack。

```npm
npm create better-fullstack@latest my-rust-cli -- \
  --ecosystem rust \
  --rust-web-framework none \
  --rust-frontend none \
  --rust-orm none \
  --rust-api none \
  --rust-cli ratatui \
  --rust-logging none
```

## 創建內容

- Rust CLI 項目。
- Ratatui 作為終端 UI 選項。
- 不含 web framework、frontend、ORM 或 API layer。

## 生成結構

這個 stack 創建的是終端應用，而不是 server。 Ratatui 負責把 widget 渲染到 terminal 中，app code 擁有 state、event 和 command behavior。

一個有用的心智模式是這樣的循環：

- 讀取鍵盤或終端事件。
- 更新應用程式狀態。
- 將目前狀態渲染成 Ratatui widget。
- 當用戶要求退出時乾淨地關閉。

## 代表性文件樹

```txt
my-rust-cli/
  Cargo.toml
  src/
    main.rs
    app.rs
    event.rs
    ui.rs
```

讓 `app.rs` 專注狀態轉換，`event.rs` 專注輸入處理，`ui.rs` 專注繪製。這樣的拆分讓終端行為更容易測試。

## CLI 與 UI 範例

一個小的 app state 類型可以讓 UI 渲染保持確定性：

```rust
#[derive(Debug, Default)]
pub struct App {
    pub should_quit: bool,
    pub selected_index: usize,
    pub items: Vec<String>,
}

impl App {
    pub fn quit(&mut self) {
        self.should_quit = true;
    }

    pub fn next(&mut self) {
        if !self.items.is_empty() {
            self.selected_index = (self.selected_index + 1) % self.items.len();
        }
    }
}
```

draw 函數應該接收狀態並渲染它，不要改變應用行為：

```rust
use ratatui::{
    layout::{Constraint, Direction, Layout},
    widgets::{Block, Borders, List, ListItem, Paragraph},
    Frame,
};

use crate::app::App;

pub fn draw(frame: &mut Frame, app: &App) {
    let chunks = Layout::default()
        .direction(Direction::Vertical)
        .constraints([Constraint::Min(1), Constraint::Length(1)])
        .split(frame.area());

    let items = app.items.iter().map(|item| ListItem::new(item.as_str()));
    let list = List::new(items).block(Block::default().title("Tasks").borders(Borders::ALL));
    let help = Paragraph::new("q: quit  j/k: move");

    frame.render_widget(list, chunks[0]);
    frame.render_widget(help, chunks[1]);
}
```

事件處理可以保持小而明確：

```rust
use crossterm::event::{KeyCode, KeyEvent};

use crate::app::App;

pub fn handle_key(app: &mut App, key: KeyEvent) {
    match key.code {
        KeyCode::Char('q') => app.quit(),
        KeyCode::Char('j') | KeyCode::Down => app.next(),
        _ => {}
    }
}
```

## 相容性說明

CLI 項目應避免無關的 web framework 和資料庫選擇，除非你明確需要它們。本指南將這些 flag 保持為 `none`。

為 terminal-first 項目保留 `--rust-cli ratatui`，並讓 web/API/database flag 保持為 `none`。如果之後加入 backend 或 database，請將它當作 hybrid app，並引入清晰的模組邊界，不要把網路呼叫混進 rendering code。

## 何時選擇

Ratatui 適合 dashboard、開發者工具、基礎架構工具和 terminal-first workflow。

當使用者大部分時間都在 shell 裡，並且需要一個專注介面來重複執行操作時，它尤其有用。

## 測試與分發

大多數 Ratatui logic 最適合在 terminal layer 以下測試。直接測試 state transition 和 command parsing：

```bash
cargo test
```

release build 使用 Cargo 的常規最佳化建構：

```bash
cargo build --release
```

分發 binary 時，記錄 terminal expectation，例如 color support、keyboard shortcut、config file location，以及工具讀取的 environment variable。

## 故障排查

- 如果 crash 後 terminal 留在異常模式，請確保 app 在 shutdown 時恢復 raw mode 並退出 alternate screen。
- 如果渲染閃爍，請檢查 event loop 是否在沒有 input 或 tick 的情況下過度重繪。
- 如果文字意外裁切，用窄 terminal width 測試，並使用 Ratatui layout constraint，而不是 hard-coded 座標。
- 如果鍵盤快速鍵在 remote shell 中不起作用，先比較 terminal emulator 行為，再修改 app logic。
