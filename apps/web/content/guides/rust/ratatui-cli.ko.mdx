웹 서비스 대신 터미널 UI를 구축하려는 경우 이 스택을 사용하십시오.

```npm
npm create better-fullstack@latest my-rust-cli -- \
  --ecosystem rust \
  --rust-web-framework none \
  --rust-frontend none \
  --rust-orm none \
  --rust-api none \
  --rust-cli ratatui \
  --rust-logging none
```

## 이것이 만드는 것

- Rust CLI 프로젝트.
- Ratatui을 터미널 UI 옵션으로 사용합니다.
- 웹 프레임워크, 프런트엔드, ORM 또는 API 레이어가 없습니다.

## 생성된 모양

이 스택은 서버가 아닌 터미널 애플리케이션을 생성합니다. Ratatui은 위젯을 터미널에 렌더링하는 작업을 처리하고 앱 코드는 상태, 이벤트 및 명령 동작을 소유합니다.

유용한 정신 모델은 루프입니다.

- 키보드 또는 터미널 이벤트를 읽습니다.
- 애플리케이션 상태를 업데이트합니다.
- 현재 상태를 Ratatui 위젯으로 렌더링합니다.
- 사용자가 종료를 요청하면 깔끔하게 종료됩니다.

## 대표 파일 트리

```txt
my-rust-cli/
  Cargo.toml
  src/
    main.rs
    app.rs
    event.rs
    ui.rs
```

`app.rs`은 상태 전환에 집중하고, `event.rs`은 입력 처리에, `ui.rs`는 그리기에 집중하세요. 이렇게 분할하면 터미널 동작을 더 쉽게 테스트할 수 있습니다.

## CLI 및 UI 예시

작은 앱 상태 유형은 UI 렌더링을 결정적으로 유지합니다.

```rust
#[derive(Debug, Default)]
pub struct App {
    pub should_quit: bool,
    pub selected_index: usize,
    pub items: Vec<String>,
}

impl App {
    pub fn quit(&mut self) {
        self.should_quit = true;
    }

    pub fn next(&mut self) {
        if !self.items.is_empty() {
            self.selected_index = (self.selected_index + 1) % self.items.len();
        }
    }
}
```

그리기 함수는 애플리케이션 동작을 변경하지 않고 상태를 수신하고 렌더링해야 합니다.

```rust
use ratatui::{
    layout::{Constraint, Direction, Layout},
    widgets::{Block, Borders, List, ListItem, Paragraph},
    Frame,
};

use crate::app::App;

pub fn draw(frame: &mut Frame, app: &App) {
    let chunks = Layout::default()
        .direction(Direction::Vertical)
        .constraints([Constraint::Min(1), Constraint::Length(1)])
        .split(frame.area());

    let items = app.items.iter().map(|item| ListItem::new(item.as_str()));
    let list = List::new(items).block(Block::default().title("Tasks").borders(Borders::ALL));
    let help = Paragraph::new("q: quit  j/k: move");

    frame.render_widget(list, chunks[0]);
    frame.render_widget(help, chunks[1]);
}
```

이벤트 처리는 작고 명시적일 수 있습니다.

```rust
use crossterm::event::{KeyCode, KeyEvent};

use crate::app::App;

pub fn handle_key(app: &mut App, key: KeyEvent) {
    match key.code {
        KeyCode::Char('q') => app.quit(),
        KeyCode::Char('j') | KeyCode::Down => app.next(),
        _ => {}
    }
}
```

## 호환성 참고사항

CLI 프로젝트는 명시적으로 필요하지 않는 한 관련 없는 웹 프레임워크 및 데이터베이스 선택을 피해야 합니다. 이 가이드에서는 해당 플래그를 `none`로 설정합니다.

터미널 우선 프로젝트의 경우 `--rust-cli ratatui` 및 web/API/database 플래그를 `none`로 설정합니다. 나중에 백엔드나 데이터베이스를 추가하는 경우 이를 하이브리드 앱으로 처리하고 렌더링 코드에 네트워크 호출을 혼합하는 대신 명확한 모듈 경계를 도입하세요.

## 그것을 선택하는 경우

대시보드, 개발자 도구, 인프라 도구 및 터미널 우선 워크플로의 경우 Ratatui을 선택하세요.

이는 사용자가 대부분의 하루를 셸에서 보내고 반복 작업을 위해 집중된 인터페이스가 필요한 경우에 특히 유용합니다.

## 테스트 및 배포 참고 사항

대부분의 Ratatui 로직은 터미널 레이어 아래에서 테스트하는 것이 가장 쉽습니다. 단위 테스트 상태 전환 및 명령 구문 분석:

```bash
cargo test
```

릴리스 빌드의 경우 Cargo의 일반 최적화 빌드를 사용하세요.

```bash
cargo build --release
```

바이너리를 배포할 때 색상 지원, 키보드 단축키, 구성 파일 위치 및 도구가 읽는 모든 환경 변수와 같은 터미널 기대치를 문서화하십시오.

## 문제 해결

- 충돌 후 터미널이 비정상적인 모드로 유지되는 경우 앱이 원시 모드를 복원하고 종료 중에 대체 화면을 벗어나는지 확인하세요.
- 렌더링이 깜박이는 경우 입력이나 틱 없이 이벤트 루프가 공격적으로 다시 그려지지 않는지 확인하세요.
- 예기치 않게 텍스트가 잘리는 경우 좁은 터미널 너비로 테스트하고 하드 코딩된 좌표 대신 Ratatui 레이아웃 제약 조건을 사용하세요.
- 원격 셸에서 키보드 단축키가 작동하지 않는 경우 앱 로직을 변경하기 전에 터미널 에뮬레이터 동작을 비교하세요.
