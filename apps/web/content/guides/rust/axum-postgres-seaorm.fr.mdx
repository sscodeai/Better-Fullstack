Utilisez cette pile lorsque vous souhaitez un backend Rust avec un framework HTTP moderne, une persistance relationnelle, une journalisation et une gestion explicite des erreurs.

```npm
npm create better-fullstack@latest my-rust-api -- \
  --ecosystem rust \
  --rust-web-framework axum \
  --database postgres \
  --rust-orm sea-orm \
  --rust-logging tracing \
  --rust-error-handling anyhow-thiserror
```

## Ce que cela crée

- Un projet Rust API utilisant Axum.
- PostgreSQL comme option de base de données.
- SeaORM pour l'accès à la base de données.
- Traçage pour la journalisation structurée.
- Quoi qu'il en soit, et gestion des erreurs de style Thiserror.

## Forme générée

Cette pile est destinée à produire un service backend, pas une application fullstack. Le projet généré vous donne une limite HTTP typée dans Axum, une couche de base de données appartenant à SeaORM et une journalisation à l'échelle des requêtes via `tracing`.

Les limites de mise en œuvre importantes sont :

- Les routes restent fines et traduisent les entrées HTTP en appels d'application.
- SeaORM modèles et entités possèdent leur propre mappage de base de données.
- Les erreurs sont modélisées explicitement, puis converties en réponses HTTP à la périphérie.
- La configuration est lue une fois au démarrage afin que les pools de bases de données, les routeurs et la journalisation puissent être câblés de manière prévisible.

## Arborescence de fichiers représentative

Les noms de fichiers exacts peuvent changer à mesure que le générateur évolue, mais c'est la forme à attendre d'un Axum API avec PostgreSQL et SeaORM :

```txt
my-rust-api/
  Cargo.toml
  .env.example
  src/
    main.rs
    config.rs
    error.rs
    routes/
      mod.rs
      health.rs
      users.rs
    entities/
      mod.rs
      prelude.rs
      user.rs
    services/
      mod.rs
      users.rs
```

Utilisez `routes/` pour la mécanique HTTP, `services/` pour le comportement des applications et `entities/` pour les types de modèles de base de données générés ou gérés par SeaORM.

## Exemples d'itinéraires et de modèles

Un gestionnaire Axum doit généralement accepter l'état extrait et les données de requête saisies, puis renvoyer un résultat que votre type d'erreur partagé peut convertir en réponse :

```rust
use axum::{extract::State, Json};
use serde::{Deserialize, Serialize};

use crate::{error::AppError, services::users::UserService, AppState};

#[derive(Debug, Deserialize)]
pub struct CreateUserRequest {
    pub email: String,
    pub name: String,
}

#[derive(Debug, Serialize)]
pub struct UserResponse {
    pub id: i32,
    pub email: String,
    pub name: String,
}

pub async fn create_user(
    State(state): State<AppState>,
    Json(input): Json<CreateUserRequest>,
) -> Result<Json<UserResponse>, AppError> {
    let user = UserService::new(state.db)
        .create(input.email, input.name)
        .await?;

    Ok(Json(UserResponse {
        id: user.id,
        email: user.email,
        name: user.name,
    }))
}
```

SeaORM maintient les insertions explicites via les modèles actifs :

```rust
use sea_orm::{ActiveModelTrait, ActiveValue::Set, DatabaseConnection};

use crate::entities::user;

pub async fn create_user(
    db: &DatabaseConnection,
    email: String,
    name: String,
) -> Result<user::Model, sea_orm::DbErr> {
    user::ActiveModel {
        email: Set(email),
        name: Set(name),
        ..Default::default()
    }
    .insert(db)
    .await
}
```

## Quand le choisir

Choisissez Axum lorsque les performances, l'exactitude et le code serveur explicite sont importants. Il convient parfaitement aux API, aux services backend et aux systèmes dans lesquels le système de types de Rust fait partie de la stratégie produit.

## Compromis

Rust demande généralement plus de modélisation de type initiale que TypeScript, Python ou Go. En retour, vous bénéficiez de contrôles rigoureux au moment de la compilation et d’un comportement d’exécution prévisible.

SeaORM est productif lorsque votre équipe souhaite générer des types d'entités et des générateurs de requêtes composables. Il peut sembler plus lourd que SQLx pour les services gourmands en requêtes où le SQL écrit à la main fait partie de la conception.

## Notes de tests et de déploiement

Commencez par tester les gestionnaires et les services séparément. Les tests de gestionnaire peuvent créer un routeur Axum avec un état de test, tandis que les tests de service peuvent s'exécuter sur une base de données PostgreSQL jetable.

```bash
cargo test
```

Avant le déploiement, assurez-vous que l'environnement d'exécution fournit une chaîne de connexion PostgreSQL et toute étape de migration utilisée par votre projet. Pour la production, gardez `RUST_LOG` explicite afin que la sortie de traçage soit utile sans être trop bruyante :

```bash
RUST_LOG=info cargo run
```

## Dépannage

- Si l'application démarre mais que les appels à la base de données échouent, vérifiez que les clés `.env.example` générées correspondent aux variables définies dans votre shell ou votre plateforme de déploiement.
- Si les types SeaORM ne correspondent pas à la base de données, réexécutez l'entité du projet ou le workflow de migration au lieu de modifier manuellement le code généré.
- Si les erreurs renvoient sous forme de réponses génériques 500, vérifiez le type d'erreur partagé et son implémentation `IntoResponse`.
- Si les journaux ne contiennent pas de contexte de demande, confirmez que l'abonné de suivi est initialisé avant que le routeur ne commence à desservir le trafic.
