Verwenden Sie diesen Stack, wenn Sie ein Rust-Backend benötigen, das für Service-to-Service-APIs und typisierte gRPC-Verträge konzipiert ist.

```npm
npm create better-fullstack@latest my-rust-grpc -- \
  --ecosystem rust \
  --rust-web-framework axum \
  --rust-frontend none \
  --rust-orm sqlx \
  --rust-api tonic \
  --rust-cli none \
  --rust-logging tracing \
  --rust-error-handling anyhow-thiserror
```

## Was dadurch entsteht

- Ein Rust-Dienst, der Axum verwendet.
- Tonic für gRPC.
- SQLx für den Datenbankzugriff.
- Nachverfolgung und strukturierte Fehlerbehandlung.

## Erzeugte Form

Dieser Stapel ist für typisierte Service-zu-Service-Aufrufe konzipiert. Tonic besitzt den gRPC-Vertrag und die Serverimplementierung, SQLx besitzt den Datenbankzugriff und Axum kann weiterhin betriebsbereite HTTP-Routen wie Gesundheitsprüfungen bereitstellen, wenn das generierte Projekt diese enthält.

Die Kerngrenzen sind:

- `proto/` definiert den Servicevertrag.
- Generierte Rust-Typen gRPC bleiben nah an der Dienstimplementierung Tonic.
- SQLx-Abfragen bleiben in Repository- oder Servicemodulen und nicht in Transporthandlern.
- Axum-Routen sind, falls verwendet, betriebliche Endpunkte und nicht das Hauptprodukt API.

In der Datei `build.rs` wird normalerweise die Protobuf-Generierung verkabelt. Halten Sie generierten Code von handbearbeiteter Geschäftslogik fern.

## Prototypen- und Servicebeispiele

Ein kleiner Protobuf-Vertrag gibt Clients und Servern die gleiche typisierte Grenze:

```proto
syntax = "proto3";

package users.v1;

service UserService {
  rpc GetUser(GetUserRequest) returns (User);
}

message GetUserRequest {
  int64 id = 1;
}

message User {
  int64 id = 1;
  string email = 2;
  string name = 3;
}
```

Die Tonic-Implementierung sollte Transportfehler in gRPC-Status übersetzen und den Datenbankzugriff in einem separaten Modul behalten:

```rust
use tonic::{Request, Response, Status};

use crate::grpc::users_v1::{
    user_service_server::UserService, GetUserRequest, User,
};
use crate::repositories::users::UserRepository;

pub struct UsersGrpc {
    repo: UserRepository,
}

#[tonic::async_trait]
impl UserService for UsersGrpc {
    async fn get_user(
        &self,
        request: Request<GetUserRequest>,
    ) -> Result<Response<User>, Status> {
        let id = request.into_inner().id;
        let user = self
            .repo
            .find_by_id(id)
            .await
            .map_err(|error| Status::internal(error.to_string()))?
            .ok_or_else(|| Status::not_found("user not found"))?;

        Ok(Response::new(User {
            id: user.id,
            email: user.email,
            name: user.name,
        }))
    }
}
```

SQLx behält die Abfrageform explizit bei:

```rust
use sqlx::PgPool;

pub struct UserRow {
    pub id: i64,
    pub email: String,
    pub name: String,
}

pub async fn find_user(pool: &PgPool, id: i64) -> Result<Option<UserRow>, sqlx::Error> {
    sqlx::query_as!(
        UserRow,
        "select id, email, name from users where id = $1",
        id
    )
    .fetch_optional(pool)
    .await
}
```

## Kompatibilitätshinweise

Behalten Sie `--rust-api tonic` bei, wenn Sie die Serviceverkabelung gRPC generieren möchten. Wenn Sie `--rust-api none` ändern, werden die Beispiele protobuf und Tonic in diesem Handbuch zu einer manuellen Ergänzung. SQLx eignet sich gut für Serviceverträge, da die Abfrageausgabe direkt den Protobuf-Antworttypen zugeordnet werden kann.

## Wann man es wählt

Wählen Sie dies für interne Dienste, Microservices und Backend-Systeme, bei denen gRPC-Verträge besser passen als JSON-Routen.

## Kompromisse

gRPC eignet sich hervorragend für stark typisierte Servicegrenzen, fügt jedoch Protokoll- und Toolerwartungen hinzu. Beginnen Sie für browserseitige HTTP-APIs mit [Axum mit PostgreSQL und SeaORM](/guides/rust/axum-postgres-seaorm/).

SQLx hält das Datenbankverhalten transparent, Sie müssen jedoch SQL-Dateien, Migrationen und Abfragekompatibilität bewusst verwalten.

## Test- und Bereitstellungshinweise

Testen Sie das Verhalten auf Protobuf-Ebene mit generierten Clients oder Tonic-Anforderungstypen und testen Sie dann Repository-Funktionen anhand einer echten PostgreSQL-Datenbank, wenn die Abfragekorrektheit wichtig ist.

```bash
cargo test
```

Stellen Sie in CI sicher, dass Protobuf-Tools verfügbar sind, wenn das Projekt während des Builds Rust-Typen generiert. Stellen Sie für die Bereitstellung den von Clients erwarteten Port gRPC bereit und halten Sie alle HTTP-Integritätsrouten von der Dienstadresse gRPC getrennt, wenn Ihre Plattform unterschiedliche Tests erfordert.

## Fehlerbehebung

- Wenn Protobuf-Typen fehlen, überprüfen Sie `build.rs` und stellen Sie sicher, dass der Pfad `.proto` enthalten ist.
- Wenn die Kompilierungszeitprüfungen von SQLx fehlschlagen, überprüfen Sie die Datenbank-URL oder die Offline-Abfragemetadaten, die vom generierten Projekt erwartet werden.
- Wenn Clients `UNIMPLEMENTED` erhalten, bestätigen Sie, dass der generierte Dienstserver zum Server-Builder Tonic hinzugefügt wird.
- Wenn die Integritätsprüfungen erfolgreich sind, gRPC-Aufrufe jedoch fehlschlagen, überprüfen Sie, ob der gRPC-Port derjenige ist, der von Ihrem Prozessmanager oder Container bereitgestellt wird.
