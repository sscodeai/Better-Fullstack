생성된 앱의 서버 측과 프런트엔드 측 모두에서 Rust를 원할 때 이 스택을 사용하세요.

```npm
npm create better-fullstack@latest my-rust-fullstack -- \
  --ecosystem rust \
  --rust-web-framework axum \
  --rust-frontend leptos \
  --rust-orm sea-orm \
  --rust-api none \
  --rust-cli none \
  --rust-logging tracing \
  --rust-error-handling anyhow-thiserror
```

## 이것이 만드는 것

- Axum 기반 Rust 서버.
- Rust 프런트엔드 옵션으로 Leptos.
- 데이터베이스 액세스용 SeaORM.
- 추적 및 구조적 오류 처리 기본값.

## 생성된 모양

이 스택은 Rust 우선 풀스택 앱입니다. Axum는 서버 런타임을 소유하고, Leptos은 프런트엔드 표면을 소유하며, SeaORM은 백엔드에 형식화된 지속성 레이어를 제공합니다.

생성된 앱은 하나의 Rust 작업공간을 보유하려는 경우에 유용합니다.

- 서버 시작, 라우팅 및 공유 상태.
- Leptos 뷰 및 대화형 UI 코드.
- 데이터베이스 엔터티 및 서버 측 데이터 액세스.
- 로깅 및 오류 처리가 처음부터 구성됩니다.

## 대표 파일 트리

하나의 프로젝트에 유지하면서 서버 문제와 UI 문제를 분리하는 레이아웃을 기대하세요.

```txt
my-rust-fullstack/
  Cargo.toml
  .env.example
  src/
    main.rs
    app.rs
    error.rs
    routes/
      mod.rs
      api.rs
    entities/
      mod.rs
      user.rs
    pages/
      mod.rs
      home.rs
      users.rs
    components/
      mod.rs
```

생성된 트리를 경계 가이드로 사용하세요. Leptos 구성 요소는 렌더링 및 상호 작용에 집중해야 하고 데이터베이스 작업은 서버 측에서 유지되어야 합니다.

## 서버 및 UI 예시

Axum 경로는 서버 측 작업 또는 통합 클라이언트에 대한 JSON 엔드포인트를 노출할 수 있습니다.

```rust
use axum::{extract::State, Json};
use serde::Serialize;

use crate::{error::AppError, AppState};

#[derive(Debug, Serialize)]
pub struct HealthResponse {
    pub status: &'static str,
}

pub async fn health(State(_state): State<AppState>) -> Result<Json<HealthResponse>, AppError> {
    Ok(Json(HealthResponse { status: "ok" }))
}
```

Leptos 페이지는 UI 코드에서 데이터베이스 연결을 여는 대신 서버 함수 또는 API 도우미를 호출해야 합니다.

```rust
use leptos::prelude::*;

#[component]
pub fn HomePage() -> impl IntoView {
    view! {
        <main>
            <h1>"Dashboard"</h1>
            <p>"Rust server, Rust UI, and one generated project shape."</p>
        </main>
    }
}
```

데이터베이스 지원 UI의 경우 서버 경계 뒤에 데이터베이스 작업을 유지합니다.

```rust
use sea_orm::{DatabaseConnection, EntityTrait};

use crate::entities::user;

pub async fn list_users(db: &DatabaseConnection) -> Result<Vec<user::Model>, sea_orm::DbErr> {
    user::Entity::find().all(db).await
}
```

## 호환성 참고사항

이 가이드에서는 TypeScript 프런트엔드/백엔드 플래그 대신 Rust 관련 플래그를 사용합니다. Rust 프런트엔드 선택 항목을 `--rust-frontend` 아래에 유지하세요.

`--rust-api none` 및 `--rust-cli none` 플래그는 이 스캐폴드를 웹 앱에 계속 집중시킵니다. 생성된 프로젝트가 의도적으로 여러 인터페이스를 제공하는 경우에만 gRPC 또는 CLI 옵션을 추가하세요. API 전용 백엔드를 원한다면 `--rust-frontend none`를 사용하고 대신 Axum API 가이드를 따르세요.

## 그것을 선택하는 경우

프로젝트가 앱 경계 전반에 걸쳐 Rust 우선 아키텍처의 이점을 누리고 팀이 Rust의 컴파일 타임 모델에 익숙해지면 이 옵션을 선택하세요.

이는 JavaScript 프런트엔드 생태계를 채택하는 것보다 서버와 UI 전반에 걸쳐 Rust 유형과 검증을 공유하는 것이 더 가치 있는 내부 도구, 관리 표면 및 제품 영역에 매우 적합합니다.

## 트레이드오프

Fullstack Rust는 언어 전환을 줄이지만 프런트엔드 작업 흐름은 React, Vue 또는 Svelte와 다릅니다. Rust 컴파일 시간, Leptos 특정 라우팅 패턴 및 Rust 생성 UI 코드에서 브라우저 동작을 디버깅하는 데 능숙한 팀을 계획하세요.

## 테스트 및 배포 참고 사항

서비스 및 모델 로직에 대해 Rust 단위 테스트를 사용한 다음 가장 중요한 흐름 주위에 경로 또는 브라우저 수준 테스트를 추가하세요.

```bash
cargo test
```

배포를 위해 앱의 두 부분을 모두 확인하세요. 즉, 서버는 환경 변수를 받아야 하며 프런트엔드 자산은 생성된 프로젝트 스크립트를 사용하여 구축되어야 합니다. 앱이 사용자 데이터를 저장하는 경우 배포 경로에 데이터베이스 마이그레이션을 유지합니다.

## 문제 해결

- UI가 빌드되었지만 서버 데이터가 누락된 경우 서버 경로 또는 서버 기능이 Axum 라우터에 등록되어 있는지 확인하세요.
- 데이터베이스 유형이 드리프트되면 스캐폴드 문서의 워크플로를 통해 SeaORM 엔터티를 재생성하거나 업데이트합니다.
- 브라우저 경로가 로컬에서는 작동하지만 배포 후에는 작동하지 않는 경우 생성된 서버 라우팅과 정적 자산 구성을 함께 확인하세요.
- 로그가 너무 조용하면 서버를 시작하기 전에 `RUST_LOG=info`을 설정하세요.
