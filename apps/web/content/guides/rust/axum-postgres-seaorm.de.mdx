Verwenden Sie diesen Stack, wenn Sie ein Rust-Backend mit einem modernen HTTP-Framework, relationaler Persistenz, Protokollierung und expliziter Fehlerbehandlung wünschen.

```npm
npm create better-fullstack@latest my-rust-api -- \
  --ecosystem rust \
  --rust-web-framework axum \
  --database postgres \
  --rust-orm sea-orm \
  --rust-logging tracing \
  --rust-error-handling anyhow-thiserror
```

## Was dadurch entsteht

- Ein Rust API-Projekt mit Axum.
- PostgreSQL als Datenbankoption.
- SeaORM für den Datenbankzugriff.
- Ablaufverfolgung für strukturierte Protokollierung.
- Wie auch immer und Fehlerbehandlung im Thiserror-Stil.

## Erzeugte Form

Dieser Stack soll einen Backend-Dienst und keine Fullstack-App erstellen. Das generierte Projekt bietet Ihnen eine typisierte HTTP-Grenze in Axum, einer Datenbankebene im Besitz von SeaORM, und eine anforderungsbezogene Protokollierung über `tracing`.

Die wichtigen Implementierungsgrenzen sind:

- Routen bleiben dünn und übersetzen HTTP-Eingaben in Anwendungsaufrufe.
- SeaORM Modelle und Entitäten besitzen eine eigene Datenbankzuordnung.
- Fehler werden explizit modelliert und dann am Edge in HTTP-Antworten umgewandelt.
- Die Konfiguration wird beim Start einmal gelesen, sodass Datenbankpools, Router und Protokollierung vorhersehbar verkabelt werden können.

## Repräsentativer Dateibaum

Die genauen Dateinamen können sich mit der Weiterentwicklung des Generators ändern, aber dies ist die Form, die man von einem Axum API mit PostgreSQL und SeaORM erwarten kann:

```txt
my-rust-api/

      health.rs

    entities/
      mod.rs

      mod.rs

```

Verwenden Sie `routes/` für HTTP-Mechaniken, `services/` für das Anwendungsverhalten und `entities/` für die generierten oder verwalteten Datenbankmodelltypen von SeaORM.

## Routen- und Modellbeispiele

Ein Axum-Handler sollte normalerweise extrahierte Status- und typisierte Anforderungsdaten akzeptieren und dann ein Ergebnis zurückgeben, das Ihr gemeinsamer Fehlertyp in eine Antwort umwandeln kann:

```rust
use axum::{extract::State, Json};
use serde::{Deserialize, Serialize};

use crate::{error::AppError, services::users::UserService, AppState};

#[derive(Debug, Deserialize)]
pub struct CreateUserRequest {
    pub email: String,
    pub name: String,
}

#[derive(Debug, Serialize)]
pub struct UserResponse {
    pub id: i32,
    pub email: String,
    pub name: String,
}

pub async fn create_user(
    State(state): State<AppState>,
    Json(input): Json<CreateUserRequest>,
) -> Result<Json<UserResponse>, AppError> {
    let user = UserService::new(state.db)
        .create(input.email, input.name)
        .await?;

    Ok(Json(UserResponse {
        id: user.id,
        email: user.email,
        name: user.name,
    }))
}
```

SeaORM hält Einfügungen durch aktive Modelle explizit:

```rust
use sea_orm::{ActiveModelTrait, ActiveValue::Set, DatabaseConnection};

use crate::entities::user;

pub async fn create_user(
    db: &DatabaseConnection,
    email: String,
    name: String,
) -> Result<user::Model, sea_orm::DbErr> {
    user::ActiveModel {
        email: Set(email),
        name: Set(name),
        ..Default::default()
    }
    .insert(db)
    .await
}
```

## Kompatibilitätshinweise

<GuideCompatibilityNote kind="supported" title="Compatibility rule">
Halten Sie `--rust-web-framework axum`, `--database postgres` und `--rust-orm sea-orm` zusammen, wenn Sie die Form dieser Anleitung wünschen. Wenn Sie den ORM auf `sqlx` umstellen, erwarten Sie mehr handgeschriebenes SQL und weniger Modelltypen im Entitätsstil. Wenn Sie `--rust-web-framework none` festlegen, gelten die Routenbeispiele in diesem Handbuch nicht mehr.
</GuideCompatibilityNote>

## Wann man es wählt

Wählen Sie Axum, wenn Leistung, Korrektheit und expliziter Servercode wichtig sind. Es eignet sich gut für APIs, Backend-Dienste und Systeme, bei denen das Typsystem von Rust Teil der Produktstrategie ist.

## Kompromisse

Rust verlangt normalerweise mehr Vorab-Typmodellierung als TypeScript, Python oder Go. Im Gegenzug erhalten Sie starke Prüfungen zur Kompilierungszeit und ein vorhersehbares Laufzeitverhalten.

SeaORM ist produktiv, wenn Ihr Team generierte Entitätstypen und zusammensetzbare Abfrage-Builder benötigt. Bei abfrageintensiven Diensten, bei denen handgeschriebenes SQL Teil des Designs ist, kann es sich schwerer anfühlen als SQLx.

## Test- und Bereitstellungshinweise

Beginnen Sie damit, Handler und Dienste separat zu testen. Handler-Tests können einen Axum-Router mit Teststatus erstellen, während Diensttests für eine verfügbare PostgreSQL-Datenbank ausgeführt werden können.

```bash
cargo test
```

Stellen Sie vor der Bereitstellung sicher, dass die Laufzeitumgebung eine PostgreSQL-Verbindungszeichenfolge und alle von Ihrem Projekt verwendeten Migrationsschritte bereitstellt. Halten Sie für die Produktion `RUST_LOG` explizit, damit die Ablaufverfolgung der Ausgabe nützlich ist, ohne zu laut zu sein:

```bash
RUST_LOG=info cargo run
```

## Fehlerbehebung

- Wenn die App gestartet wird, Datenbankaufrufe jedoch fehlschlagen, überprüfen Sie, ob die generierten `.env.example`-Schlüssel mit den in Ihrer Shell oder Bereitstellungsplattform festgelegten Variablen übereinstimmen.
- Wenn die Typen SeaORM nicht mit der Datenbank übereinstimmen, führen Sie den Entitäts- oder Migrationsworkflow des Projekts erneut aus, anstatt den generierten Code manuell zu bearbeiten.
- Wenn Fehler als generische 500-Antworten zurückgegeben werden, überprüfen Sie den gemeinsamen Fehlertyp und seine `IntoResponse`-Implementierung.
- Wenn in den Protokollen der Anforderungskontext fehlt, vergewissern Sie sich, dass der Tracing-Abonnent initialisiert wurde, bevor der Router mit der Verarbeitung des Datenverkehrs beginnt.
