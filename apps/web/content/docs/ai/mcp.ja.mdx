Better Fullstack MCP サーバーを使用すると、エージェントは CLI フラグを推測する代わりに、構造化されたツール呼び出しを通じてプロジェクトをスキャフォールディングできます。スキーマの検索、互換性の検証、ドライラン プレビュー、プロジェクトの作成、機能の追加を処理します。

このページでは、Better Fullstack MCP サーバーについて説明します。生成されたプロジェクトの `mcp` アドオンは別のものです。このアドオンは、MCP クライアント/サーバー構成をスキャフォールディングされたアプリに書き込み、外部セットアップ コマンドを実行する場合があります。

より軽量な CLI 主導のエージェント ワークフローの場合は、[Better Fullstack エージェント スキル](/docs/ai/skills/) をインストールします。スキルと MCP サーバーはどちらも同じジェネレーターを使用します。スキルは高速なフラグベースのスキャフォールディング用に最適化されており、MCP は構造化されたツール呼び出し用に最適化されています。

## 設定

使用するパッケージ マネージャーを使用してサーバーを起動します。

<PMTabs
  npm="npx -y create-better-fullstack@latest mcp"
  pnpm="pnpm dlx create-better-fullstack@latest mcp"
  bun="bunx create-better-fullstack@latest mcp"
  yarn="yarn dlx create-better-fullstack@latest mcp"
/>

### Claude Code

```bash
claude mcp add --transport stdio better-fullstack -- npx -y create-better-fullstack@latest mcp
```

pnpm、Bun、または Yarn を使用する場合は、上記と同じランチャー テーブルを使用します。

### カーソル

```json
{
  "mcpServers": {
    "better-fullstack": {
      "command": "npx",
      "args": ["-y", "create-better-fullstack@latest", "mcp"]
    }
  }
}
```

### VSコード

```json
{
  "servers": {
    "better-fullstack": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "create-better-fullstack@latest", "mcp"]
    }
  }
}
```

## ツール

完全な入力、出力、例、安全な書き込み動作については、[MCP ツール リファレンス](/docs/ai/mcp-tools/) を参照してください。

| 道具 | 目的 |
| --- | --- |
| `bfs_get_guidance` | ワークフロー ルールとフィールド セマンティクスを返します。まずはこれに電話してください。 |
| `bfs_get_schema` | 構成カテゴリの有効なオプションを返します。 |
| `bfs_check_compatibility` | スタックを検証し、警告付きで調整された選択を返します。 |
| `bfs_plan_project` | メモリ内でプロジェクト生成をドライランします。 |
| `bfs_create_project` | 新しいプロジェクトをディスクに書き込みます。 |
| `bfs_plan_addition` | 既存のプロジェクトのアドオンの変更を検証します。 |
| `bfs_add_feature` | サポートされているアドオンを既存のプロジェクトに追加します。 |

<Callout kind="info" title="Install behavior">
MCP プロジェクトの作成では、長時間実行されるエージェント呼び出しを回避するために、通常の依存関係のインストールがスキップされます。生成されたプロジェクトに `mcp` アドオンが含まれている場合、そのアドオン セットアップは引き続き MCP クライアント構成の外部セットアップ コマンドを実行する可能性があります。
</Callout>

## リソース

サーバーは次の MCP リソースを公開します。

| リソース | コンテンツ |
| --- | --- |
| `docs://compatibility-rules` | スタックの組み合わせの互換性ルール。 |
| `docs://stack-options` | カテゴリごとに利用可能なテクノロジー オプション。 |
| `docs://getting-started` | サポートされているエコシステムのクイックスタート レシピ。 |

## 推奨されるツールの順序

新しいプロジェクトの場合:

1. `bfs_get_guidance`
2. `bfs_get_schema`
3. `bfs_check_compatibility`
4. `bfs_plan_project`
5. `bfs_create_project`

既存のプロジェクトの場合:

1. `bfs_plan_addition`
2. `bfs_add_feature`

## エージェント向けの重要なルール

- `frontend` は配列であり、複数のフロントエンド ターゲットを含めることができます。
- `none` はカテゴリを無効にします。
- エージェントが正しいフィールド ファミリを使用できるように、最初に `ecosystem` を設定します。
- 配列フィールドには、`frontend`、`addons`、`examples`、`rustLibraries`、`pythonAi`、`javaLibraries`、および `javaTestingLibraries` が含まれます。
- プロジェクトの計画、作成、および追加の応答には、グラフに基づくスタック ビューの `graphSummary`、`effectiveStack`、および `stackPartSpecs` が含まれます。
- 現在の MCP プロジェクト スキーマは、`aiDocs` を作成フィールドとして公開しません。プロジェクトの作成では、サーバーの組み込み AI-doc 動作が使用されます。
- プロジェクトを作成する前に、必ず `bfs_check_compatibility` を呼び出してください。

## エージェントプロンプト

```text
Use Better Fullstack MCP. First call bfs_get_guidance to understand
the rules, then bfs_get_schema for the relevant categories, then
bfs_check_compatibility to validate the stack, then bfs_plan_project
for a dry-run. Only call bfs_create_project after the dry-run succeeds.
Skip normal dependency installation and tell me the exact commands to
install, test, and run the project.
```
