---
title: MCP Référence des outils
description: Référence détaillée pour les outils, les entrées, les sorties et les flux de travail sécurisés Better Fullstack MCP.
---

Le serveur MCP expose des outils structurés pour les agents. Utilisez-les au lieu de demander à un agent de deviner les indicateurs CLI à partir de prose ou de copier manuellement de longues commandes.

## Flux de travail recommandés

### Nouveau projet

1. `bfs_get_guidance`
2. `bfs_get_schema`
3. `bfs_check_compatibility`
4. `bfs_plan_project`
5. `bfs_create_project`

### Projet existant

1. `bfs_get_guidance`
2. `bfs_plan_addition`
3. `bfs_add_feature`

<Callout kind="warn" title="Mutation behavior">
`bfs_plan_project` est un essai à sec uniquement. `bfs_create_project` écrit un nouveau projet sur le disque. `bfs_add_feature` mute un projet existant. La création de projets MCP et l'ajout de fonctionnalités n'installent pas de dépendances.
</Callout>

<Callout kind="info" title="Graph metadata">
Les outils de planification, de création et d'ajout de projet incluent `graphSummary`, `effectiveStack` et `stackPartSpecs` afin que les agents puissent inspecter la même vue de pile dérivée d'un graphique que `bts.jsonc` persiste.
</Callout>

## Référence de l'outil

### `bfs_get_guidance`

| Champ | Détails |
| --- | --- |
| But | Renvoie les règles de workflow, la sémantique des champs, les règles d'ambiguïté et les contraintes critiques. |
| Quand appeler | Tout d’abord, avant les autres outils Better Fullstack MCP. |
| Écrit des fichiers | Non. |
| Entrées requises | Aucun. |
| Entrées facultatives | Aucun. |
| Sortir | Conseils JSON couvrant l’ordre des outils recommandé, les champs de tableau, les règles de l’écosystème et des exemples. |

```json
{}
```

### `bfs_get_schema`

| Champ | Détails |
| --- | --- |
| But | Renvoie les valeurs d'option valides pour une catégorie ou toutes les catégories. |
| Quand appeler | Avant de planifier une pile, en particulier lorsque l'utilisateur a nommé les technologies de manière informelle. |
| Écrit des fichiers | Non. |
| Entrées requises | Aucun. |
| Entrées facultatives | `category`, `ecosystem`. |
| Sortir | Données de schéma/options JSON filtrées par catégorie ou écosystème. |

```json
{
  "category": "database",
  "ecosystem": "typescript"
}
```

### `bfs_check_compatibility`

| Champ | Détails |
| --- | --- |
| But | Valide une combinaison de pile et renvoie les sélections ajustées avec des avertissements. |
| Quand appeler | Avant le `bfs_plan_project` ou le `bfs_create_project`. |
| Écrit des fichiers | Non. |
| Entrées requises | `ecosystem`. |
| Entrées facultatives | Frontend, backend, runtime, base de données, ORM, API, authentification, services, UI/CSS, champs mobiles, modules complémentaires, exemples, gestionnaire de packages, champs de déploiement et champs spécifiques à l'écosystème. |
| Sortir | Analyse de compatibilité JSON avec les modifications, les notes et les données de pile ajustées. |

```json
{
  "ecosystem": "typescript",
  "frontend": ["next"],
  "backend": "self",
  "runtime": "none",
  "database": "postgres",
  "orm": "drizzle",
  "api": "trpc"
}
```

### `bfs_plan_project`

| Champ | Détails |
| --- | --- |
| But | Génère un projet en mémoire et renvoie un résumé de l'arborescence des fichiers. |
| Quand appeler | Après validation de compatibilité et avant écriture des fichiers. |
| Écrit des fichiers | Non. |
| Entrées requises | Aucun, mais `projectName` et `ecosystem` sont recommandés. |
| Entrées facultatives | Sélections complètes de la pile de projets. |
| Sortir | Indicateur de réussite JSON plus résumé du fichier/répertoire généré, `graphSummary`, `effectiveStack` et `stackPartSpecs`. |

```json
{
  "projectName": "my-app",
  "ecosystem": "typescript",
  "frontend": ["tanstack-router"],
  "backend": "hono",
  "runtime": "bun",
  "database": "sqlite",
  "orm": "drizzle"
}
```

### `bfs_create_project`

| Champ | Détails |
| --- | --- |
| But | Écrit un nouveau projet généré sur le disque. |
| Quand appeler | Seulement après que `bfs_plan_project` réussisse et que l'utilisateur approuve. |
| Écrit des fichiers | Oui, crée le répertoire du projet et les fichiers générés. |
| Entrées requises | `projectName`. |
| Entrées facultatives | Sélections complètes de la pile de projets. |
| Sortir | Indicateur de réussite JSON, répertoire du projet, nombre de fichiers, métadonnées du graphique, avertissements du module complémentaire et instructions de commande d'installation. |

```json
{
  "projectName": "my-app",
  "ecosystem": "java",
  "javaWebFramework": "spring-boot",
  "javaBuildTool": "maven",
  "javaTestingLibraries": ["junit5"]
}
```

### `bfs_plan_addition`

| Champ | Détails |
| --- | --- |
| But | Lit `bts.jsonc` et valide les ajouts proposés pour un projet existant. |
| Quand appeler | Avant de muter un projet existant avec `bfs_add_feature`. |
| Écrit des fichiers | Non. |
| Entrées requises | `projectDir`. |
| Entrées facultatives | `addons`, `webDeploy`, `serverDeploy`. |
| Sortir | Résumé de la configuration existante, ajouts proposés, métadonnées graphiques pour les deux, modules complémentaires déjà présents et avertissements de compatibilité. |

```json
{
  "projectDir": "/absolute/path/to/my-app",
  "addons": ["mcp", "skills"]
}
```

### `bfs_add_feature`

| Champ | Détails |
| --- | --- |
| But | Ajoute des modules complémentaires ou des paramètres de déploiement pris en charge à un projet Better Fullstack existant. |
| Quand appeler | Après que `bfs_plan_addition` réussisse et que l'utilisateur approuve. |
| Écrit des fichiers | Oui, mute le projet existant. |
| Entrées requises | `projectDir`. |
| Entrées facultatives | `addons`, `webDeploy`, `serverDeploy`, `packageManager`. |
| Sortir | Résultat JSON du gestionnaire d'ajout, graphique des métadonnées après la mise à jour et instructions d'installation lorsqu'elles sont disponibles. |

```json
{
  "projectDir": "/absolute/path/to/my-app",
  "addons": ["biome"],
  "packageManager": "bun"
}
```

## Règles des agents

- Traitez `frontend`, `addons`, `examples`, `rustLibraries`, `pythonAi`, `javaLibraries` et `javaTestingLibraries` comme des tableaux.
- Définissez `ecosystem` avant de sélectionner les champs spécifiques à l'écosystème.
- Préférez `none` lorsque l'utilisateur souhaite explicitement désactiver une catégorie.
- La création de projet MCP n'expose actuellement pas tous les indicateurs de flux de travail CLI. Par exemple, `aiDocs` utilise la valeur par défaut intégrée du serveur lors de la création du projet.
- N'appelez pas les outils d'écriture tant que l'utilisateur n'a pas approuvé le plan d'exécution à sec ou d'ajout.
- Indiquez à l'utilisateur quelles commandes d'installation/test exécuter après la création ou l'ajout de MCP, car MCP n'installe pas de dépendances.
