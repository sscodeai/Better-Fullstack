---
title: Déploiement
description: Choisissez les cibles de déploiement Web et serveur dans une seule section de pile.
---

Le déploiement est une section de pile, et non un pilier de documentation distinct. Il contrôle si Better Fullstack émet des fichiers de plateforme lors de la création du projet avec `--web-deploy` et `--server-deploy`.

Utilisez les indicateurs de déploiement au moment de l'échafaudage lorsque vous souhaitez que les fichiers de plateforme générés correspondent à la disposition du frontend, du backend, du runtime et de l'espace de travail sélectionnés.

## Indicateurs de déploiement

| Drapeau | Valeurs |
| --- | --- |
| `--web-deploy` | `vercel` `cloudflare` `fly` `railway` `render` `netlify` `docker` `sst` `none` |
| `--server-deploy` | `vercel` `cloudflare` `fly` `railway` `render` `docker` `sst` `none` |

Les choix de déploiement sont vérifiés en termes de compatibilité par rapport à la pile. Certains frameworks fullstack se déploient via l'application Web, certains backends n'ont pas besoin d'un déploiement de serveur séparé et Convex ou Encore gèrent leur propre chemin de plateforme.

## Cibles

| Cible | Meilleur ajustement | Remarques |
| --- | --- | --- |
| Docker | Auto-hébergement, images de production reproductibles et orchestration des services locaux. | Séparé du module complémentaire `docker-compose`, qui se concentre sur les services locaux. |
| Vercel | Déploiements React, Nuxt, SvelteKit, Solid et sans serveur pris en charge. | Les interfaces Fullstack se déploient généralement via l'application Web plutôt que via un serveur cible distinct. |
| Flare nuageuse | Piles d’exécution Workers et serveurs Hono de pointe. | La compatibilité s'étend aux environnements d'exécution de serveur compatibles avec Workers. |
| Fly.io | Hébergement d'applications soutenu par Docker avec placement régional et services de longue durée. | Bon pour les processus serveur persistants. |
| Chemin de fer | Hébergement d'applications connectées à Git avec des versions basées sur Dockerfile et des services gérés. | Configurez les variables de l'environnement de production avant le déploiement. |
| Rendre | Hébergement d'applications basé sur un plan avec des versions basées sur Dockerfile (`render.yaml` à la racine du dépôt). | Déploiement Web et serveur en tant que services Docker distincts ; définissez les variables d'environnement dans le tableau de bord de rendu. |
| Netlifier | Frontends Web statiques et SSR (Next.js, Vite SPA, SvelteKit, Nuxt). | Cible Web uniquement ; définissez le répertoire de base du site sur `apps/web`. |
| SST | Infrastructure orientée AWS sous forme de code pour les workflows d'application TypeScript. | Idéal lorsque vous souhaitez posséder explicitement des ressources AWS. |

## Avant le déploiement

- Définissez les variables d'environnement de production pour les secrets d'authentification, les URL de base de données, les clés de fournisseur et les URL d'applications publiques.
- Exécutez des migrations de bases de données sur la base de données de production avant d'acheminer le trafic.
- Confirmez que la configuration de plate-forme générée correspond à l'application qui doit recevoir les demandes.
- Conservez les conteneurs de services locaux uniquement, les informations d'identification de test et les journaux générés hors du contrôle de code source.

## Ajouter après l'échafaudage

La commande `add` peut enregistrer les indicateurs de déploiement, mais elle traite actuellement les modules complémentaires en premier. Si aucun nouveau module complémentaire n'est sélectionné, les commandes `add` de déploiement uniquement se terminent plus tôt. Préférez choisir les cibles de déploiement pendant `create` jusqu'à ce que les flux d'ajout de déploiement uniquement soient développés.
