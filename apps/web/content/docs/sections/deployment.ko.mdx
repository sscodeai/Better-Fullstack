---

배포는 별도의 문서 기둥이 아닌 하나의 스택 섹션입니다. `--web-deploy` 및 `--server-deploy`을 사용하여 프로젝트를 생성하는 동안 Better Fullstack이 플랫폼 파일을 내보낼지 여부를 제어합니다.

생성된 플랫폼 파일이 선택한 프런트엔드, 백엔드, 런타임 및 작업 영역 레이아웃과 일치하도록 하려면 스캐폴드 시 배포 플래그를 사용하세요.

## 배포 플래그

| 깃발 | 가치 |
| --- | --- |
| `--web-deploy` | `vercel` `cloudflare` `fly` `railway` `render` `netlify` `docker` `sst` `none` |
| `--server-deploy` | `vercel` `cloudflare` `fly` `railway` `render` `docker` `sst` `none` |

배포 선택 사항은 스택에 대해 호환성을 검사합니다. 일부 전체 스택 프레임워크는 웹 앱을 통해 배포되고, 일부 백엔드는 별도의 서버 배포가 필요하지 않으며, Convex 또는 Encore는 자체 플랫폼 경로를 관리합니다.

## 대상

| 목표 | 최적의 핏 | 메모 |
| --- | --- | --- |
| 도커 | 자체 호스팅, 재현 가능한 프로덕션 이미지 및 로컬 서비스 오케스트레이션. | 로컬 서비스에 초점을 맞춘 `docker-compose` 애드온과 별개입니다. |
| 베르셀 | 지원되는 React, Nuxt, SvelteKit, Solid 및 서버리스 배포. | 풀스택 프런트엔드는 일반적으로 별도의 서버 대상이 아닌 웹앱을 통해 배포됩니다. |
| 클라우드플레어 | 작업자 런타임 스택 및 에지 우선 Hono 서버. | 호환성 범위는 작업자 호환 서버 런타임으로 지정됩니다. |
| Fly.io | 지역 배치 및 장기 실행 서비스를 갖춘 Docker 지원 앱 호스팅입니다. | 지속적인 서버 프로세스에 적합합니다. |
| 철도 | Dockerfile 기반 빌드 및 관리형 서비스를 갖춘 Git 연결 앱 호스팅입니다. | 배포하기 전에 프로덕션 환경 변수를 구성합니다. |
| 세우다 | Dockerfile 기반 빌드(repo 루트의 `render.yaml`)를 사용한 Blueprint 기반 앱 호스팅. | 웹과 서버는 별도의 Docker 서비스로 배포됩니다. 렌더 대시보드에서 환경 변수를 설정합니다. |
| 넷티파이 | 정적 및 SSR 웹 프런트엔드(Next.js, Vite SPA, SvelteKit, Nuxt). | 웹 전용 대상; 사이트 기본 디렉터리를 `apps/web`로 설정합니다. |
| SST | TypeScript 앱 워크플로를 위한 코드형 AWS 기반 인프라입니다. | AWS 리소스를 명시적으로 소유하려는 경우에 가장 적합합니다. |

## 배포하기 전에

- 인증 비밀, 데이터베이스 URL, 공급자 키 및 공개 앱 URL에 대한 프로덕션 환경 변수를 설정합니다.
- 트래픽을 라우팅하기 전에 프로덕션 데이터베이스에 대해 데이터베이스 마이그레이션을 실행하세요.
- 생성된 플랫폼 구성이 요청을 수신해야 하는 앱과 일치하는지 확인하세요.
- 로컬 전용 서비스 컨테이너, 테스트 자격 증명, 생성된 로그를 소스 제어 외부에 보관하세요.

## 비계 뒤에 추가

`add` 명령은 배포 플래그를 기록할 수 있지만 현재는 애드온을 먼저 처리합니다. 새 애드온을 선택하지 않으면 배포 전용 `add` 명령이 일찍 종료됩니다. 배포 전용 추가 흐름이 확장될 때까지 `create` 중에 배포 대상을 선택하는 것이 좋습니다.
