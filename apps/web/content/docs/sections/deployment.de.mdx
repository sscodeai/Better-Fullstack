---
title: Einsatz
description: Wählen Sie Web- und Serverbereitstellungsziele als einen Stack-Abschnitt aus.
---

Die Bereitstellung erfolgt in einem Stapelabschnitt und nicht in einer separaten Dokumentationssäule. Es steuert, ob Better Fullstack während der Projekterstellung mit `--web-deploy` und `--server-deploy` Plattformdateien ausgibt.

Verwenden Sie Bereitstellungsflags zur Gerüstzeit, wenn Sie möchten, dass generierte Plattformdateien mit dem ausgewählten Frontend-, Backend-, Laufzeit- und Arbeitsbereichslayout übereinstimmen.

## Bereitstellungsflags

| Flagge | Werte |
| --- | --- |
| `--web-deploy` | `vercel` `cloudflare` `fly` `railway` `render` `netlify` `docker` `sst` `none` |
| `--server-deploy` | `vercel` `cloudflare` `fly` `railway` `render` `docker` `sst` `none` |

Die Bereitstellungsoptionen werden auf Kompatibilität mit dem Stack überprüft. Einige Fullstack-Frameworks werden über die Web-App bereitgestellt, einige Backends benötigen keine separate Serverbereitstellung und Convex oder Encore verwalten ihren eigenen Plattformpfad.

## Ziele

| Ziel | Beste Passform | Notizen |
| --- | --- | --- |
| Docker | Selbsthosting, reproduzierbare Produktionsbilder und lokale Service-Orchestrierung. | Getrennt vom Add-on `docker-compose`, das sich auf lokale Dienste konzentriert. |
| Vercel | Unterstützte React-, Nuxt-, SvelteKit-, Solid- und serverlose Bereitstellungen. | Fullstack-Frontends werden normalerweise über die Web-App und nicht über ein separates Serverziel bereitgestellt. |
| Wolkenflare | Worker-Laufzeitstapel und Edge-First-Hono-Server. | Die Kompatibilität ist auf Workers-kompatible Serverlaufzeiten beschränkt. |
| Fly.io | Docker-gestütztes App-Hosting mit regionaler Platzierung und langlebigen Diensten. | Gut für persistente Serverprozesse. |
| Eisenbahn | Mit Git verbundenes App-Hosting mit Dockerfile-basierten Builds und verwalteten Diensten. | Konfigurieren Sie Produktionsumgebungsvariablen vor der Bereitstellung. |
| Machen | Blueprint-gesteuertes App-Hosting mit Dockerfile-basierten Builds (`render.yaml` im Repo-Stammverzeichnis). | Web und Server werden als separate Docker-Dienste bereitgestellt; Legen Sie Umgebungsvariablen im Render-Dashboard fest. |
| Netlify | Statische und SSR-Web-Frontends (Next.js, Vite SPAs, SvelteKit, Nuxt). | Nur-Web-Ziel; Legen Sie das Site-Basisverzeichnis auf `apps/web` fest. |
| SST | AWS-orientierte Infrastruktur als Code für TypeScript-App-Workflows. | Am besten, wenn Sie AWS-Ressourcen explizit besitzen möchten. |

## Vor der Bereitstellung

- Legen Sie Produktionsumgebungsvariablen für Authentifizierungsgeheimnisse, Datenbank-URLs, Anbieterschlüssel und öffentliche App-URLs fest.
- Führen Sie Datenbankmigrationen für die Produktionsdatenbank durch, bevor Sie den Datenverkehr weiterleiten.
- Bestätigen Sie, dass die generierte Plattformkonfiguration mit der App übereinstimmt, die Anfragen empfangen soll.
- Halten Sie lokale Dienstcontainer, Testanmeldeinformationen und generierte Protokolle von der Quellcodeverwaltung fern.

## Nach dem Gerüst hinzufügen

Der Befehl `add` kann Bereitstellungsflags aufzeichnen, verarbeitet jedoch derzeit Add-Ons zuerst. Wenn kein neues Add-on ausgewählt ist, werden `add`-Befehle, die nur für die Bereitstellung bestimmt sind, vorzeitig beendet. Bevorzugen Sie die Auswahl von Bereitstellungszielen während `create`, bis reine Bereitstellungs-Add-Flows erweitert werden.
