---
title: 導入
description: Web およびサーバーのデプロイメント ターゲットを 1 つのスタック セクションとして選択します。
---

デプロイは 1 つのスタック セクションであり、個別のドキュメントの柱ではありません。 `--web-deploy` および `--server-deploy` を使用したプロジェクトの作成中に、Better Fullstack がプラットフォーム ファイルを発行するかどうかを制御します。

生成されたプラットフォーム ファイルを、選択したフロントエンド、バックエンド、ランタイム、およびワークスペースのレイアウトと一致させる必要がある場合は、スキャフォールド時にデプロイメント フラグを使用します。

## 展開フラグ

| フラグ | 価値観 |
| --- | --- |
| `--web-deploy` | `vercel` `cloudflare` `fly` `railway` `render` `netlify` `docker` `sst` `none` |
| `--server-deploy` | `vercel` `cloudflare` `fly` `railway` `render` `docker` `sst` `none` |

デプロイメントの選択は、スタックに対して互換性がチェックされます。一部のフルスタック フレームワークは Web アプリを通じてデプロイされ、一部のバックエンドは別個のサーバー デプロイメントを必要とせず、Convex または Encore は独自のプラットフォーム パスを管理します。

## ターゲット

| ターゲット | ベストフィット | 注意事項 |
| --- | --- | --- |
| ドッカー | 自己ホスティング、再現可能な本番イメージ、およびローカル サービス オーケストレーション。 | ローカル サービスに重点を置いた `docker-compose` アドオンとは別のものです。 |
| ヴェルセル | React、Nuxt、SvelteKit、Solid、およびサーバーレス デプロイメントがサポートされています。 | フルスタック フロントエンドは通常、別のサーバー ターゲットではなく、Web アプリを通じてデプロイされます。 |
| クラウドフレア | ワーカー ランタイム スタックとエッジファースト Hono サーバー。 | 互換性は、Workers 互換サーバー ランタイムに限定されます。 |
| Fly.io | 地域的な配置と長期実行サービスを備えた Docker ベースのアプリ ホスティング。 | 永続的なサーバープロセスに適しています。 |
| 鉄道 | Dockerfile ベースのビルドとマネージド サービスを使用した Git 接続アプリのホスティング。 | デプロイ前に実稼働環境変数を構成します。 |
| 与える | Dockerfile ベースのビルド (リポジトリ ルートの `render.yaml`) を使用したブループリント主導のアプリのホスティング。 | Web とサーバーは別個の Docker サービスとしてデプロイされます。レンダリング ダッシュボードで環境変数を設定します。 |
| ネットリファイ | 静的および SSR Web フロントエンド (Next.js、Vite SPA、SvelteKit、Nuxt)。 | Web のみのターゲット。サイトのベース ディレクトリを `apps/web` に設定します。 |
| SST | TypeScript アプリのワークフロー用のコードとしての AWS 指向のインフラストラクチャ。 | AWS リソースを明示的に所有したい場合に最適です。 |

## 導入前に

- 認証シークレット、データベース URL、プロバイダー キー、およびパブリック アプリ URL の運用環境変数を設定します。
- トラフィックをルーティングする前に、運用データベースに対してデータベース移行を実行します。
- 生成されたプラットフォーム構成がリクエストを受信するアプリと一致していることを確認します。
- ローカル専用のサービス コンテナー、テスト資格情報、生成されたログをソース管理の対象外に保ちます。

## 足場の後に追加

`add` コマンドは展開フラグを記録できますが、現在はアドオンが最初に処理されます。新しいアドオンが選択されていない場合、展開専用の `add` コマンドは早期に終了します。デプロイメントのみの追加フローが拡張されるまでは、`create` 中にデプロイメント ターゲットを選択することを優先します。
