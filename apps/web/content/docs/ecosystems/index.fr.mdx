title: Écosystèmes

---

Better Fullstack utilise un CLI pour plusieurs écosystèmes. Chaque écosystème possède sa propre surface d'échafaudage, sa propre chaîne d'outils et ses propres règles de compatibilité.

Pour les piles de produits couvrant plusieurs langues, utilisez le [mode multi-écosystème] (/docs/ecosystems/multi-ecosystem/). Il modélise le projet sous forme d'éléments de pile, tels que `frontend:typescript:next`, `backend:go:gin`, `backend.orm:go:gorm` et `database:universal:postgres`.

| Écosystème | Idéal pour | Remarques |
| --- | --- | --- |
| [TypeScript](/docs/ecosystems/typescript/) | Web Fullstack, APIs, Workers, bureau et la plus grande matrice d'intégration Web. | Les options Web sont TypeScript en premier. |
| [React Native](/docs/ecosystems/react-native/) | Expo applications mobiles avec navigation, interface utilisateur, stockage, tests, push, OTA et liens profonds. | Utilise la surface de l'écosystème React Native au lieu des catégories d'interface Web TypeScript. |
| [Rouille](/docs/ecosystems/rust/) | Services backend, applications CLI, GraphQL/gRPC, interfaces Web Rust et bibliothèques fortement typées. | Utilise du fret ; les modèles sont conditionnels aux options Rust sélectionnées. |
| [Python](/docs/ecosystems/python/) | Services orientés API et AI avec framework, ORM, validation, file d'attente, GraphQL et choix de qualité. | Utilise `uv` ; Python AI est une catégorie distincte à sélection multiple. |
| [Aller](/docs/ecosystems/go/) | Services API, gRPC, CLIs, journalisation, ORM et assistants d'authentification Go. | Utilise les modules Go ; GoBetterAuth est sélectionné avec global `--auth go-better-auth`. |
| [Java](/docs/ecosystems/java/) | Projets Spring Boot, Quarkus ou Java simple avec Maven/Gradle, persistance, authentification, bibliothèques et tests. | Java utilise les wrappers Maven ou Gradle lorsqu'un outil de génération est sélectionné. |
| [Élixir](/docs/ecosystems/elixir/) | Applications Phoenix et Phoenix LiveView avec Ecto, Absinthe, temps réel, tâches, tests, qualité et choix de déploiement. | Utilise le mélange ; quelques options au niveau du schéma restent désactivées jusqu'à ce que les modèles rattrapent leur retard. |
| [.NET](/docs/ecosystems/dotnet/) | Services ASP.NET Core avec EF Core ou Dapper, Identity, Minimal APIs ou Hot Chocolate, SignalR, tâches et tests. | Utilise le SDK .NET ; les tests et l'observabilité sont des catégories à sélection multiple. |

## Ligne de base CLI par défaut

L'acceptation des valeurs par défaut crée un projet Web TypeScript. Le gestionnaire de packages suit le gestionnaire de packages utilisé pour lancer le CLI.

```json
{
  "ecosystem": "typescript",
  "frontend": ["tanstack-router"],
  "backend": "hono",
  "runtime": "bun",
  "database": "sqlite",
  "orm": "drizzle",
  "auth": "better-auth",
  "api": "trpc",
  "cssFramework": "tailwind",
  "uiLibrary": "shadcn-ui",
  "testing": "vitest",
  "addons": ["turborepo"],
  "versionChannel": "stable",
  "aiDocs": ["claude-md"]
}
```

Le runtime et le gestionnaire de packages sont des choix distincts. Par exemple, un projet TypeScript généré peut utiliser npm, pnpm, Yarn ou Bun pour les installations tout en sélectionnant Bun comme environnement d'exécution de l'application.

## Les options partagées ne sont pas universelles

Certains indicateurs sont globaux car ils sont stockés dans une configuration de projet, mais tous les indicateurs ne s'appliquent pas à chaque écosystème. Utilisez les pages de l'écosystème ci-dessous pour ce qui est généré et utilisez [Options de référence](/docs/reference/options/) pour les valeurs exactes CLI.
