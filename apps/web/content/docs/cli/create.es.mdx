---
title: Comando Create
description: Genera un proyecto Better Fullstack nuevo con prompts, defaults o flags explicitos.
---

El comando `create` escribe un proyecto nuevo desde una configuración de stack tipada. Úsalo de forma interactiva para explorar, o pasa flags cuando necesites la misma forma de proyecto cada vez.

<PMTabs
  npm="npm create better-fullstack@latest [project-name] -- [flags]"
  pnpm="pnpm create better-fullstack@latest [project-name] [flags]"
  bun="bun create better-fullstack@latest [project-name] [flags]"
  yarn="yarn create better-fullstack@latest [project-name] [flags]"
/>

Con npm, los flags de Better Fullstack van después de `--`. Los comandos npm generados pueden usar `npx create-better-fullstack@latest`; eso ejecuta el mismo entrypoint del paquete en forma no interactiva.

## Modos del comando

| Modo | Forma del comando | Úsalo cuando |
| --- | --- | --- |
| Wizard | `[launcher] my-app` | Quieres prompts y defaults. |
| Defaults | `[launcher] my-app [flags] --yes` | Quieres el stack por defecto sin prompts. |
| Scripted | `[launcher] my-app [flags]` | Quieres output repetible. |
| Dry run | `[launcher] my-app [flags] --dry-run` | Quieres inspeccionar archivos antes de escribir. |

## Comportamiento de prompts

- Los flags explicitos saltan solo las categorías que cubren.
- `--yes` acepta defaults y entra en conflicto con flags centrales del stack como `--frontend`, `--backend`, `--database`, `--orm`, `--auth` y `--api`.
- Usa `none` para desactivar categorías opcionales, incluido `--ai-docs none`.
- Usa `--no-install` y `--no-git` cuando quieras inspeccionar archivos generados antes de efectos secundarios.
- `--verify` ejecuta checks generados después del scaffold sin iniciar dev servers.

## Ejemplos

Crea una app TypeScript con flags explicitos:

```npm
npm create better-fullstack@latest my-app -- \
  --ecosystem typescript \
  --frontend tanstack-router \
  --backend hono \
  --runtime node \
  --database postgres \
  --orm drizzle \
  --db-setup none \
  --api orpc \
  --auth better-auth \
  --css-framework tailwind \
  --ui-library shadcn-ui \
  --ai none \
  --payments none \
  --email none \
  --examples none \
  --addons none \
  --ai-docs none \
  --version-channel stable \
  --no-install \
  --no-git
```

Crea un proyecto multi-ecosystem:

```npm
npm create better-fullstack@latest my-mixed-app -- \
  --part frontend:typescript:next \
  --part backend:go:gin \
  --part backend.orm:go:gorm \
  --part backend.auth:go:jwt \
  --part database:universal:postgres \
  --part mobile:react-native:native-bare \
  --addons turborepo docker-compose
```

## Compatibilidad

- La CLI valida compatibilidad de stack antes de generar.
- TypeScript `backend self` es para frontends fullstack y se combina con `runtime none`.
- tRPC está orientado a React.
- El soporte de Workers runtime depende de la combinacion frontend/backend.
- UI libraries, targets de deployment, providers y addons dependen del stack.
- Para una comprobacion visual, usa la [Compatibility Matrix](/docs/reference/compatibility/) o el [Stack Builder](https://better-fullstack.dev/new).
