---
title: Befehl erstellen
description: Erstellen Sie ein Gerüst für ein neues Better Fullstack-Projekt mit Eingabeaufforderungen, Standardwerten oder expliziten Flags.
---

Der Befehl `create` schreibt ein neues Projekt aus einer typisierten Stack-Konfiguration. Nutzen Sie es interaktiv zur Erkundung oder übergeben Sie Flags, wenn Sie jedes Mal die gleiche Projektform benötigen.

<PMTabs
  npm="npm create better-fullstack@latest [project-name] -- [flags]"
  pnpm="pnpm create better-fullstack@latest [project-name] [flags]"
  bun="bun create better-fullstack@latest [project-name] [flags]"
  yarn="yarn create better-fullstack@latest [project-name] [flags]"
/>

Für npm stehen Better Fullstack-Flags hinter `--`. Generierte npm-Befehle können `npx create-better-fullstack@latest` verwenden; das denselben Paket-Einstiegspunkt in einer nicht interaktiven Form ausführt.

## Befehlsmodi

| Modus | Befehlsform | Verwenden Sie es wann |
| --- | --- | --- |
| Zauberer | `[launcher] my-app` | Sie möchten Eingabeaufforderungen und Standardeinstellungen. |
| Standardeinstellungen | `[launcher] my-app [flags] --yes` | Sie möchten den Standardstapel ohne Eingabeaufforderungen. |
| Geskriptet | `[launcher] my-app [flags]` | Sie möchten eine wiederholbare Ausgabe. |
| Trockenlauf | `[launcher] my-app [flags] --dry-run` | Sie möchten Dateien vor dem Schreiben überprüfen. |

## Promptes Verhalten

- Explizite Flags überspringen nur die Kategorien, die sie abdecken.
- `--yes` akzeptiert Standardwerte und Konflikte mit Core-Stack-Flags wie `--frontend`, `--backend`, `--database`, `--orm`, `--auth` und `--api`.
- Verwenden Sie `none`, um optionale Kategorien zu deaktivieren, einschließlich `--ai-docs none`.
- Verwenden Sie `--no-install` und `--no-git`, wenn Sie generierte Dateien vor Nebenwirkungen überprüfen möchten.
- `--verify` führt generierte Prüfungen nach dem Gerüstaufbau aus, ohne Entwicklungsserver zu starten.

## Beispiele

Erstellen Sie eine TypeScript-App mit expliziten Flags:

```npm
npm create better-fullstack@latest my-app -- \
  --ecosystem typescript \
  --frontend tanstack-router \
  --backend hono \
  --runtime node \
  --database postgres \
  --orm drizzle \
  --db-setup none \
  --api orpc \
  --auth better-auth \
  --css-framework tailwind \
  --ui-library shadcn-ui \
  --ai none \
  --payments none \
  --email none \
  --examples none \
  --addons none \
  --ai-docs none \
  --version-channel stable \
  --no-install \
  --no-git
```

Erstellen Sie ein Multi-Ökosystem-Projekt:

```npm
npm create better-fullstack@latest my-mixed-app -- \
  --part frontend:typescript:next \
  --part backend:go:gin \
  --part backend.orm:go:gorm \
  --part backend.auth:go:jwt \
  --part database:universal:postgres \
  --part mobile:react-native:native-bare \
  --addons turborepo docker-compose
```

## Kompatibilität

- Der CLI validiert die Stack-Kompatibilität vor der Generierung.
- TypeScript `backend self` ist für Fullstack-Frontends und Paare mit `runtime none`.
- tRPC ist reaktionsorientiert.
- Die Worker-Laufzeitunterstützung hängt von der Frontend-/Backend-Kombination ab.
- UI-Bibliotheken, Bereitstellungsziele, Anbieter und Add-ons sind stapelsensitiv.
- Für eine visuelle Prüfung verwenden Sie die [Kompatibilitätsmatrix](/docs/reference/compatibility/) oder die [Stack Builder](https://better-fullstack.dev/new).
