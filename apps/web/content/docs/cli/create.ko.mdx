title: 명령 생성

---

`create` 명령은 형식화된 스택 구성에서 새 프로젝트를 작성합니다. 탐색을 위해 대화형으로 사용하거나 매번 동일한 프로젝트 형태가 필요할 때 플래그를 전달하세요.

<PMTabs
  npm="npm create better-fullstack@latest [project-name] -- [flags]"
  pnpm="pnpm create better-fullstack@latest [project-name] [flags]"
  bun="bun create better-fullstack@latest [project-name] [flags]"
  yarn="yarn create better-fullstack@latest [project-name] [flags]"
/>

npm의 경우 Better Fullstack 플래그는 `--` 뒤에 옵니다. 생성된 npm 명령은 `npx create-better-fullstack@latest`을 사용할 수 있습니다. 비대화형 형식으로 동일한 패키지 진입점을 실행합니다.

## 명령 모드

| 방법 | 명령 모양 | 다음과 같은 경우에 사용하세요. |
| --- | --- | --- |
| 마법사 | `[launcher] my-app` | 프롬프트와 기본값이 필요합니다. |
| 기본값 | `[launcher] my-app [flags] --yes` | 프롬프트 없이 기본 스택을 원합니다. |
| 스크립트 | `[launcher] my-app [flags]` | 반복 가능한 출력을 원합니다. |
| 드라이런 | `[launcher] my-app [flags] --dry-run` | 쓰기 전에 파일을 검사하고 싶습니다. |

## 신속한 행동

- 명시적 플래그는 해당하는 범주만 건너뜁니다.
- `--yes`은 기본값을 허용하며 `--frontend`, `--backend`, `--database`, `--orm`, `--auth` 및 `--api`과 같은 코어 스택 플래그와의 충돌을 허용합니다.
- `none`을 사용하여 `--ai-docs none`을 포함한 선택적 범주를 비활성화합니다.
- 부작용이 발생하기 전에 생성된 파일을 검사하려면 `--no-install` 및 `--no-git`을 사용하세요.
- `--verify`은 개발 서버를 시작하지 않고 스캐폴딩 후에 생성된 검사를 실행합니다.

## 예

명시적 플래그를 사용하여 TypeScript 앱을 만듭니다.

```npm
npm create better-fullstack@latest my-app -- \
  --ecosystem typescript \
  --frontend tanstack-router \
  --backend hono \
  --runtime node \
  --database postgres \
  --orm drizzle \
  --db-setup none \
  --api orpc \
  --auth better-auth \
  --css-framework tailwind \
  --ui-library shadcn-ui \
  --ai none \
  --payments none \
  --email none \
  --examples none \
  --addons none \
  --ai-docs none \
  --version-channel stable \
  --no-install \
  --no-git
```

다중 생태계 프로젝트를 생성합니다:

```npm
npm create better-fullstack@latest my-mixed-app -- \
  --part frontend:typescript:next \
  --part backend:go:gin \
  --part backend.orm:go:gorm \
  --part backend.auth:go:jwt \
  --part database:universal:postgres \
  --part mobile:react-native:native-bare \
  --addons turborepo docker-compose
```

## 호환성

- CLI은 생성 전에 스택 호환성을 검증합니다.
- TypeScript `backend self`은 풀스택 프런트엔드용이며 `runtime none`과 쌍을 이룹니다.
- tRPC은 반응 지향적입니다.
- 작업자 런타임 지원은 프런트엔드/백엔드 조합에 따라 다릅니다.
- UI 라이브러리, 배포 대상, 공급자 및 애드온은 스택에 민감합니다.
- 시각적으로 확인하려면 [호환성 매트릭스](/docs/reference/compatibility/) 또는 [Stack Builder](https://better-fullstack.dev/new)를 사용하세요.
