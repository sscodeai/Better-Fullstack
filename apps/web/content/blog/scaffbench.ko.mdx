코딩 에이전트는 코드 작성에는 매우 능숙하지만 프로젝트 *시작*에는 놀라울 만큼 좋지 않습니다. 하나에게 물어보세요
프로덕션급 풀스택 모노레포를 기반으로 하면 손으로 직접 쓰는 데 10분 정도가 걸릴 것입니다.
매니페스트, 잠금 파일 및 구성 — 결과가 빌드되지 않는 경우가 많습니다.

ScaffBench는 빈 작업공간에 있는 에이전트, 프로젝트 사양 및 하드를 정확하게 측정합니다.
마지막 질문 — **생성된 프로젝트가 설치 및 빌드됩니까?** 동일한 사양을 실행했습니다.
세 가지 생성 경로를 통해 도구가 얼마나 도움이 되는지 분리합니다.

- **MCP** — 에이전트는 Better-Fullstack MCP 서버를 사용하여 프로젝트를 계획하고 비계합니다.
- **BF 언급** — MCP 없음; 프롬프트는 Better-Fullstack CLI 및 문서의 에이전트를 가리키며
`create` 명령 자체를 구성합니다.
- **프롬프트** — 더 나은 풀스택이 전혀 없습니다. 에이전트는 모든 파일을 처음부터 직접 작성합니다.

두 개의 서로 다른 에이전트 CLI에서 전체 제품군을 두 번 실행했습니다. **OpenAI Codex CLI** 세 개의 GPT
모델(6월 10일) 및 3개의 Claude 모델을 사용한 **Claude Code**(6월 12일) — 총 72회 실행. 같은
사양, 동일한 프롬프트, 동일한 검증. 그런 다음 의도적으로 [가벼운 세 번째 스윕](#the-light-sweep-gemini-kilo-and-opencode)을 추가했습니다.
— 가장 가벼운 사양으로만 **Gemini CLI**, **Kilo** 및 **오픈코드**에 걸쳐 9개의 추가 모델 —
총 102개의 실행과 15개의 모델을 가져옵니다.

## 헤드라인 결과

헤드라인 번호는 Claude Code 스윕에서 나옵니다 — 36회 실행: 3개 모델(Fable 5, Opus 4.8,
Sonnet 4.6) × 3개의 생성 경로 × 4개의 프로젝트 사양. 동일한 기계, 동일한 하네스, 동일한 프롬프트
생성 모드 지침을 제외하고. [GPT 스윕](#gpt-models-on-codex-cli)은 다음을 보여줍니다.
동일한 구조이며 아래에 별도로 보고됩니다.

| 생성 경로 | 평균 시간 | 중앙 시간 | 평균 출력 토큰 | 빌드 통과 |
| --- | ---: | ---: | ---: | ---: |
| MCP | 113.4초 | 85.9초 | 5,553 | 12/12 (100%) |
| BF 언급 | 219.6초 | 172.1초 | 11,060 | 9/9 (100%)* |
| 즉각적인 | 516.2초 | 424.0초 | 25,859 | 9/12 (75%) |

\* 3개의 BF 언급 실행은 템플릿에서 실패했기 때문에 분모에서 제외됩니다.
에이전트가 수행한 작업이 아닌 우리 측의 생성기 버그입니다. [점수 정책](#what-counts-as-a-failure)을 참조하세요.

세 가지가 눈에 띄었습니다.

1. **MCP은(는) 프롬프트 전용보다 ~4.6배 빠르며** 평균적으로(중앙값은 ~4.9배), ~4.7배 더 적습니다.
토큰을 출력합니다. 에이전트는 코드가 아닌 구성을 제공합니다.
2. **모든 MCP 실행이 모든 모델에 대해 검증을 통과했습니다**. 프롬프트 전용 실행이 75% 통과 - 2개의 실행 적중
15분의 시간 초과와 하나는 빌드되지 않은 모노레포를 생성했습니다.
3. **툴링은 모델 차이를 압축합니다.** MCP 경로에서는 가장 작은 모델이라도
100% 합격률의 프론티어 모델 — 더 빠르고 저렴하게 출시되었습니다.

## 방법론

### 세 가지 창조 경로

모든 실행은 동일한 기본 프롬프트가 있는 빈 디렉터리에서 시작됩니다.

```text
You are running in an empty benchmark workspace:
{run_dir}

Create exactly one project directory named `{project_name}`.
Do not ask questions. Do not start a dev server. Do not write outside the
current working directory.
At the end, report the commands you ran and any errors you hit.

Benchmark target: {spec_title}
Requirements:
{spec_requirements}
```

경로 간의 유일한 차이점은 해당 프롬프트에 추가된 생성 모드 지침입니다.
프롬프트 전용 실행은 우리의 어떤 것도 건드리는 것이 명시적으로 금지되어 있습니다.

```text
Creation mode: prompt-only.
Do not use the Better-Fullstack MCP server, Better-Fullstack CLI,
Better-Fullstack website, or files from the Better-Fullstack repository.
Create the project from scratch by writing the files and manifests needed
for a runnable starter.
```

MCP 실행은 반대 결과를 얻습니다.

```text
Creation mode: Better-Fullstack MCP.
Use the Better-Fullstack MCP tools, starting with bfs_get_guidance. Then use
schema/compatibility/plan as needed and call bfs_create_project to create
the project.
```

그리고 BF 언급 실행은 MCP 도구는 없지만 프롬프트 이름은 CLI과 README 사이에 있습니다.
에이전트는 비대화형 `create` 명령 자체를 작성하기 전에 읽을 수 있습니다.

### 네 가지 프로젝트 사양

사양은 "주말 프로젝트"에서 "실제 팀이 주장하는 스택"으로 의도적으로 확산됩니다.
약 일주일 동안":

| 투기 | 무엇을 요구하는가 |
| --- | --- |
| `light-ts` | Bun, tRPC, SQLite + Drizzle의 React + Vite, Hono, Tailwind + DaisyUI, Pino, Vitest, Biome — 인증 없음, 결제 없음 |
| `heavy-ts` | Next.js **및** Expo/React Native 앱, Hono on Bun, oRPC, PostgreSQL + Drizzle, Better Auth, Stripe, Resend, UploadThing, Effect, TanStack Store/Form/Query, Valibot, Vitest + Playwright, Vercel AI SDK, Socket.IO, Inngest, Framer Motion, OpenTelemetry, PostHog, Umami, Sanity, Upstash Redis, Algolia, S3, shadcn/ui, Turborepo, MSW, Storybook, PWA, Docker compose |
| `python-ai` | FastAPI, SQLModel, PostgreSQL, Pydantic, JWT 인증, Celery, Strawberry GraphQL, LangChain + OpenAI SDK + LangGraph + CrewAI, Ruff |
| `multi-ecosystem` | TypeScript Next.js 프런트엔드, Python FastAPI 백엔드(SQLModel/Pydantic/LangChain/Celery), Go Gin 서비스(GORM, gRPC, Cobra, Zap), 공유 PostgreSQL |

`heavy-ts`은 스트레스 테스트입니다. 또한 이 벤치마크에서 모든 흥미로운 실패가 발생하는 곳이기도 합니다.
일어났다.

### 하네스

두 번의 스윕, 하나의 하네스 디자인:

- **Claude Sweep:** 헤드리스 모드(`claude -p`)에서 Claude Code, JSON 이벤트 스트림이 캡처됨
달리다. 모델: `claude-fable-5`, `claude-opus-4-8`, `claude-sonnet-4-6`, 기본 추론 노력.
- **GPT 스윕:** OpenAI Codex CLI(`codex exec`), 실행별로 캡처된 JSON 이벤트 스트림. 모델:
`gpt-5.3-codex-spark`(높은 노력), `gpt-5.4`(중간), `gpt-5.5`(중간).
- **격리:** 실행될 때마다 새로운 빈 작업 공간이 확보됩니다. MCP 실행은 다음과 같은 엄격한 MCP 구성을 가져옵니다.
Better-Fullstack 서버가 연결되었습니다.
- **시간 초과:** 실행당 900초. 시간 초과는 에이전트의 실패로 간주됩니다.
- **측정항목:** 벽시계 시간, 에이전트 자체 사용 보고서의 토큰 사용량, MCP 도구 호출
이벤트 스트림. 달러 비용은 Claude 스윕에서만 포착됩니다. Codex 하네스는 그렇지 않습니다.
측정된 비용을 보고합니다.

한 가지 중요한 비대칭성: GPT 스윕은 이틀 전에 생성기에 대해 실행되었습니다.
노출된 템플릿 버그를 수정했습니다. 이것이 바로 아래 생성기 버그 제외가 GPT에 적용되는 이유입니다.
결과가 더 어려워지고 두 번의 스윕을 합격률에 대해 정면으로 비교해서는 안되는 이유.

### 합격으로 간주되는 것

에이전트가 완료된 *후* 디스크에 남아 있는 모든 항목에 대해 유효성 검사가 실행됩니다.

- **TypeScript:** `bun install`, 프로젝트 자체 `build`(또는 `check`/`lint`/`test`) 스크립트.
- **Python:** virtualenv 및 공급업체 패키지를 제외한 프로젝트 전반에 걸쳐 `compileall`.
- **이동:** `go mod download` 모듈을 컴파일해야 합니다.

적용 가능한 모든 검사가 0을 종료하는 경우에만 실행이 통과됩니다. "프로젝트처럼 보입니다"는 중요하지 않습니다.
구축해야합니다.

### 실패로 간주되는 것

<Callout kind="info">
**점수 정책:** 자체 템플릿 생성기의 버그로 인해 발생한 실패는 제외됩니다.
합격률 분모 - 에이전트가 작성한 실패만 에이전트에 포함됩니다. 벤치마크의
임무는 에이전트를 측정하는 것이지 버그를 점수에 세탁하는 것이 아닙니다. 반대 측면: 우리는 출판합니다
아래의 생성기 버그를 숨기는 대신에 알려주세요.
</Callout>

Claude 스윕에서 이는 정확히 세 번 중요했습니다. 세 모델 모두 BF 언급 경로에서,
`heavy-ts`을 CLI을 통해 올바르게 스캐폴드했습니다. 그리고 세 개 모두 우리에서 동일한 생성기 버그에 부딪혔습니다.
`app.json`이 `expo-router`에서만 작동하는 웹 출력 모드를 설정하는 기본(Expo) 템플릿입니다.
에이전트는 모든 일을 올바르게 수행했습니다. 템플릿을 빌드할 수 없습니다. 그 세 번의 실행은 제외됩니다.
프롬프트 경로 실패(2번의 시간 초과, 1번의 손으로 작성한 빌드 중단)는 전적으로 에이전트가 작성하고
전체를 계산합니다.

접두사 생성기에 대해 실행된 GPT 스윕에서는 7번 중요했습니다. `heavy-ts`
MCP 및 CLI를 통해 이후 수정된 Storybook/Expo 버그로 인해 세 가지 모델 모두 실패했습니다.
체인과 생성된 `go.sum` 없이 Go 서비스가 제공되는 다중 생태계 스캐폴드 1개.
7개 모두 같은 이유로 제외됩니다. GPT 상담원의 자체 실패(3개가 손상됨)
프롬프트 경로에 직접 작성한 빌드)는 전체로 계산됩니다.

GPT 스윕이 상황에 추가하는 것:

- **구조는 공급업체에 독립적입니다.** 모든 GPT 모델에서 MCP은 가장 빠른 경로이며
프롬프트 전용은 출력 토큰에서 가장 비쌉니다. 클로드 결과와 같은 모양입니다.
다른 공급업체 모델 *및* 다른 에이전트 CLI.
- **Spark는 속도를 위해 만들어졌음을 보여줍니다.** GPT-5.3 Codex Spark는 MCP당 평균 32초입니다.

`heavy-ts`. GPT 스윕이 가장 먼저 타격을 입혔습니다. 9개의 `heavy-ts` 빌드 중 8개는 실패했고, 6개는 실패했습니다.
