Les agents de codage sont très bons pour écrire du code et étonnamment mauvais pour *démarrer* des projets. Demandez à quelqu'un de
échafaudez un monorepo fullstack de qualité production et il passera volontiers dix minutes à écrire à la main
manifestes, fichiers de verrouillage et configuration – et le résultat ne se construit souvent pas.

ScaffBench mesure exactement cela : un agent dans un espace de travail vide, une spécification de projet et un disque dur.
question à la fin — **le projet généré s'installe-t-il et se construit-il ?** Nous avons exécuté les mêmes spécifications
à travers trois chemins de création pour isoler dans quelle mesure les outils sont utiles :

- **MCP** — l'agent utilise le serveur Better-Fullstack MCP pour planifier et échafauder le projet.
- **Mention BF** — non MCP ; l'invite pointe l'agent vers Better-Fullstack CLI et la documentation, et il
compose la commande `create` elle-même.
- **Invite** — pas de Better-Fullstack du tout ; l'agent écrit manuellement chaque fichier à partir de zéro.

Nous avons exécuté la suite complète deux fois, sur deux agents CLI différents : **OpenAI Codex CLI** avec trois GPT
modèles (10 juin) et **Claude Code** avec trois modèles Claude (12 juin) — 72 exécutions au total. Même
spécifications, mêmes invites, même validation. Nous avons ensuite ajouté délibérément un [troisième balayage léger](#the-light-sweep-gemini-kilo-and-opencode)
— neuf modèles supplémentaires sur **Gemini CLI**, **Kilo** et **opencode**, sur les spécifications les plus légères uniquement —
portant le total à 102 courses et quinze modèles.

## Résultats principaux

Les numéros de titre proviennent du balayage Claude Code — 36 runs : trois modèles (Fable 5, Opus 4.8,
Sonnet 4.6) × trois chemins de création × quatre spécifications de projet. Même machine, même harnais, mêmes invites
en dehors des instructions du mode création. Le [balayage GPT](#gpt-models-on-codex-cli) affiche le
même structure et est présenté séparément ci-dessous.

| Chemin de création | Temps moyen | Temps médian | Jetons de sortie moyens | Construit en passant |
| --- | ---: | ---: | ---: | ---: |
| MCP | 113,4 s | 85,9 s | 5 553 | 12/12 (100%) |
| mention petit ami | 219,6 s | 172,1 s | 11 060 | 9/9 (100%)* |
| Rapide | 516,2 s | 424,0s | 25 859 | 9/12 (75%) |

\* Trois exécutions avec mention BF sont exclues du dénominateur car elles ont échoué sur un modèle
bug du générateur de notre côté, pas sur ce que l'agent a fait - voir [la politique de notation] (#what-counts-as-a-failure).

Trois choses sont ressorties :

1. **MCP est ~4,6× plus rapide que l'invite uniquement** en moyenne (et ~4,9× sur les médianes), avec ~4,7× de moins
jetons de sortie. L'agent fournit la configuration, pas le code.
2. **Chaque exécution de MCP a réussi la validation**, pour chaque modèle. Les exécutions avec invite uniquement ont dépassé 75 % – deux exécutions ont été réalisées
le délai d'attente de 15 minutes et l'un d'entre eux ont produit un monorepo qui n'a pas été construit.
3. **Les outils compressent les différences de modèle.** Sur le chemin MCP, même le plus petit modèle correspondait au
modèles frontières à un taux de réussite de 100 % – ils sont simplement arrivés plus rapidement et moins cher.

## Méthodologie

### Les trois chemins de création

Chaque exécution démarre dans un répertoire vide avec la même invite de base :

```text
You are running in an empty benchmark workspace:
{run_dir}

Create exactly one project directory named `{project_name}`.
Do not ask questions. Do not start a dev server. Do not write outside the
current working directory.
At the end, report the commands you ran and any errors you hit.

Benchmark target: {spec_title}
Requirements:
{spec_requirements}
```

La seule différence entre les chemins réside dans l'instruction de mode de création ajoutée à cette invite.
Il est explicitement interdit aux exécutions d'invites uniquement de toucher à quoi que ce soit de nous :

```text
Creation mode: prompt-only.
Do not use the Better-Fullstack MCP server, Better-Fullstack CLI,
Better-Fullstack website, or files from the Better-Fullstack repository.
Create the project from scratch by writing the files and manifests needed
for a runnable starter.
```

MCP exécute le contraire :

```text
Creation mode: Better-Fullstack MCP.
Use the Better-Fullstack MCP tools, starting with bfs_get_guidance. Then use
schema/compatibility/plan as needed and call bfs_create_project to create
the project.
```

Et les exécutions avec mention BF se situent entre les deux - pas d'outils MCP, mais l'invite nomme le CLI et un README le
L'agent peut lire avant de composer lui-même une commande `create` non interactive.

Les spécifications sont délibérément réparties du "projet du week-end" à "la pile", dirait une vraie équipe.
environ pendant une semaine » :

### Le harnais

Deux balayages, une conception de harnais :

- **Claude Sweep :** Claude Code en mode sans tête (`claude -p`), flux d'événements JSON capturé par
courir. Modèles : `claude-fable-5`, `claude-opus-4-8`, `claude-sonnet-4-6`, effort de raisonnement par défaut.
- ** Balayage GPT :** OpenAI Codex CLI (`codex exec`), flux d'événements JSON capturé par exécution. Modèles :
`gpt-5.3-codex-spark` (effort élevé), `gpt-5.4` (moyen), `gpt-5.5` (moyen).
- **Isolement :** chaque exécution obtient un nouvel espace de travail vide ; MCP exécute une configuration MCP stricte avec seulement
le serveur Better-Fullstack connecté.
- **Délai d'expiration :** 900 secondes par exécution. Un délai d'attente compte comme un échec pour l'agent.
- **Mesures :** temps d'horloge, utilisation des jetons à partir du propre rapport d'utilisation de l'agent, appels d'outils MCP depuis
le flux d'événements. Le coût en dollars est capturé uniquement sur le balayage Claude - notre harnais Codex ne le fait pas
signaler le coût mesuré.

Une asymétrie importante : le balayage GPT a eu lieu deux jours plus tôt, contre le générateur *avant* nous
correction des bugs du modèle qu'il exposait. C'est pourquoi les exclusions de bogues du générateur ci-dessous touchent le GPT.
les résultats sont plus difficiles – et pourquoi les deux balayages ne devraient pas être comparés face à face sur les taux de réussite.

### Ce qui compte comme un passage

La validation s'exécute *après* la fin de l'agent, par rapport à ce qu'il a laissé sur le disque :

- **TypeScript :** `bun install`, puis le propre script `build` (ou `check`/`lint`/`test`) du projet.
- **Python :** `compileall` dans l'ensemble du projet, à l'exclusion des virtualenvs et des packages vendus.
- **Allez :** `go mod download` et le module doit compiler.

Une exécution ne réussit que si chaque contrôle applicable sort de zéro. « Ça ressemble à un projet » ne compte pas ;
il faut qu'il se construise.

### Ce qui compte comme un échec

<Callout kind="info">
**Politique de notation :** les échecs causés par des bugs dans notre propre générateur de modèles sont exclus de la
dénominateur du taux de réussite : seuls les échecs créés par l'agent sont pris en compte pour l'agent. Le benchmark
Le travail consiste à mesurer les agents, pas à blanchir nos bugs dans leur score. Le revers de la médaille : nous publions
ces bugs du générateur ci-dessous au lieu de les cacher.
</Callout>

Lors du balayage Claude, cela comptait exactement trois fois : les trois modèles, sur le chemin de la mention BF,
échafaudé correctement `heavy-ts` via le CLI - et tous les trois ont rencontré le même bug de générateur dans notre
modèle natif (Expo) où `app.json` définit un mode de sortie Web qui ne fonctionne qu'avec `expo-router`.
Les agents ont tout fait correctement ; le modèle n'a pas pu être construit. Ces trois courses sont exclues.
Les échecs du chemin d'invite (deux délais d'attente, une version écrite à la main cassée) sont entièrement créés par l'agent et
compter au complet.

Lors du balayage GPT – qui s'est déroulé contre le générateur de préfixes – cela comptait sept fois : `heavy-ts`
via MCP et via CLI ont échoué pour les trois modèles sur le bug Storybook/Expo corrigé depuis
chaîne, plus un échafaudage multi-écosystèmes dont le service Go a été livré sans `go.sum` généré.
Tous les sept sont exclus pour la même raison ; les propres échecs des agents GPT (trois échecs
les versions manuscrites sur le chemin de l'invite) comptent dans leur intégralité.

différentes versions de générateur. Comparer les chemins au sein d’un balayage est l’expérience ; comparer les fournisseurs
à travers les balayages ne l'est pas.

une version volontairement allégée : **une spécification (`light-ts`) × trois chemins de création × une exécution par cellule**.
Neuf modèles répartis sur trois autres agents CLI, tous le 12 juin :

- **Gemini CLI** — Gemini 3.1 Pro (la dernière version de Google et la valeur par défaut de CLI).
- **opencode** (abonnement Go) — Kimi K2.6, GLM-5.1, MiniMax M3, Qwen3.7 Max, DeepSeek-V4 Pro :
le niveau phare actuel des poids ouverts.

contre une exécution (les journaux stricts sont dans les artefacts d'exécution). Les Gémeaux ont eu une seconde répétition accidentelle de

Ce que le balayage lumineux ajoute :

- **Les chemins assistés ont été de 18 pour 18.** Chaque modèle a réussi les mentions MCP et BF. Cela comprend
Étape 3.7 Flash - un petit modèle rapide et gratuit - échafaudant un monorepo fullstack correct en 16
secondes et 640 jetons de sortie via le chemin de mention BF. La structure que nous avons vue sur Claude et GPT
n'est pas un phénomène de type frontière ; l'outillage uniformise les règles du jeu jusqu'au bout.
- **L'invite uniquement est l'endroit où le champ est divisé.** Trois des neuf modèles ont expédié des projets défectueux, chacun dans
de manière instructive : Étape 3.7 Flash a épinglé `pino-http` sur une version qui n'existe pas (installer
échoué); MiniMax M3 a nommé une procédure tRPC `useContext`, entrant en collision avec le client intégré de tRPC
méthode (échec de la vérification du type) ; Nex N2-Pro a écrit une importation qui ne se résout pas (échec du rollup).
Aucun de ces modes de défaillance n'existe sur les chemins assistés, car l'agent n'écrit jamais manuellement un
manifeste.
- **Les données économiques s'appuient sur un troisième, un quatrième et un cinquième harnais.** Chaque course assistée s'est terminée sous
3 000 jetons de sortie. Chaque exécution avec invite uniquement coûte entre 8 et 36 000 – 36 000 du MiniMax M3 étant le plus cher
une seule exécution dans ScaffBench jusqu'à présent, pour un projet qui n'a pas été construit.
- **Gemini 3.1 Pro est excellent dans cette tâche.** Cellule de mention BF la plus rapide de l'ensemble du benchmark
(26,7 secondes, 863 jetons), un flux MCP à quatre appels manuel et l'un des rares modèles à réussir
invite uniquement. Son échantillon répété constitue également un contrôle d'honnêteté utile : la première exécution de l'invite a échoué. Biome
charpie, la seconde s'est déroulée proprement - les cellules à passage unique sont directionnelles et non définitives.

Mises en garde concernant le balayage léger : une exécution par cellule et 25 des exécutions exécutées simultanément, donc horloge murale
les temps comportent du bruit de contention (la course de mention BF de Gemini mesurait 27 secondes en solo et 50 secondes en parallèle
lot – même résultat, trafic différent). Traitez la structure du chemin et la réussite/l'échec comme le signal, exact
secondes à titre approximatif.

## Analyse qualitative

### MCP maintient les agents sur les rails

MCP exécutions ont convergé sur la même séquence d'outils : `bfs_get_guidance` → contrôle de compatibilité → plan →
`bfs_create_project`, entre 3 et 10 appels par exécution (la spécification multi-écosystèmes était la plus nécessaire
itérations pour établir une pile valide). La production reste faible car le travail de l'agent se réduit à
*choisir une configuration* — l'écriture du fichier est effectuée par le générateur, de manière déterministe. C'est
aussi pourquoi les taux de réussite ne se dégradent pas à mesure que la spécification augmente : `heavy-ts` à MCP réussis pour chaque
modèle.

### Les agents rapides se noient dans les spécifications lourdes

Sur le chemin de l'invite, `heavy-ts` a vaincu presque tout le monde. Fable 5 et Opus 4.8 étaient encore en train d'écrire
fichiers après un délai d'attente de 15 minutes. Sonnet 4.6 a terminé un monorepo de 90 fichiers en 767 secondes — et il
n'a pas construit. GPT-5.3 Codex Spark a sprinté vers un projet écrit à la main en 68 secondes – ce n'est pas le cas
construire non plus. La seule exception dans les deux balayages était GPT-5.5, qui a été corrigée via `heavy-ts`.
invite seulement en 542 secondes et réussi. Les spécifications les plus légères ont pour la plupart satisfait aux invites uniquement, mais en quelques minutes
d'horloge murale et de 10 000 à 37 000 jetons de sortie chacun. Écrire à la main un démarreur est quelque chose de pionnier
*peut* faire ; c'est simplement le moyen le plus lent, le plus coûteux et le moins fiable d'en obtenir un.

### Le benchmark a audité nos propres modèles

Les échecs les plus utiles furent les nôtres. La validation de chaque projet généré a fait surface **six couches
bogues de modèle** dans notre générateur, la plupart se cachant derrière un seul symptôme « Échec de la construction du livre d'histoires » sur
`heavy-ts`. Le balayage GPT les a frappés en premier : huit de ses neuf builds `heavy-ts` ont échoué, six d'entre elles ont échoué.
les sur cette chaîne de générateurs - et le balayage de Claude deux jours plus tard a confirmé quels correctifs étaient retenus :

1. Les modèles de livre d'histoires ont testé le frontend avec une vérification d'égalité, mais `frontend` est un tableau.
2. Une incompatibilité de chemin de package de base de données a laissé le package de base de données sans ses dépendances en mode partie graphique.
3. `expo-network` manquait dans une variante de modèle natif.
4. Les packages de framework Storybook 8 ne réexportent pas les types `Meta`/`StoryObj` - les importations ont dû être déplacées vers
les packages de rendu.
5. `@better-auth/expo` nécessitait quatre dépendances homologues Expo installées explicitement.
6. Native `app.json` définit un mode de sortie Web statique qui ne fonctionne qu'avec `expo-router` — celui-ci est
toujours ouvert, et c'est le bug derrière les trois exécutions exclues de la mention BF.

Cinq des six ont été corrigés et expédiés dans `create-better-fullstack` 2.0.2 avant la publication de ce message.
dehors. Exécuter votre propre produit via un benchmark d'agent s'avère être un contrôle qualité extrêmement efficace
passer.

### Caractère du modèle

Sonnet 4.6 était le plus rapide et le moins cher sur tous les chemins, et avec ses outils, il n'a rien cédé dans
fiabilité. Fable 5 était la plus délibérée – le plus grand nombre de jetons, les séries les plus longues et, sur le
chemin rapide, cette minutie n'était toujours pas suffisante pour battre le délai d'attente sur `heavy-ts`. Opus 4.8 samedi
fiable entre les deux. Le classement n'a jamais changé d'une voie à l'autre ; l'*écart* l'a fait. L'outillage est le meilleur
égaliseur : sur MCP, l'écart entre le meilleur et le pire modèle était de 102 secondes ; rapide-seulement c'était
minutes variables et, deux fois, une horloge murale au plafond.

## Limites

- **Une exécution par cellule.** 102 exécutions suffisent pour voir la structure, pas pour mettre des intervalles de confiance
dessus. Les temps d'échafaudage varient également en fonction de la charge API - et le lot parallèle du balayage léger ajoute
bruit de contention en haut (la propre répétition de Gemini montrait à la fois un écart de temps de 2 × et un échec/réussite de charpie
retourner entre des courses identiques).
- **Les balayages ne se font pas en face-à-face.** Les exécutions GPT ont utilisé un agent différent CLI (Codex), différent
paramètres d'effort de raisonnement, une version antérieure du générateur et rapport sans coût financier ; la lumière
le balayage couvre une spécification sur trois autres CLI. Les comparaisons au sein du chemin de balayage sont solides ;
Les classements des modèles multi-fournisseurs ne sont pas ce que mesure cette référence.
- **Nous avons comparé notre propre outil.** L'exploitation, les invites et la validation sont publiques dans l'esprit —
les invites et la politique sont citées ci-dessus – et le tableau complet par exécution se trouve en annexe. Traitez le
comparaison entre les chemins (outillage vs pas d'outillage) comme découverte, et non comme affirmation sur d'autres
échafaudages.
- **Builds, pas fonctionnalités.** La validation prouve que le projet s'installe et se construit - pas que l'authentification
le flux fonctionne ou le webhook Stripe est correctement câblé. Un projet de réussite avec invite uniquement peut toujours être
il manque plus de spécifications qu'une spécification générée.
- **Sous-déclaration du coût du délai d'attente.** Les exécutions tuées à 900 s rapportent un coût de 0 $, ce qui flatte l'invite
les totaux du chemin.
