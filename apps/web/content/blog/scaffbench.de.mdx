Programmierer sind sehr gut darin, Code zu schreiben, aber überraschend schlecht darin, Projekte zu *starten*. Bitten Sie einen darum
Bauen Sie ein Fullstack-Monorepo in Produktionsqualität auf, und es wird gerne zehn Minuten mit dem Schreiben von Hand verbringen
Manifeste, Sperrdateien und Konfigurationen – und das Ergebnis wird häufig nicht erstellt.

ScaffBench misst genau das: einen Agenten in einem leeren Arbeitsbereich, eine Projektspezifikation und eine harte
Frage am Ende: **Wird das generierte Projekt installiert und erstellt?** Wir haben die gleichen Spezifikationen ausgeführt
durch drei Erstellungspfade, um zu isolieren, wie viel Werkzeug hilft:

- **MCP** – Der Agent verwendet den Better-Fullstack-Server MCP, um das Projekt zu planen und zu unterstützen.
- **BF-Erwähnung** – nein MCP; Die Eingabeaufforderung verweist den Agenten auf Better-Fullstack CLI und Dokumente und darauf
stellt den Befehl `create` selbst zusammen.
- **Prompt** – überhaupt kein Better-Fullstack; Der Agent schreibt jede Datei von Grund auf handschriftlich.

Wir haben die vollständige Suite zweimal ausgeführt, auf zwei verschiedenen Agenten CLIs: **OpenAI Codex CLI** mit drei GPT
Modelle (10. Juni) und **Claude Code** mit drei Claude-Modellen (12. Juni) – insgesamt 72 Läufe. Dasselbe
Spezifikationen, gleiche Eingabeaufforderungen, gleiche Validierung. Wir haben dann absichtlich einen [leichten dritten Sweep](#the-light-sweep-gemini-kilo-and-opencode) hinzugefügt.
— neun weitere Modelle in **Gemini CLI**, **Kilo** und **Opencode**, nur in der leichtesten Spezifikation —
Damit beläuft sich die Gesamtzahl auf 102 Läufe und fünfzehn Modelle.

## Schlagzeilenergebnisse

Die Schlagzeilenzahlen stammen aus dem Claude Code-Sweep – 36 Durchläufe: drei Modelle (Fable 5, Opus 4.8,
Sonnet 4.6) × drei Erstellungspfade × vier Projektspezifikationen. Gleiche Maschine, gleicher Kabelbaum, gleiche Eingabeaufforderungen
abgesehen von den Anweisungen für den Erstellungsmodus. Der [GPT-Sweep](#gpt-models-on-codex-cli) zeigt die
haben die gleiche Struktur und werden nachstehend separat berichtet.

| Schöpfungspfad | Durchschnittliche Zeit | Mittlere Zeit | Durchschnittliche Ausgabetokens | Baut vorbei |
| --- | ---: | ---: | ---: | ---: |
| MCP | 113,4s | 85,9s | 5.553 | 12/12 (100 %) |
| BF-Erwähnung | 219,6s | 172,1s | 11.060 | 9/9 (100 %)* |
| Prompt | 516,2s | 424,0s | 25.859 | 9/12 (75 %) |

\* Drei BF-Erwähnungsläufe werden vom Nenner ausgeschlossen, weil sie bei einer Vorlage fehlgeschlagen sind
Generatorfehler auf unserer Seite, nicht auf irgendetwas, was der Agent getan hat – siehe [die Bewertungsrichtlinie](#what-counts-as-a-failure).

Drei Dinge fielen auf:

1. **MCP ist im Durchschnitt etwa 4,6-mal schneller als nur mit Eingabeaufforderung** (und etwa 4,9-mal im Median), mit etwa 4,7-mal weniger
Ausgabetoken. Der Agent versendet die Konfiguration, keinen Code.
2. **Jeder MCP-Lauf hat die Validierung bestanden**, für jedes Modell. 75 % bestandene Läufe, bei denen es nur um Aufforderung ging – zwei Läufe waren erfolgreich
die 15-minütige Auszeit und eine erzeugte ein Monorepo, das nicht erstellt wurde.
3. **Werkzeuge komprimieren Modellunterschiede.** Auf dem Pfad MCP stimmte sogar das kleinste Modell mit überein
Grenzmodelle mit einer Erfolgsquote von 100 % – es ist einfach schneller und günstiger angekommen.

## Methodik

### Die drei Schöpfungspfade

Jeder Lauf startet in einem leeren Verzeichnis mit demselben Basis-Prompt:

```text
You are running in an empty benchmark workspace:
{run_dir}

Create exactly one project directory named `{project_name}`.
Do not ask questions. Do not start a dev server. Do not write outside the
current working directory.
At the end, report the commands you ran and any errors you hit.

Benchmark target: {spec_title}
Requirements:
{spec_requirements}
```

Der einzige Unterschied zwischen den Pfaden besteht in der an diese Eingabeaufforderung angehängten Anweisung zum Erstellungsmodus.
Bei Läufen, bei denen es nur um Aufforderung geht, ist es ausdrücklich verboten, irgendetwas von uns zu berühren:

```text
Creation mode: prompt-only.
Do not use the Better-Fullstack MCP server, Better-Fullstack CLI,
Better-Fullstack website, or files from the Better-Fullstack repository.
Create the project from scratch by writing the files and manifests needed
for a runnable starter.
```

MCP-Läufe bewirken das Gegenteil:

```text
Creation mode: Better-Fullstack MCP.
Use the Better-Fullstack MCP tools, starting with bfs_get_guidance. Then use
schema/compatibility/plan as needed and call bfs_create_project to create
the project.
```

Und BF-Erwähnungsläufe liegen dazwischen – keine MCP-Tools, aber die Eingabeaufforderung nennt die CLI und eine README-Datei
Der Agent kann lesen, bevor er selbst einen nicht interaktiven `create`-Befehl verfasst.

### Die vier Projektspezifikationen

Die Spezifikationen sind bewusst vom „Wochenendprojekt“ bis zum „Stack, den ein echtes Team vertreten würde“ gestreut
etwa eine Woche lang“:

| Spez | Was es verlangt |
| --- | --- |
| `light-ts` | Reagiere + Vite, Hono auf Bun, tRPC, SQLite + Drizzle, Tailwind + DaisyUI, Pino, Vitest, Biome – keine Authentifizierung, keine Zahlungen |
| `heavy-ts` | Next.js **und** eine Expo/React Native-App, Hono auf Bun, oRPC, PostgreSQL + Drizzle, Better Auth, Stripe, Resend, UploadThing, Effect, TanStack Store/Form/Query, Valibot, Vitest + Playwright, Vercel AI SDK, Socket.IO, Inngest, Framer Motion, OpenTelemetry, PostHog, Umami, Sanity, Upstash Redis, Algolia, S3, shadcn/ui, Turborepo, MSW, Storybook, PWA, Docker compose |
| `python-ai` | FastAPI, SQLModel, PostgreSQL, Pydantic, JWT auth, Celery, Strawberry GraphQL, LangChain + OpenAI SDK + LangGraph + CrewAI, Ruff |
| `multi-ecosystem` | TypeScript Next.js Frontend, Python FastAPI Backend (SQLModel/Pydantic/LangChain/Celery), Go Gin-Dienst (GORM, gRPC, Cobra, Zap), geteilt PostgreSQL |

`heavy-ts` ist der Stresstest. Hier finden sich auch alle interessanten Fehler in diesem Benchmark
passiert.

### Das Geschirr

Zwei Sweeps, ein Kabelbaum-Design:

- **Claude Sweep:** Claude Code im Headless-Modus (`claude -p`), JSON-Ereignisstrom erfasst pro
laufen. Modelle: `claude-fable-5`, `claude-opus-4-8`, `claude-sonnet-4-6`, Standardbegründungsaufwand.
- **GPT-Sweep:** OpenAI Codex CLI (`codex exec`), JSON-Ereignisstrom, der pro Lauf erfasst wird. Modelle:
`gpt-5.3-codex-spark` (hoher Aufwand), `gpt-5.4` (mittel), `gpt-5.5` (mittel).
- **Isolierung:** Jeder Lauf erhält einen neuen leeren Arbeitsbereich. MCP-Ausführungen erhalten nur eine strikte MCP-Konfiguration
der Better-Fullstack-Server angeschlossen.
- **Timeout:** 900 Sekunden pro Lauf. Eine Zeitüberschreitung gilt als Fehler für den Agenten.
- **Metriken:** Arbeitszeit, Token-Nutzung aus dem eigenen Nutzungsbericht des Agenten, MCP Tool-Aufrufe von
Der Event-Stream. Die Dollarkosten werden nur beim Claude-Sweep erfasst – bei unserem Codex-Geschirr nicht
Melden Sie die gemessenen Kosten.

Eine wichtige Asymmetrie: Der GPT-Sweep lief zwei Tage zuvor, gegen den Generator *vor* uns
Die aufgedeckten Vorlagenfehler wurden behoben. Aus diesem Grund gelten die unten aufgeführten Generator-Fehlerausschlüsse für die GPT
Ergebnisse werden schwieriger – und warum die beiden Sweeps hinsichtlich der Erfolgsquoten nicht direkt miteinander verglichen werden sollten.

### Was gilt als bestanden

Die Validierung wird *nachdem* der Agent abgeschlossen ist, für alles durchgeführt, was auf der Festplatte verblieben ist:

- **TypeScript:** `bun install`, dann das projekteigene Skript `build` (oder `check`/`lint`/`test`).
- **Python:** `compileall` im gesamten Projekt, ausgenommen Virtualenvs und bereitgestellte Pakete.
- **Go:** `go mod download` und das Modul muss kompiliert werden.

Ein Lauf ist nur dann erfolgreich, wenn alle anwendbaren Prüfungen Null ergeben. „Es sieht aus wie ein Projekt“ zählt nicht;
es muss gebaut werden.

### Was gilt als Misserfolg

<Callout kind="info">
**Bewertungsrichtlinie:** Fehler, die durch Fehler in unserem eigenen Vorlagengenerator verursacht werden, sind von der Bewertung ausgeschlossen
Erfolgsquoten-Nenner – Nur vom Agenten verursachte Fehler zählen für den Agenten. Der Maßstab
Die Aufgabe besteht darin, die Agenten zu messen, und nicht darin, unsere Fehler in ihren Score einfließen zu lassen. Die Kehrseite: Wir veröffentlichen
diese Generatorfehler unten, anstatt sie zu verstecken.
</Callout>

Beim Claude-Sweep war das genau dreimal von Bedeutung: Alle drei Modelle, auf dem BF-Erwähnungspfad,
`heavy-ts` korrekt über CLI eingerüstet – und alle drei haben in unserem den gleichen Generatorfehler festgestellt
native (Expo) Vorlage, wobei `app.json` einen Web-Ausgabemodus festlegt, der nur mit `expo-router` funktioniert.
Die Agenten haben alles richtig gemacht; Die Vorlage konnte nicht erstellt werden. Diese drei Läufe sind ausgeschlossen.
Fehler im Eingabeaufforderungspfad (zwei Zeitüberschreitungen, ein fehlerhafter handgeschriebener Build) sind vollständig vom Agenten verursacht und
voll zählen.

Beim GPT-Sweep – der gegen den Präfixgenerator lief – war es sieben Mal von Bedeutung: `heavy-ts`
bis MCP und bis CLI ist bei allen drei Modellen aufgrund des seitdem behobenen Storybook/Expo-Fehlers fehlgeschlagen
Kette, plus ein Multi-Ökosystem-Gerüst, dessen Go-Dienst ohne generierten `go.sum` ausgeliefert wurde.
Alle sieben sind aus demselben Grund ausgeschlossen; Die eigenen Ausfälle der GPT-Agenten (drei defekte
handgeschriebene Builds auf dem Prompt-Pfad) zählen vollständig.

### Kosten

Der Claude-Sweep kostete **60,60 $** an gemessener API-Nutzung (weniger als die tatsächliche Zahl – die beiden).
Timeout-Läufe melden $0; das Codex-Geschirr meldet überhaupt keine Kosten). Das Lesen der Verbreitung ist
Interessanter als die Gesamtheit:

- Günstigster Passdurchlauf: **0,12 $** – Sonnet 4.6 schafft `light-ts` durch den CLI in 21 Sekunden.
- Teuerste Passläufe: **3,50–4,21 $** – Fable 5-Handschriftprojekte auf dem Prompt-Pfad.
- Für Sonnet kostete der Eingabeaufforderungspfad etwa das Fünffache seines MCP-Pfads (7,11 $ gegenüber 1,47 $) und war für a etwa 6,6 mal langsamer
*geringere* Erfolgsquote.

Das Muster gilt für alle drei Modelle: Je mehr vom Projekt der Agent verfassen muss
desto mehr zahlen Sie für ein weniger zuverlässiges Ergebnis.

- **Die Struktur ist herstellerunabhängig.** Für jedes GPT-Modell ist MCP der schnellste Pfad und

- **Spark ist auf Geschwindigkeit ausgelegt, und das merkt man.** GPT-5.3 Codex Spark benötigte durchschnittlich 32 Sekunden pro MCP
Gerüst – die schnellsten Zellen im gesamten Benchmark – aber auf dem schnellen Weg kam seine Geschwindigkeit zu kurz
Der Preis der Zuverlässigkeit: Projekte wurden in weniger als einer Minute handgeschrieben und nur die Hälfte davon umgesetzt.
- **GPT-5.5 ist der Ausreißer beim Eingabeaufforderungspfad.** Es war das einzige Modell in beiden Durchläufen, das *alle* bestanden hat.
Nur-Prompt-Spezifikation, einschließlich `heavy-ts` – bei 542 Sekunden und 28.000 Ausgabetokens für diesen einen Lauf,
im Vergleich zu 108 Sekunden bis MCP.
- **GPT-Modelle stützen sich stärker auf die Tools.** Codex-Läufe führten zu 10–34 MCP-Aufrufen pro Gerüst im Vergleich zu
Claudes 3–10, erneute Überprüfung der Kompatibilität und häufigere Neuplanung – mit dem gleichen Endergebnis.

Ein Vorbehalt, der es wert ist, wiederholt zu werden: Die beiden Durchsuchungen wurden an unterschiedlichen Agentensystemen, an unterschiedlichen Tagen und an unterschiedlichen Tagen durchgeführt
verschiedene Generatorversionen. Der Vergleich von Pfaden innerhalb eines Sweeps ist ein Experiment. Anbieter vergleichen
über Sweeps hinweg ist dies nicht der Fall.

Was der Light Sweep hinzufügt:

- **Die unterstützten Pfade gingen 18 zu 18.** Jedes Modell bestand sowohl die MCP- als auch die BF-Erwähnung. Das beinhaltet
Schritt 3.7 Flash – ein kleines, schnelles, kostenloses Modell – das in 16 Schritten ein korrektes Fullstack-Monorepo erstellt
Sekunden und 640 Ausgabetokens über den BF-Erwähnungspfad. Die Struktur, die wir bei Claude und GPT gesehen haben
ist kein Grenzmodellphänomen; Die Werkzeuge ebnen das Spielfeld ganz nach unten.
- **Bei „Nur Eingabeaufforderung“ wird das Feld aufgeteilt.** Drei von neun Modellen haben fehlerhafte Projekte ausgeliefert, jeweils in

Methode (Typprüfung fehlgeschlagen); Nex N2-Pro hat einen Import geschrieben, der nicht aufgelöst werden kann (Rollup fehlgeschlagen).
Auf den unterstützten Pfaden ist keiner dieser Fehlermodi vorhanden, da der Agent niemals einen von Hand schreibt
Manifest.

3.000 Ausgabetoken. Jeder Durchlauf nur mit Eingabeaufforderung kostete 8–36.000 – der 36.000 des MiniMax M3 ist der teuerste
Einzellauf in ScaffBench bisher für ein Projekt, das nicht erstellt wurde.
- **Gemini 3.1 Pro meistert diese Aufgabe hervorragend.** Schnellste BF-Erwähnungszelle des gesamten Benchmarks
(26,7 Sekunden, 863 Token), ein lehrbuchmäßiger MCP-Fluss mit vier Aufrufen und eines der wenigen Modelle, die bestanden wurden
Nur mit Eingabeaufforderung. Seine Wiederholungsprobe ist auch eine nützliche Ehrlichkeitsprüfung: Der erste Prompt-Lauf schlug bei Biome fehl
Flusen, der zweite hat sauber bestanden – Single-Run-Zellen sind gerichtet, nicht endgültig.

Vorsichtsmaßnahmen beim Light-Sweep: ein Lauf pro Zelle und 25 der Läufe werden gleichzeitig ausgeführt, also Wanduhr
Die Zeiten bringen Konkurrenzgeräusche mit sich (Geminis BF-Erwähnungslauf maß 27 Sekunden alleine und 50 Sekunden parallel
Batch – gleiches Ergebnis, unterschiedlicher Traffic). Behandeln Sie die Pfadstruktur und „Bestanden/Nicht bestanden“ genau wie das Signal
Sekunden als ungefähre Angabe.

## Qualitative Analyse

### MCP hält Agenten auf Trab

MCP läuft konvergierend auf der gleichen Werkzeugsequenz: `bfs_get_guidance` → Kompatibilitätsprüfung → Plan →
`bfs_create_project`, zwischen 3 und 10 Aufrufe pro Lauf (die Multi-Ökosystem-Spezifikation benötigt am meisten).
Iterationen, um einen gültigen Stapel zu begleichen). Die Ausgabe bleibt gering, da der Job des Agenten zusammenbricht
*Auswahl einer Konfiguration* – das Schreiben der Datei erfolgt deterministisch durch den Generator. Das ist
auch, warum die Erfolgsquoten nicht sinken, wenn die Spezifikation wächst: `heavy-ts` bis MCP für alle bestanden
Modell.

### Nur-Prompt-Agenten ertrinken in der schweren Spezifikation

Auf dem schnellen Weg besiegte `heavy-ts` fast alle. Fable 5 und Opus 4.8 waren noch in der Schreibphase
Dateien nach dem 15-minütigen Timeout. Sonnet 4.6 beendete ein Monorepo mit 90 Dateien in 767 Sekunden – und das
nicht gebaut. GPT-5.3 Codex Spark sprintete in 68 Sekunden zu einem handgeschriebenen Projekt – das war nicht der Fall
entweder bauen. Die einzige Ausnahme bei beiden Durchläufen war GPT-5.5, das `heavy-ts` durchlief
prompt-only in 542 Sekunden und bestanden. Die leichteren Spezifikationen bestanden meist nur nach Eingabeaufforderung, aber nach Minuten
von jeweils 10.000 bis 37.000 Ausgabe-Tokens. Das handschriftliche Schreiben eines Starters ist etwas, was Grenzmodelle betrifft
*kann* tun; Es ist einfach der langsamste, teuerste und unzuverlässigste Weg, eins zu bekommen.

### Der Benchmark prüfte unsere eigenen Vorlagen

Die nützlichsten Fehler waren unsere. Bei der Validierung jedes generierten Projekts wurden **sechs Ebenen angezeigt
Vorlagenfehler** in unserem Generator, die sich meist hinter einem einzelnen „Storybook-Build-Fehler“-Symptom verstecken
`heavy-ts`. Der GPT-Sweep traf sie zuerst – acht seiner neun `heavy-ts`-Builds schlugen fehl, sechs davon
sie auf dieser Generatorkette – und der Claude-Sweep zwei Tage später bestätigte, welche Korrekturen zutrafen:

1. Storybook-Vorlagen haben das Frontend mit einer Gleichheitsprüfung getestet, aber `frontend` ist ein Array.
2. Aufgrund einer Nichtübereinstimmung des Datenbankpaketpfads blieb das DB-Paket ohne seine Abhängigkeiten im Diagrammteilmodus.
3. `expo-network` fehlte in einer nativen Vorlagenvariante.
4. Storybook 8-Framework-Pakete exportieren die Typen `Meta`/`StoryObj` nicht erneut – Importe mussten verschoben werden
die Renderer-Pakete.
5. Für `@better-auth/expo` waren vier Expo-Peer-Abhängigkeiten erforderlich, die explizit installiert wurden.
6. Native `app.json` legt einen statischen Web-Ausgabemodus fest, der nur mit `expo-router` funktioniert – dieser ist es
noch offen, und ist der Fehler hinter den drei ausgeschlossenen BF-Erwähnungsläufen.

Fünf der sechs wurden in `create-better-fullstack` 2.0.2 behoben und ausgeliefert, bevor dieser Beitrag veröffentlicht wurde
aus. Es erweist sich als äußerst wirksame Qualitätssicherung, Ihr eigenes Produkt einem Agenten-Benchmark zu unterziehen
passieren.

### Vorbildlicher Charakter

Sonnet 4.6 war in jeder Hinsicht das schnellste und günstigste Gerät, und bei der Ausstattung gab es nichts auf
Zuverlässigkeit. Fable 5 war am bewusstesten – höchste Token-Anzahl, längste Läufe und mehr
prompter Pfad, dass Gründlichkeit immer noch nicht ausreichte, um die Zeitüberschreitung bei `heavy-ts` zu überwinden. Opus 4.8 sat
zuverlässig dazwischen. Die Rangfolge hat sich zwischen den Pfaden nie geändert; Die *Lücke* tat es. Die Werkzeugausstattung ist großartig
Ausgleich: Bei MCP betrug die Spanne zwischen bestem und schlechtestem Modell 102 Sekunden; Prompt-nur es war
unterschiedliche Minuten und zweimal eine Wanduhr-Decke.

## Einschränkungen

- **Ein Lauf pro Zelle.** 102 Läufe reichen aus, um die Struktur zu erkennen, nicht um Konfidenzintervalle festzulegen
darauf. Die Gerüstzeiten variieren auch mit der API-Last – und die parallelen Batch-Zugaben des Light Sweeps
Konkurrenzgeräusch obendrein (Geminis eigene Wiederholung zeigte sowohl eine 2-fache Zeitspanne als auch ein Flusen-Nichtbestehen/Bestehen
Wechsel zwischen identischen Läufen).
- **Die Sweeps sind kein Kopf-an-Kopf-Rennen.** Die GPT-Läufe verwendeten einen anderen Agenten CLI (Codex), anders
Reasoning-Effort-Einstellungen, eine frühere Generatorversion und keine Dollarkosten melden; das Licht
Der Sweep deckt eine Spezifikation für drei weitere CLIs ab. Vergleiche innerhalb des Sweep-Pfads sind solide;
Anbieterübergreifende Modellrankings sind nicht das, was dieser Benchmark misst.
- **Wir haben unser eigenes Tool einem Benchmarking unterzogen.** Das System, die Eingabeaufforderungen und die Validierung sind im Sinne der Öffentlichkeit –
Eingabeaufforderungen und Richtlinien sind oben aufgeführt – und die vollständige Tabelle pro Lauf finden Sie im Anhang. Behandeln Sie die
Vergleich zwischen Pfaden (Werkzeug vs. kein Werkzeug) als Ergebnis, nicht als Aussage über andere
Gerüstbauer.
- **Builds, keine Features.** Die Validierung beweist, dass das Projekt installiert und erstellt wird – nicht, dass die Authentifizierung erfolgt
Flow funktioniert oder der Stripe-Webhook ist korrekt verkabelt. Ein vorübergehendes Projekt, das nur auf Eingabeaufforderungen ausgerichtet ist, kann immer noch vorhanden sein
Es fehlt mehr von der Spezifikation als von einer generierten.
- **Zeitüberschreitungskosten werden nicht ausreichend gemeldet.** Läufe, die bei 900 Sekunden beendet wurden, melden Kosten von 0 $, was der Eingabeaufforderung schmeichelt
Gesamtsummen des Pfades.
