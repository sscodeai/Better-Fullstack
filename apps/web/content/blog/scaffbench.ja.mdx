コーディング エージェントはコードを書くのが得意ですが、プロジェクトを「開始」することは驚くほど苦手です。一人に尋ねてください
プロダクショングレードのフルスタックモノリポジトリを足場にすれば、喜んで手書きで 10 分を費やすことができます。
マニフェスト、ロックファイル、構成 - そして結果がビルドされないことがよくあります。

ScaffBench は、空のワークスペース内のエージェント、プロジェクトの仕様、およびハードウェアの状態を正確に測定します。
最後の質問 — **生成されたプロジェクトはインストールおよびビルドされますか?** 同じ仕様を実行しました
3 つの作成パスを通じて、ツールがどの程度役立つかを特定します。

- **MCP** — エージェントは Better-Fullstack MCP サーバーを使用して、プロジェクトの計画と足場を作成します。
- **BF への言及** — MCP はありません。プロンプトはエージェントに Better-Fullstack CLI とドキュメントを指示します。
`create` コマンド自体を作成します。
- **プロンプト** — Better-Fullstack はまったくありません。エージェントはすべてのファイルを最初から手書きします。

2 つの異なるエージェント CLIs で完全なスイートを 2 回実行しました: **OpenAI Codex CLI** (3 つの GPT を使用)
モデル (6 月 10 日)、および 3 つのクロード モデルを使用した **Claude Code** (6 月 12 日) — 合計 72 回の実行。同じ
仕様、同じプロンプト、同じ検証。次に、意図的に [軽い 3 回目のスイープ](#the-light-sweep-gemini-kilo-and-opencode) を追加しました。
— **Gemini CLI**、**Kilo**、**opencode** のさらに 9 つのモデル (最軽量仕様のみ) —
合計 102 回の実行と 15 個のモデルになります。

## 見出しの結果

ヘッドラインの番号は Claude Code スイープから取得されています — 36 回の実行: 3 つのモデル (Fable 5、Opus 4.8、
Sonnet 4.6) × 3 つの作成パス × 4 つのプロジェクト仕様。同じマシン、同じハーネス、同じプロンプト
作成モードの指示は別として。 [GPT スイープ](#gpt-models-on-codex-cli) は、
同じ構造であり、以下で個別に報告されます。

| 作成パス | 平均時間 | 中央時間 | 平均出力トークン | ビルドの合格 |
| --- | ---: | ---: | ---: | ---: |
| MCP | 113.4秒 | 85.9秒 | 5,553 | 12/12 (100%) |
| BFの言及 | 219.6秒 | 172.1秒 | 11,060 | 9/9 (100%)* |
| プロンプト | 516.2秒 | 424.0秒 | 25,859 | 9/12 (75%) |

\* 3 つの BF メンションの実行は、テンプレートで失敗したため分母から除外されます
エージェントが行ったことではなく、私たち側のジェネレーターのバグです。[スコアリング ポリシー](#what-counts-as-a-failure) を参照してください。

目立った点は 3 つあります。

1. **MCP は、プロンプトのみ**よりも平均で約 4.6 倍高速 (中央値では約 4.9 倍)、約 4.7 倍少ない
トークンを出力します。エージェントはコードではなく構成を提供します。
2. **すべてのモデルについて、すべての MCP 実行が検証に合格しました**。プロンプトのみのランが 75% を通過 - 2 ランがヒット
15 分のタイムアウトと、構築されなかったモノリポジトリが生成されたものが 1 つありました。
3. **ツールによりモデルの差異が圧縮されます。** MCP パスでは、最小のモデルでも一致しました。
100% の合格率でフロンティア モデルを開発します。より早く、より安く到着します。

## 方法論

### 3 つの創造の道

すべての実行は、同じベース プロンプトを持つ空のディレクトリで開始されます。

```text
You are running in an empty benchmark workspace:
{run_dir}

Create exactly one project directory named `{project_name}`.
Do not ask questions. Do not start a dev server. Do not write outside the
current working directory.
At the end, report the commands you ran and any errors you hit.

Benchmark target: {spec_title}
Requirements:
{spec_requirements}
```

パス間の唯一の違いは、そのプロンプトに追加される作成モードの指示です。
プロンプトのみの実行では、私たちのものに触れることは明示的に禁止されています。

```text
Creation mode: prompt-only.
Do not use the Better-Fullstack MCP server, Better-Fullstack CLI,
Better-Fullstack website, or files from the Better-Fullstack repository.
Create the project from scratch by writing the files and manifests needed
for a runnable starter.
```

MCP を実行すると、次の逆の結果が得られます。

```text
Creation mode: Better-Fullstack MCP.
Use the Better-Fullstack MCP tools, starting with bfs_get_guidance. Then use
schema/compatibility/plan as needed and call bfs_create_project to create
the project.
```

そして、BF メンションの実行はその間にあります。MCP ツールはありませんが、プロンプトには CLI という名前が付けられ、README には
エージェントは、非対話型 `create` コマンド自体を作成する前に読み取ることができます。

### ハーネス

2 つのスイープ、1 つのハーネス設計:

- **クロード スイープ:** ヘッドレス モードの Claude Code (`claude -p`)、ごとにキャプチャされた JSON イベント ストリーム
走る。モデル: `claude-fable-5`、`claude-opus-4-8`、`claude-sonnet-4-6`、デフォルトの推論作業。
- **GPT スイープ:** OpenAI Codex CLI (`codex exec`)、実行ごとにキャプチャされた JSON イベント ストリーム。モデル:
`gpt-5.3-codex-spark` (高労力)、`gpt-5.4` (中)、`gpt-5.5` (中)。
- **分離:** 実行するたびに、新しい空のワークスペースが取得されます。 MCP を実行すると、厳密な MCP 構成が取得されます。
Better-Fullstack サーバーが接続されています。
- **タイムアウト:** 実行ごとに 900 秒。タイムアウトはエージェントの失敗としてカウントされます。
- **メトリクス:** 実時間、エージェント自身の使用状況レポートからのトークン使用量、からの MCP ツール呼び出し
イベントストリーム。ドルコストはクロード スイープでのみ取得されます。当社のコーデックス ハーネスでは取得されません。
従量制コストをレポートします。

重要な非対称性の 1 つ: GPT スイープは 2 日前、ジェネレーターに対して実行されました。
公開されたテンプレートのバグを修正しました。そのため、以下のジェネレーターのバグ除外が GPT に該当します。
結果はさらに難しくなります。そして、なぜ 2 つのスイープの合格率を直接比較すべきではないのか。

### 何が合格とみなされるのか

エージェントが終了した後、ディスク上に残ったものに対して検証が実行されます。

- **TypeScript:** `bun install`、次にプロジェクト独自の `build` (または `check`/`lint`/`test`) スクリプト。
- **Python:** プロジェクト全体で `compileall` (virtualenvs とベンダー パッケージを除く)。
- **Go:** `go mod download` とモジュールをコンパイルする必要があります。

実行は、該当するすべてのチェックがゼロで終了した場合にのみ合格します。 「プロジェクトのように見える」ということは重要ではありません。
それは構築しなければなりません。

### 何が失敗とみなされるのか

<Callout kind="info">
**スコア ポリシー:** 独自のテンプレート ジェネレーターのバグが原因で発生した失敗はスコアから除外されます。
合格率の分母 — エージェントが作成した失敗のみがエージェントに対してカウントされます。ベンチマークの
仕事はエージェントを測定することであり、バグを洗浄してスコアに組み込むことではありません。裏側: 私たちは出版します
ジェネレーターのバグを非表示にする代わりに、以下に示します。
</Callout>

クロード スイープでは、これはちょうど 3 回問題になりました。BF 言及パス上の 3 つのモデルすべて、
CLI を介して `heavy-ts` を正しくスキャフォールディングしました。そして、3 つすべてが同じジェネレーターのバグに遭遇しました。
ネイティブ (Expo) テンプレート。`app.json` は、`expo-router` でのみ機能する Web 出力モードを設定します。
エージェントはすべてを正しく行いました。テンプレートを構築できませんでした。この3回のランは除外されます。
プロンプト パスの障害 (タイムアウト 2 回、手書きビルドの破損 1 回) はすべてエージェントによって作成され、
全部数えます。

GPT スイープ (プレフィックス ジェネレーターに対して実行) では、7 回重要でした: `heavy-ts`
MCP 経由と CLI 経由は、修正された Storybook/Expo バグにより 3 つのモデルすべてで失敗しました
チェーンに加えて、生成された `go.sum` なしで Go サービスが出荷されたマルチエコシステム スキャフォールドが 1 つあります。
7 つすべてが同じ理由で除外されます。 GPT エージェント自身の失敗 (3 つの壊れた)
プロンプト パス上の手書きのビルドも完全にカウントされます。

- **構造はベンダーに依存しません。** すべての GPT モデルで、MCP が最速のパスであり、

クロードの 3 ～ 10 では、互換性を再チェックし、より頻繁に再計画を行っていますが、最終結果は同じです。

繰り返す価値のある注意点: 2 つのスイープは、異なるエージェント ハーネス、異なる日に実行されました。

アクロススイープはそうではありません。
